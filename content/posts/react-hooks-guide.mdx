React Hooks는 함수 컴포넌트에서 상태 관리와 부수 효과를 처리할 수 있게 해주는 기능입니다. 2019년 React 16.8에서 도입된 이후 클래스 컴포넌트를 거의 완전히 대체했고, 2026년 현재 React 개발의 근간이 되는 개념입니다.

저는 React를 처음 배울 때 클래스 컴포넌트부터 시작했는데, Hooks를 접하고 나서 "아, 이게 React가 원래 의도한 방향이었구나"라고 느꼈습니다. 하지만 Hooks가 간단해 보인다고 해서 제대로 이해하기 쉬운 것은 아닙니다. 특히 `useEffect`의 의존성 배열, `useMemo`/`useCallback`의 적절한 사용 시점, 커스텀 훅 설계 등은 실무 경험을 쌓아야 제대로 감이 옵니다.

이 글에서는 3년 넘게 React Hooks를 프로덕션에서 사용하면서 체득한 패턴, 흔한 실수, 최적화 판단 기준, 그리고 팀에서 실제로 사용하는 커스텀 훅 레시피를 체계적으로 정리합니다.

## useState - 상태 관리의 기본이지만 함정이 많은 훅

`useState`는 가장 기본적인 Hook으로, 컴포넌트에 상태 변수를 추가합니다. 간단해 보이지만 실무에서는 의외로 많은 실수가 발생하는 곳이기도 합니다.

```typescript
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
      <button onClick={() => setCount(prev => prev - 1)}>-1</button>
    </div>
  );
}
```

### 핵심 패턴과 주의사항

**함수형 업데이트 - 반드시 써야 하는 상황**: 이전 상태를 기반으로 새 상태를 계산할 때는 반드시 콜백 함수를 사용해야 합니다. 제가 처음 React를 배울 때 이걸 무시하고 직접 값을 사용했다가 연속 클릭 시 카운터가 제대로 증가하지 않는 버그를 겪은 적이 있습니다.

```typescript
// 위험한 패턴 - 빠른 연속 클릭에서 누락 발생 가능
setCount(count + 1);
setCount(count + 1); // 이 두 호출이 같은 count 값을 참조!

// 안전한 패턴 - 항상 최신 상태를 기반으로 업데이트
setCount(prev => prev + 1);
setCount(prev => prev + 1); // 각각 최신 값을 참조
```

**객체 상태 업데이트와 불변성**: 객체를 상태로 사용할 때는 스프레드 연산자로 불변성을 유지해야 합니다. React는 참조 비교로 상태 변경을 감지하기 때문에, 객체를 직접 수정하면 리렌더링이 트리거되지 않습니다.

```typescript
const [user, setUser] = useState({ name: "", email: "", age: 0 });

// 잘못된 패턴 - 리렌더링이 트리거되지 않음!
user.name = "Jay";
setUser(user); // 같은 참조이므로 React가 변경을 감지 못함

// 올바른 패턴 - 새 객체를 생성하여 업데이트
setUser(prev => ({ ...prev, name: "Jay" }));
```

실무에서 복잡한 객체 상태를 다루다 보면 깊은 중첩 구조의 불변 업데이트가 매우 번거로워집니다. 이런 경우 Immer 라이브러리의 `useImmer` 훅을 사용하면 코드가 훨씬 직관적이 됩니다. 저희 팀에서는 3단계 이상 중첩된 객체 상태에는 `useImmer`를 사용하는 것을 규칙으로 정했습니다.

```typescript
import { useImmer } from "use-immer";

const [settings, updateSettings] = useImmer({
  profile: {
    personal: { name: "", avatar: "" },
    preferences: { theme: "light", language: "ko" },
  },
});

// Immer를 사용하면 직접 수정하는 것처럼 작성 가능
updateSettings(draft => {
  draft.profile.preferences.theme = "dark";
});
```

**초기값 지연 계산 - 성능 최적화의 첫 걸음**: 초기값 계산이 비용이 큰 경우 함수를 전달하면 최초 렌더링에서만 실행됩니다. 저는 한번 localStorage에서 대량의 데이터를 파싱하는 초기값을 함수로 감싸지 않아서 매 렌더링마다 불필요한 파싱이 발생한 적이 있습니다.

```typescript
// 매 렌더링마다 expensiveComputation이 호출됨 (결과는 첫 렌더링에서만 사용)
const [data, setData] = useState(expensiveComputation());

// 첫 렌더링에서만 expensiveComputation이 호출됨
const [data, setData] = useState(() => expensiveComputation());
```

### useState vs useReducer: 언제 어떤 것을 선택할까

상태가 복잡해지면 `useState`의 한계를 느끼게 됩니다. 저희 팀에서는 다음 기준으로 선택합니다.

| 기준 | useState | useReducer |
|------|----------|-----------|
| 상태의 종류 | 독립적인 단순 값 | 서로 연관된 복합 상태 |
| 업데이트 로직 | 간단한 값 변경 | 복잡한 비즈니스 로직 |
| 상태 전이 패턴 | 없음 | 명확한 액션 기반 전이 |
| 테스트 필요성 | 낮음 | 높음 (reducer 함수 단위 테스트 가능) |
| 디버깅 | 직관적 | 액션 로그로 추적 가능 |

예를 들어, 폼의 각 입력 필드는 `useState`로 충분하지만, 다단계 폼 위저드의 전체 상태(현재 단계, 각 단계의 유효성, 이전/다음 네비게이션 등)는 `useReducer`가 훨씬 적합합니다.

## useEffect - 가장 많은 버그가 발생하는 곳

`useEffect`는 데이터 페칭, 구독, DOM 조작 등 컴포넌트 외부와 상호작용하는 작업에 사용합니다. 솔직히 말하면, React 버그의 80% 이상이 `useEffect`에서 발생한다고 해도 과언이 아닙니다. 의존성 배열의 미묘한 동작, 클린업 타이밍, 무한 루프 등 함정이 정말 많습니다.

```typescript
import { useEffect, useState } from "react";

function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState(null);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    let cancelled = false; // 경쟁 조건 방지

    async function fetchUser() {
      try {
        setError(null);
        const response = await fetch(`/api/users/${userId}`);
        if (!response.ok) throw new Error("Failed to fetch");
        const data = await response.json();
        if (!cancelled) {
          setUser(data);
        }
      } catch (err) {
        if (!cancelled) {
          setError(err instanceof Error ? err.message : "Unknown error");
        }
      }
    }
    fetchUser();

    return () => {
      cancelled = true; // 컴포넌트 언마운트 시 이전 요청 무시
    };
  }, [userId]);

  if (error) return <p>Error: {error}</p>;
  if (!user) return <p>Loading...</p>;
  return <p>{user.name}</p>;
}
```

위 코드에서 `cancelled` 변수를 사용한 것에 주목하세요. userId가 빠르게 변경되면 이전 요청이 아직 진행 중인데 새 요청이 시작될 수 있습니다. `cancelled` 플래그 없이는 이전 요청의 결과가 나중에 도착하면서 잘못된 데이터가 표시되는 **경쟁 조건(race condition)** 버그가 발생합니다. 이 버그는 재현이 어려워서 실무에서 정말 고통스럽습니다.

### 의존성 배열의 이해: 내가 겪은 3가지 대표적 실수

**실수 1: 의존성 배열에 객체/배열을 넣는 것**

```typescript
const options = { page: 1, limit: 10 };

// 매 렌더링마다 options는 새 객체이므로 무한 루프!
useEffect(() => {
  fetchData(options);
}, [options]);

// 해결: 원시값으로 분해하거나 useMemo 사용
useEffect(() => {
  fetchData({ page, limit });
}, [page, limit]);
```

**실수 2: 의존성 배열을 빈 배열로 강제하면서 외부 값 참조**

```typescript
function SearchResults({ query }: { query: string }) {
  // ESLint 경고를 무시하고 빈 배열을 사용하면
  // query가 변경되어도 검색이 업데이트되지 않음!
  useEffect(() => {
    search(query);
  }, []); // query가 의존성에 빠짐 - 버그!

  useEffect(() => {
    search(query);
  }, [query]); // 올바른 방법
}
```

**실수 3: 불필요한 useEffect 사용**

이것이 가장 흔하고 중요한 실수입니다. 많은 개발자들이 "파생 상태"를 useEffect로 동기화하려 합니다.

```typescript
// 안티패턴: useEffect로 파생 상태 동기화
const [items, setItems] = useState([]);
const [filteredItems, setFilteredItems] = useState([]);

useEffect(() => {
  setFilteredItems(items.filter(item => item.active));
}, [items]);

// 올바른 패턴: 렌더링 중 계산 (필요하면 useMemo)
const filteredItems = items.filter(item => item.active);
// 또는
const filteredItems = useMemo(
  () => items.filter(item => item.active),
  [items]
);
```

React 공식 문서에서도 강조하는 것처럼, **"You Might Not Need an Effect"**입니다. useEffect를 작성하기 전에 "이 로직이 정말 부수 효과인가, 아니면 렌더링 중에 계산할 수 있는 것인가?"를 먼저 생각하세요.

### 클린업 함수: 메모리 누수를 방지하는 필수 습관

이벤트 리스너, 타이머, WebSocket 구독 등을 설정한 경우 반드시 클린업 함수를 반환해야 합니다. 클린업을 누락하면 컴포넌트가 언마운트된 후에도 리스너가 남아서 메모리 누수와 예상치 못한 동작이 발생합니다.

```typescript
useEffect(() => {
  const handleResize = () => setWidth(window.innerWidth);
  window.addEventListener("resize", handleResize);

  return () => {
    window.removeEventListener("resize", handleResize);
  };
}, []);

// WebSocket 예시 - 실무에서 자주 사용하는 패턴
useEffect(() => {
  const ws = new WebSocket("wss://api.example.com/ws");

  ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    setMessages(prev => [...prev, data]);
  };

  ws.onerror = (error) => {
    console.error("WebSocket error:", error);
    setConnectionStatus("error");
  };

  return () => {
    ws.close(); // 반드시 연결 종료
  };
}, []);
```

저희 팀에서는 코드 리뷰 시 "useEffect에 return이 없는 경우"를 반드시 체크합니다. 구독이나 이벤트 리스너가 없는 순수 데이터 페칭에서도 `cancelled` 플래그를 통한 클린업을 권장합니다.

## useRef - DOM 접근 그 이상의 활용

`useRef`는 두 가지 용도로 사용됩니다. DOM 요소에 접근하는 것과, 렌더링 사이에서 값을 유지하는 것입니다. 특히 두 번째 용도가 실무에서 매우 중요합니다.

### DOM 요소 접근

```typescript
function TextInput() {
  const inputRef = useRef<HTMLInputElement>(null);

  const focusInput = () => {
    inputRef.current?.focus();
  };

  return (
    <div>
      <input ref={inputRef} type="text" />
      <button onClick={focusInput}>Focus</button>
    </div>
  );
}
```

### 렌더링 사이에서 값 유지 - useRef의 숨은 강점

`useRef`로 생성된 값은 변경되어도 리렌더링을 트리거하지 않습니다. 이 특성을 이용하면 "렌더링에 영향을 주지 않으면서 값을 기억해야 하는" 경우에 매우 유용합니다.

```typescript
// 패턴 1: 이전 값 기억하기
function usePrevious<T>(value: T): T | undefined {
  const ref = useRef<T>();
  useEffect(() => {
    ref.current = value;
  });
  return ref.current;
}

// 패턴 2: 타이머 ID 관리
function Timer() {
  const intervalRef = useRef<NodeJS.Timeout | null>(null);

  const start = () => {
    intervalRef.current = setInterval(() => {
      console.log("tick");
    }, 1000);
  };

  const stop = () => {
    if (intervalRef.current) {
      clearInterval(intervalRef.current);
    }
  };

  return (
    <div>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
    </div>
  );
}

// 패턴 3: 최신 콜백 참조 유지 (useCallback 대안)
function useLatestCallback<T extends (...args: any[]) => any>(callback: T) {
  const ref = useRef(callback);
  ref.current = callback;
  return useCallback((...args: Parameters<T>) => ref.current(...args), []) as T;
}
```

패턴 3은 제가 실무에서 특히 자주 사용하는 패턴입니다. useCallback의 의존성 배열이 복잡해지는 경우, useRef를 사용해서 항상 최신 콜백을 참조하는 안정적인 함수를 만들 수 있습니다.

### useRef로 isMount 판별하기

컴포넌트가 처음 마운트되었는지 아닌지를 판별해야 할 때 useRef가 유용합니다. 예를 들어, 특정 useEffect를 마운트 시에는 건너뛰고 업데이트 시에만 실행하고 싶을 때 사용합니다.

```typescript
function useUpdateEffect(effect: () => void, deps: any[]) {
  const isMounted = useRef(false);

  useEffect(() => {
    if (isMounted.current) {
      return effect();
    }
    isMounted.current = true;
  }, deps);
}
```

## useMemo와 useCallback - 최적화의 양날의 검

### useMemo: 비용이 큰 계산을 메모이제이션

```typescript
function ProductList({ products, query }: Props) {
  const filteredProducts = useMemo(() => {
    return products.filter(p =>
      p.name.toLowerCase().includes(query.toLowerCase())
    );
  }, [products, query]);

  return (
    <ul>
      {filteredProducts.map(p => <li key={p.id}>{p.name}</li>)}
    </ul>
  );
}
```

### useCallback: 함수 레퍼런스를 메모이제이션

```typescript
function Parent() {
  const [count, setCount] = useState(0);

  const handleClick = useCallback(() => {
    setCount(prev => prev + 1);
  }, []);

  return <Child onClick={handleClick} />;
}
```

### 최적화 판단 기준: 언제 써야 하고 언제 쓰지 말아야 하는가

이 부분이 정말 중요합니다. 많은 개발자들이 "최적화는 무조건 좋은 것"이라고 생각하고 모든 곳에 useMemo와 useCallback을 적용합니다. 하지만 **불필요한 메모이제이션은 오히려 성능을 악화시킵니다**. 메모이제이션 자체에도 비용(메모리 할당, 의존성 비교)이 있기 때문입니다.

저희 팀에서 사용하는 판단 기준을 공유합니다.

**useMemo를 사용해야 하는 경우:**
1. 계산이 실제로 비용이 큰 경우 (수천 개 이상의 배열 필터링/정렬, 복잡한 객체 변환)
2. 계산 결과가 다른 Hook의 의존성 배열에 사용되는 경우
3. `React.memo`로 감싸진 자식 컴포넌트에 객체/배열을 전달하는 경우

**useMemo를 사용하지 말아야 하는 경우:**
1. 단순한 산술 연산이나 문자열 조작
2. 매 렌더링마다 달라지는 값 (메모이제이션 효과 없음)
3. 자식 컴포넌트가 `React.memo`로 감싸져 있지 않은 경우

**useCallback을 사용해야 하는 경우:**
1. `React.memo`로 감싸진 자식 컴포넌트에 콜백을 전달하는 경우
2. 콜백이 다른 Hook의 의존성 배열에 사용되는 경우
3. 커스텀 훅에서 반환하는 함수 (안정적인 참조 보장)

```typescript
// 불필요한 최적화 예시 - 이런 건 하지 마세요
const fullName = useMemo(() => `${firstName} ${lastName}`, [firstName, lastName]);
// 문자열 템플릿 리터럴은 즉시 계산하는 게 더 빠릅니다
const fullName = `${firstName} ${lastName}`;
```

실제로 저희 팀에서 한 프로젝트의 모든 useMemo/useCallback을 감사(audit)한 적이 있는데, 약 40%가 불필요한 최적화였습니다. 불필요한 메모이제이션을 제거한 후 오히려 번들 크기가 줄고 코드 가독성이 좋아졌습니다.

### React 19의 React Compiler와 메모이제이션의 미래

2026년 현재 React 19의 React Compiler(이전 React Forget)가 안정화되면서, 수동 메모이제이션의 필요성이 크게 줄었습니다. React Compiler는 빌드 타임에 자동으로 최적의 메모이제이션을 추가해줍니다. 새 프로젝트에서는 React Compiler 도입을 적극 권장하며, 수동으로 useMemo/useCallback을 작성하는 일이 점점 줄어들 것입니다.

## useContext - 전역 상태 공유의 올바른 사용법

`useContext`는 props drilling 없이 컴포넌트 트리 전체에서 값을 공유합니다.

```typescript
// 컨텍스트 생성과 Provider
const ThemeContext = createContext<{
  theme: "light" | "dark";
  toggleTheme: () => void;
} | null>(null);

// 커스텀 훅으로 감싸기 - 널 체크와 에러 메시지 포함
function useTheme() {
  const context = useContext(ThemeContext);
  if (!context) {
    throw new Error("useTheme must be used within a ThemeProvider");
  }
  return context;
}

function ThemeProvider({ children }: { children: React.ReactNode }) {
  const [theme, setTheme] = useState<"light" | "dark">("light");

  const toggleTheme = useCallback(() => {
    setTheme(prev => prev === "light" ? "dark" : "light");
  }, []);

  const value = useMemo(() => ({ theme, toggleTheme }), [theme, toggleTheme]);

  return (
    <ThemeContext.Provider value={value}>
      {children}
    </ThemeContext.Provider>
  );
}

// 사용하는 곳에서는 깔끔하게
function Button() {
  const { theme, toggleTheme } = useTheme();
  return (
    <button
      onClick={toggleTheme}
      className={theme === "dark" ? "bg-black text-white" : "bg-white text-black"}
    >
      Toggle Theme
    </button>
  );
}
```

### useContext의 성능 함정과 해결 전략

useContext의 가장 큰 함정은 **Provider의 value가 변경되면 해당 Context를 사용하는 모든 컴포넌트가 리렌더링된다**는 것입니다. 이것 때문에 많은 프로젝트에서 성능 문제가 발생합니다.

예를 들어, 사용자 정보와 인증 상태를 하나의 Context에 넣으면 사용자 이름만 변경되어도 인증 상태만 사용하는 컴포넌트까지 리렌더링됩니다.

**해결 전략 1: Context 분리**
```typescript
// 자주 변경되는 값과 거의 변경되지 않는 값을 분리
const UserContext = createContext<User | null>(null);      // 자주 변경
const AuthContext = createContext<AuthState | null>(null);  // 거의 변경 안 됨
```

**해결 전략 2: Provider value 메모이제이션**
```typescript
// value 객체를 useMemo로 감싸서 불필요한 리렌더링 방지
const value = useMemo(() => ({ theme, toggleTheme }), [theme, toggleTheme]);
```

**해결 전략 3: 상태 관리 라이브러리 사용**

전역 상태가 복잡해지면 Context보다 Zustand 같은 경량 상태 관리 라이브러리가 더 적합합니다. Zustand는 selector 패턴을 지원하므로, 특정 상태 조각이 변경될 때만 해당 컴포넌트가 리렌더링됩니다. 저희 팀에서는 "3개 이상의 Context가 중첩되면 Zustand 도입을 고려한다"는 규칙을 가지고 있습니다.

## 커스텀 Hook 만들기: 실무에서 검증된 레시피

반복되는 로직을 커스텀 Hook으로 추출하면 코드 재사용성이 크게 향상됩니다. 저희 팀에서 실제로 사용하는 커스텀 훅들을 공유합니다.

### useLocalStorage - 로컬 스토리지 동기화

```typescript
function useLocalStorage<T>(key: string, initialValue: T) {
  const [storedValue, setStoredValue] = useState<T>(() => {
    if (typeof window === "undefined") return initialValue;
    try {
      const item = window.localStorage.getItem(key);
      return item ? JSON.parse(item) : initialValue;
    } catch {
      return initialValue;
    }
  });

  const setValue = useCallback((value: T | ((val: T) => T)) => {
    setStoredValue(prev => {
      const valueToStore = value instanceof Function ? value(prev) : value;
      if (typeof window !== "undefined") {
        window.localStorage.setItem(key, JSON.stringify(valueToStore));
      }
      return valueToStore;
    });
  }, [key]);

  // 다른 탭에서 변경 시 동기화
  useEffect(() => {
    const handleStorageChange = (e: StorageEvent) => {
      if (e.key === key && e.newValue) {
        setStoredValue(JSON.parse(e.newValue));
      }
    };
    window.addEventListener("storage", handleStorageChange);
    return () => window.removeEventListener("storage", handleStorageChange);
  }, [key]);

  return [storedValue, setValue] as const;
}

// 사용
const [theme, setTheme] = useLocalStorage("theme", "light");
```

위 코드에서 `storage` 이벤트 리스너를 추가한 부분이 핵심입니다. 같은 사이트의 다른 탭에서 로컬 스토리지 값이 변경되면 현재 탭에서도 자동으로 동기화됩니다. 실무에서 다크모드 같은 사용자 설정을 다룰 때 이 동기화가 없으면 탭마다 다른 테마가 표시되는 이슈가 발생합니다.

### useDebounce - 검색 입력 최적화

```typescript
function useDebounce<T>(value: T, delay: number): T {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => setDebouncedValue(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);

  return debouncedValue;
}

// 사용
function Search() {
  const [query, setQuery] = useState("");
  const debouncedQuery = useDebounce(query, 300);

  useEffect(() => {
    if (debouncedQuery) {
      searchAPI(debouncedQuery);
    }
  }, [debouncedQuery]);

  return (
    <input
      value={query}
      onChange={e => setQuery(e.target.value)}
      placeholder="검색어를 입력하세요..."
    />
  );
}
```

### useMediaQuery - 반응형 로직 처리

```typescript
function useMediaQuery(query: string): boolean {
  const [matches, setMatches] = useState(false);

  useEffect(() => {
    const media = window.matchMedia(query);
    setMatches(media.matches);

    const listener = (e: MediaQueryListEvent) => setMatches(e.matches);
    media.addEventListener("change", listener);
    return () => media.removeEventListener("change", listener);
  }, [query]);

  return matches;
}

// 사용
function Layout() {
  const isMobile = useMediaQuery("(max-width: 768px)");
  const prefersDark = useMediaQuery("(prefers-color-scheme: dark)");

  return isMobile ? <MobileLayout /> : <DesktopLayout />;
}
```

### useIntersectionObserver - 무한 스크롤과 Lazy Loading

```typescript
function useIntersectionObserver(
  options?: IntersectionObserverInit
) {
  const [entry, setEntry] = useState<IntersectionObserverEntry | null>(null);
  const [node, setNode] = useState<Element | null>(null);

  useEffect(() => {
    if (!node) return;

    const observer = new IntersectionObserver(
      ([entry]) => setEntry(entry),
      options
    );

    observer.observe(node);
    return () => observer.disconnect();
  }, [node, options?.threshold, options?.root, options?.rootMargin]);

  return { ref: setNode, entry, isIntersecting: entry?.isIntersecting ?? false };
}

// 무한 스크롤 사용 예시
function InfiniteList() {
  const { ref, isIntersecting } = useIntersectionObserver({
    threshold: 0.1,
  });
  const [items, setItems] = useState<Item[]>([]);

  useEffect(() => {
    if (isIntersecting) {
      loadMoreItems().then(newItems =>
        setItems(prev => [...prev, ...newItems])
      );
    }
  }, [isIntersecting]);

  return (
    <div>
      {items.map(item => <ItemCard key={item.id} item={item} />)}
      <div ref={ref} style={{ height: 1 }} /> {/* 감지 대상 */}
    </div>
  );
}
```

### useFetch - 데이터 페칭 추상화 (교육 목적)

실무에서는 TanStack Query(React Query)를 사용하지만, 학습 목적으로 직접 구현해보는 것이 이해에 도움됩니다.

```typescript
interface UseFetchResult<T> {
  data: T | null;
  error: string | null;
  isLoading: boolean;
  refetch: () => void;
}

function useFetch<T>(url: string): UseFetchResult<T> {
  const [data, setData] = useState<T | null>(null);
  const [error, setError] = useState<string | null>(null);
  const [isLoading, setIsLoading] = useState(true);
  const [trigger, setTrigger] = useState(0);

  const refetch = useCallback(() => setTrigger(prev => prev + 1), []);

  useEffect(() => {
    let cancelled = false;
    setIsLoading(true);

    fetch(url)
      .then(res => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json();
      })
      .then(data => {
        if (!cancelled) {
          setData(data);
          setError(null);
        }
      })
      .catch(err => {
        if (!cancelled) {
          setError(err.message);
        }
      })
      .finally(() => {
        if (!cancelled) setIsLoading(false);
      });

    return () => { cancelled = true; };
  }, [url, trigger]);

  return { data, error, isLoading, refetch };
}
```

이 커스텀 훅을 보면 데이터 페칭에 얼마나 많은 보일러플레이트가 필요한지 알 수 있습니다. 이것이 TanStack Query 같은 라이브러리가 인기 있는 이유입니다. 캐싱, 재시도, 옵티미스틱 업데이트, 무한 쿼리 등까지 고려하면 직접 구현하는 것은 비현실적입니다.

## Hooks 규칙과 ESLint 설정

React Hooks에는 반드시 지켜야 할 두 가지 규칙이 있습니다.

1. **최상위에서만 호출**: 조건문, 반복문, 중첩 함수 안에서 호출하면 안 됩니다. React는 Hook 호출 순서에 의존하여 상태를 관리하기 때문입니다.
2. **React 함수에서만 호출**: 일반 JavaScript 함수가 아닌 React 컴포넌트나 커스텀 Hook(이름이 `use`로 시작하는 함수)에서만 호출합니다.

```typescript
// 잘못된 패턴
function Component({ isLoggedIn }) {
  if (isLoggedIn) {
    const [user, setUser] = useState(null); // 조건문 안에서 Hook 호출!
  }
}

// 올바른 패턴
function Component({ isLoggedIn }) {
  const [user, setUser] = useState(null); // 항상 호출

  // 조건 로직은 Hook 호출 이후에
  if (!isLoggedIn) return <LoginPage />;
  return <Dashboard user={user} />;
}
```

ESLint의 `eslint-plugin-react-hooks` 플러그인을 사용하면 이 규칙 위반을 자동으로 감지할 수 있습니다. 이 플러그인의 `exhaustive-deps` 규칙은 useEffect의 의존성 배열 누락도 감지해주므로 반드시 활성화하세요. 저희 팀에서는 이 규칙을 `error` 수준으로 설정하여 CI에서 빌드가 실패하도록 합니다.

## 2026년 기준 Hooks 모범 사례 체크리스트

마지막으로, 저희 팀에서 사용하는 Hooks 코드 리뷰 체크리스트를 공유합니다.

| 항목 | 확인 사항 |
|------|---------|
| useState | 함수형 업데이트를 사용하는가? 초기값이 비싼 계산이면 함수로 감쌌는가? |
| useEffect | 클린업 함수가 필요한 곳에 있는가? 경쟁 조건을 방지했는가? |
| useEffect | 불필요한 useEffect는 아닌가? 렌더링 중 계산으로 대체 가능한가? |
| useMemo/useCallback | 실제로 성능 이득이 있는 곳에만 사용했는가? |
| useRef | DOM ref에 null 타입을 포함했는가? |
| useContext | Provider value를 메모이제이션했는가? |
| 커스텀 훅 | `use` 접두어를 사용하는가? 반환 타입이 명확한가? |
| 의존성 배열 | ESLint exhaustive-deps 경고가 없는가? |

## 마무리: Hooks는 도구일 뿐, 핵심은 설계 감각

React Hooks는 컴포넌트 로직을 직관적이고 재사용 가능하게 만들어줍니다. `useState`, `useEffect`, `useRef`만 확실히 이해해도 대부분의 React 개발이 가능합니다. 나머지 Hook과 커스텀 Hook은 필요한 상황이 생겼을 때 학습하면 됩니다.

핵심은 각 Hook의 목적을 이해하고, 적절한 상황에서 사용하는 것입니다. 무분별하게 최적화 Hook을 남발하기보다는 먼저 간단하게 구현하고, React DevTools의 Profiler로 실제 성능 병목을 확인한 후 최적화하는 접근이 바람직합니다.

마지막으로, Hooks를 잘 사용하는 것보다 중요한 것은 **컴포넌트를 잘 설계하는 것**입니다. 작고 단일 책임을 가진 컴포넌트를 만들면 Hooks의 복잡도도 자연스럽게 낮아집니다. 하나의 컴포넌트에 useState 5개, useEffect 3개가 있다면, 그건 Hooks의 문제가 아니라 컴포넌트 분리의 문제입니다.

다음 글에서는 커스텀 훅을 활용한 실전 폼 관리 패턴에 대해 더 깊이 다루겠습니다.
