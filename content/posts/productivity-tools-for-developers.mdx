개발자에게 좋은 도구는 초능력과 같다. 같은 작업이라도 어떤 도구를 쓰느냐에 따라 소요 시간이 5배 이상 차이날 수 있다. 나는 지난 3년간 수십 개의 도구를 시도해봤고, 그중에서 살아남아 매일 사용하는 10가지를 이 글에서 공유한다.

단순히 "이런 앱이 있다"가 아니라, 내가 실제로 어떻게 설정하고 사용하는지, 그리고 왜 다른 대안 대신 이것을 선택했는지를 상세하게 다룬다. 무료 도구도 많으니 부담 없이 시도해보시기 바란다.

## 1. 터미널: Warp

터미널은 개발자의 기본 도구다. macOS 기본 터미널, iTerm2, Hyper 등 다양한 선택지가 있지만, 나는 Warp를 메인 터미널로 사용하고 있다.

### 왜 Warp인가?

iTerm2를 5년 넘게 사용하다가 Warp로 전환한 지 1년이 넘었다. 전환 이유는 세 가지다.

첫째, AI 기반 명령어 제안이 실제로 유용하다. `#`을 입력하고 자연어로 설명하면 해당하는 명령어를 생성해준다. 예를 들어 `# find all files larger than 100MB`라고 입력하면 `find / -type f -size +100M`을 제안해준다. 복잡한 find, awk, sed 명령어를 외울 필요가 줄었다.

둘째, 블록 기반 출력이 혁신적이다. 각 명령어의 입력과 출력이 하나의 블록으로 묶여서, 긴 출력 결과를 접거나 복사하기가 편하다. 이전 명령어의 출력을 찾으려고 위로 무한 스크롤하는 일이 없어졌다.

셋째, 기본 테마와 폰트가 예쁘다. 사소해 보이지만, 하루 종일 보는 터미널이 예쁘면 기분이 좋다. 물론 커스터마이징도 자유롭다.

### Warp 실전 설정

내가 실제로 사용하는 Warp 설정을 공유한다.

**쉘:** Zsh + Oh My Zsh. Warp와 Oh My Zsh는 호환이 잘 되고, Warp의 프롬프트 탐지가 대부분의 Oh My Zsh 테마와 잘 동작한다.

**주요 플러그인:** zsh-autosuggestions (타이핑 중 이전 명령어 제안), zsh-syntax-highlighting (명령어 문법 색상 표시), z (자주 가는 디렉토리 빠르게 이동).

**Warp 설정 팁:** Settings > Features에서 "Block border"를 활성화하면 블록 구분이 더 명확해진다. "Autocompletion"에서 AI 제안 모델을 최신으로 설정하면 더 정확한 제안을 받을 수 있다.

### 대안 비교

| 터미널 | 장점 | 단점 | 가격 |
|--------|------|------|------|
| Warp | AI 제안, 블록 UI, 빠름 | macOS 전용(Linux 베타), 계정 필요 | 무료/팀 유료 |
| iTerm2 | 안정적, 커스터마이징 풍부 | AI 없음, 클래식 UI | 무료 |
| Alacritty | 극한의 속도, GPU 렌더링 | 탭 없음, 설정 파일 수동 편집 | 무료 |
| Kitty | 빠름, 이미지 표시 지원 | 학습 곡선 | 무료 |

## 2. 코드 에디터: VS Code + 핵심 익스텐션

VS Code가 개발자의 표준 에디터라는 것은 새삼 말할 필요가 없을 것이다. 중요한 것은 어떤 익스텐션을 쓰고, 어떻게 설정하느냐다.

### 필수 익스텐션 5선

**Error Lens:** 에러를 코드 라인 옆에 인라인으로 표시해준다. 기본 VS Code는 빨간 밑줄만 그어주고 호버해야 에러 메시지를 볼 수 있는데, Error Lens를 쓰면 코드 옆에 바로 메시지가 보인다. 이 익스텐션을 설치한 후 디버깅 시간이 눈에 띄게 줄었다.

**GitLens:** Git 히스토리를 코드 레벨에서 볼 수 있게 해준다. 각 라인이 언제, 누가, 왜 수정했는지를 바로 확인할 수 있다. "이 코드를 왜 이렇게 짰지?"라는 궁금증이 생길 때, 해당 커밋의 PR을 바로 열어볼 수 있어서 유용하다.

**Pretty TypeScript Errors:** TypeScript의 에러 메시지는 악명 높게 길고 읽기 어렵다. 이 익스텐션은 중첩된 타입 에러를 트리 형태로 보기 좋게 변환해준다. TypeScript 프로젝트에서는 필수다.

**Auto Rename Tag:** HTML/JSX 태그의 여는 태그를 수정하면 닫는 태그도 자동으로 수정된다. 사소하지만, 매일 수십 번 하는 작업이라 누적 시간이 크다.

**GitHub Copilot:** AI 코딩 어시스턴트. 월 $10이지만 그만한 가치가 있다. 특히 반복적인 코드 패턴이나 테스트 코드 작성에서 시간을 크게 절약해준다.

### 핵심 설정

내가 실제로 사용하는 VS Code 설정 중 생산성에 직접적인 영향을 주는 것들이다.

```json
{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.bracketPairColorization.enabled": true,
  "editor.guides.bracketPairs": "active",
  "editor.stickyScroll.enabled": true,
  "editor.minimap.enabled": false,
  "editor.fontSize": 14,
  "editor.lineHeight": 1.8,
  "workbench.colorTheme": "One Dark Pro Darker",
  "terminal.integrated.defaultProfile.osx": "zsh"
}
```

특히 `editor.stickyScroll.enabled`는 비교적 최근 기능인데, 현재 스코프의 함수명/클래스명이 상단에 고정되어 표시된다. 긴 파일에서 "지금 어떤 함수 안에 있지?"를 확인하기 위해 위로 스크롤하는 일이 없어진다.

`editor.minimap.enabled: false`는 취향 차이가 있지만, 나는 미니맵이 화면 공간을 낭비한다고 느껴서 끈다. 그 공간을 코드에 쓰는 게 낫다.

## 3. Git GUI: GitKraken

Git은 CLI로 충분하다고 말하는 사람이 많지만, 복잡한 브랜치 관리나 충돌 해결에서는 GUI가 확실히 편하다. 나는 GitKraken을 유료 구독하고 있다.

### 왜 GitKraken인가

실제로 사용해보니, 가장 큰 가치는 시각적 브랜치 그래프에 있다. 10개 이상의 브랜치가 있는 프로젝트에서 각 브랜치의 상태, 머지 히스토리, 충돌 가능성을 한눈에 파악할 수 있다. CLI의 `git log --graph`보다 훨씬 직관적이다.

충돌 해결(Merge Conflict Resolution) 도구도 뛰어나다. 3-way merge 뷰어가 내장되어 있어서, 왼쪽(mine), 오른쪽(theirs), 아래(결과)를 보면서 충돌을 해결할 수 있다. VS Code의 충돌 해결 기능보다 복잡한 충돌에서 더 편하다.

인터랙티브 리베이스도 GUI로 하면 훨씬 안전하다. 커밋을 드래그해서 순서를 변경하거나, 스쿼시할 커밋을 선택하는 것이 직관적이다.

### 대안

| Git GUI | 장점 | 단점 | 가격 |
|---------|------|------|------|
| GitKraken | 예쁜 UI, 강력한 머지 툴 | 무거움, 유료 | 무료/Pro $4.95/월 |
| Fork | 가볍고 빠름, 인터랙티브 리베이스 | 기능 적음 | 무료(macOS/Windows) |
| Sourcetree | 무료, Atlassian 통합 | 느림, UI 구식 | 무료 |
| lazygit | 터미널 TUI, 빠름 | GUI 아닌 TUI | 무료 |

솔직히 무료 대안으로는 Fork를 추천한다. 가볍고 빠르면서도 필수 기능은 다 있다.

## 4. API 테스트: Bruno

Postman을 오래 사용하다가 Bruno로 전환했다. Postman이 점점 무거워지고, 클라우드 동기화를 강제하는 방향으로 가면서 불만이 쌓였기 때문이다.

### Bruno의 장점

가장 큰 장점은 파일 기반이라는 것이다. API 컬렉션이 파일 시스템에 저장되므로, Git으로 버전 관리할 수 있다. 팀원들과 API 컬렉션을 공유할 때 Postman처럼 클라우드에 올릴 필요 없이, Git 저장소에 포함시키면 된다. 우리 팀에서는 프로젝트 루트에 `/api-collections` 디렉토리를 만들어서 Bruno 파일을 관리한다.

가볍다는 것도 큰 장점이다. Postman은 Electron 기반이라 메모리를 800MB 이상 먹는 경우가 있는데, Bruno는 200MB 이하로 쾌적하다.

Bru 문법이라는 자체 문법으로 API 요청을 정의하는데, 간단하고 읽기 쉽다. 환경 변수, 사전/사후 스크립트도 지원해서 Postman의 주요 기능은 대부분 커버된다.

### 대안 비교

| 도구 | 장점 | 단점 | 가격 |
|------|------|------|------|
| Bruno | 파일 기반, 가벼움, Git 친화적 | 생태계 작음 | 무료/Golden $19 |
| Postman | 기능 풍부, 큰 생태계 | 무거움, 클라우드 강제 | 무료/유료 |
| Insomnia | 깔끔한 UI, GraphQL 지원 | Kong 인수 후 불안정 | 무료/유료 |
| HTTPie | CLI 기반, 스크립팅 용이 | GUI 없음 | 무료 |
| Thunder Client | VS Code 통합 | 기능 제한적 | 무료/유료 |

## 5. 메모와 지식 관리: Obsidian

개발하면서 배운 것들을 체계적으로 기록하는 것은 생각보다 중요하다. "분명 전에 해결했었는데, 어떻게 했더라?" 이런 경험이 있다면 메모 도구가 필요하다.

### 왜 Obsidian인가

Notion에서 Obsidian으로 전환한 이유는 두 가지다. 첫째, 마크다운 파일 기반이라 데이터가 내 로컬에 있다. Notion은 서비스가 중단되면 데이터 접근이 어려워지지만, Obsidian은 그냥 마크다운 파일이라 어떤 에디터로든 열 수 있다.

둘째, 오프라인에서도 완벽하게 동작한다. 비행기에서, 지하철에서, 인터넷이 불안정한 카페에서도 아무 문제 없이 사용할 수 있다.

### 개발자를 위한 Obsidian 활용법

내가 실제로 사용하는 메모 구조를 공유한다.

```
vault/
├── daily/          # 일일 메모 (업무 로그)
├── projects/       # 프로젝트별 메모
├── til/            # Today I Learned
├── debug-log/      # 디버깅 기록
├── snippets/       # 코드 스니펫
└── templates/       # 템플릿
```

**TIL (Today I Learned):** 매일 하나씩 배운 것을 기록한다. "React 19에서 use() 훅으로 Promise를 직접 다룰 수 있다", "PostgreSQL에서 JSONB 인덱스는 GIN보다 GiST가 작은 문서에 적합하다" 같은 짧은 메모가 쌓이면 나만의 위키가 된다.

**디버깅 로그:** 어려운 버그를 해결한 후 "증상 - 원인 - 해결법 - 교훈"을 기록한다. 나중에 비슷한 문제를 겪었을 때 검색하면 바로 찾을 수 있다. 이 습관 덕분에 같은 버그로 시간을 낭비하는 일이 크게 줄었다.

**추천 플러그인:** Dataview (메모를 데이터베이스처럼 쿼리), Templater (커스텀 템플릿), Calendar (일일 메모 달력 뷰), Git (자동 백업).

## 6. 시간 관리: Toggl Track

"오늘 하루 뭐했지?" 퇴근할 때 이런 생각이 드는 날이 있다. 열심히 일했는데 결과물이 없는 느낌. 이런 문제를 해결하기 위해 시간 추적을 시작했다.

### 사용법

Toggl Track은 타이머 기반의 시간 추적 도구다. 작업을 시작할 때 타이머를 시작하고, 끝나면 정지하면 된다. 이것을 2년간 꾸준히 했더니 몇 가지 놀라운 패턴이 발견됐다.

내가 직접 경험한 발견들을 공유한다. 첫째, 하루 8시간 근무 중 실제 집중 코딩 시간은 평균 4~5시간이었다. 나머지는 회의, 코드 리뷰, 슬랙 대화, 이메일 등에 쓰이고 있었다. 이것을 인식한 후 회의 시간을 줄이려는 노력을 했고, 실제 코딩 시간이 30분 정도 늘었다.

둘째, 오전 10시~12시가 나의 골든 타임이었다. 이 시간대에 가장 집중력이 높고 생산성이 좋았다. 이것을 알게 된 후, 이 시간대에는 회의를 잡지 않고 가장 어려운 작업에 집중하도록 일정을 조정했다.

셋째, 컨텍스트 스위칭(작업 전환)이 생각보다 큰 비용이었다. 한 작업에서 다른 작업으로 전환할 때마다 평균 15분의 재집중 시간이 필요했다. 하루에 5번 전환하면 1시간 이상이 낭비되는 셈이다.

## 7. 클립보드 관리: Raycast (내장 기능)

클립보드 히스토리는 한번 사용하면 없이는 못 사는 기능이다. 이전에는 Alfred + Clipboard History를 사용했는데, Raycast로 전환하면서 별도 앱이 필요 없어졌다.

Raycast의 클립보드 히스토리는 `Cmd + Shift + V`로 열 수 있다. 최근 복사한 텍스트, 이미지, 링크 목록이 표시되고, 검색도 가능하다. 개발하면서 여러 변수값이나 URL을 번갈아 붙여넣어야 할 때 특히 유용하다.

## 8. 스크린샷과 녹화: CleanShot X

기본 macOS 스크린샷 기능도 나쁘지 않지만, CleanShot X는 개발자에게 필요한 추가 기능들이 있다.

스크롤 캡처(긴 페이지 전체 캡처), 캡처 후 바로 어노테이션(화살표, 텍스트, 모자이크 처리), GIF/비디오 녹화, 그리고 캡처한 이미지를 바로 클라우드에 올려서 링크를 생성하는 기능이 있다.

실제로 사용해보니, 버그 리포트를 할 때 가장 유용하다. 화면을 캡처하고, 문제 부분에 화살표를 그리고, 링크를 생성해서 슬랙에 붙여넣으면 끝이다. 이 워크플로우가 없었을 때는 캡처 → 미리보기 앱으로 열기 → 편집 → 저장 → 슬랙에 파일 첨부 의 과정을 거쳤다.

## 9. 집중 모드: Focus (macOS 내장 + 커스텀)

macOS의 집중 모드를 개발 작업에 최적화해서 사용하고 있다.

**코딩 집중 모드:** 슬랙, 메일, SNS 알림을 모두 차단. 허용하는 앱은 VS Code, Warp, Chrome(개발용)만. 이 모드에서는 화면 상단 메뉴바의 아이콘이 보라색으로 바뀌어서, 스스로 "지금 집중 시간이다"를 인식할 수 있다.

**회의 모드:** 화상회의 앱과 슬랙만 허용. 코딩 관련 알림은 차단.

자동화도 설정해두었다. 매일 오전 10시~12시에 코딩 집중 모드가 자동 활성화되고, 캘린더에 회의가 있으면 회의 모드로 자동 전환된다.

## 10. 패스워드 관리: 1Password

개발자는 수십 개의 서비스 계정을 관리해야 한다. GitHub, AWS, Vercel, npm, Docker Hub, 각종 API 키 등. 이것을 안전하게 관리하려면 패스워드 매니저는 필수다.

1Password를 선택한 이유는 CLI 통합이 뛰어나기 때문이다. `op` CLI를 사용하면 터미널에서 직접 비밀 정보를 조회할 수 있다. 환경 변수에 API 키를 하드코딩하는 대신, 1Password 참조를 사용할 수 있다.

```bash
# .env 파일에 직접 키를 넣는 대신
# DATABASE_URL=postgres://user:password@host/db

# 1Password 참조를 사용
DATABASE_URL=op://Development/Database/connection-string
```

SSH 키 관리도 1Password에서 한다. SSH 에이전트가 내장되어 있어서, GitHub에 SSH로 접속할 때 1Password가 자동으로 키를 제공한다. 더 이상 `~/.ssh` 디렉토리에 키 파일을 관리할 필요가 없다.

## 도구 선택 시 고려해야 할 기준

많은 도구를 시도해보면서 깨달은 도구 선택의 기준을 공유한다.

### 1. 문제를 명확히 정의하기

"좋은 도구가 있대"라는 이유로 도구를 도입하면 안 된다. "지금 어떤 문제가 있는가?"를 먼저 정의해야 한다. 문제가 없는데 도구를 도입하면 오히려 복잡성만 증가한다.

### 2. 전환 비용 고려하기

새 도구를 배우는 시간, 기존 데이터를 마이그레이션하는 시간, 팀원들에게 전파하는 시간 등 전환 비용을 고려해야 한다. 10%의 효율 개선을 위해 2주간의 전환 비용이 든다면, 장기적으로 봐야 한다.

### 3. 무료 버전으로 충분한가?

대부분의 도구는 무료 버전으로도 충분하다. 유료 버전의 추가 기능이 정말 필요한지 신중하게 판단하자. 나는 현재 월 약 $30 정도를 도구 구독에 사용하고 있는데, 이것은 투자 대비 충분한 생산성 향상이 있기 때문이다.

## 보너스: 도구 간 연동으로 시너지 만들기

개별 도구도 좋지만, 도구 간 연동을 설정하면 생산성이 한 단계 더 올라간다. 내가 실제로 사용하는 연동 워크플로우를 공유한다.

### GitHub + VS Code + Warp 연동

GitHub PR을 리뷰할 때, VS Code의 GitHub Pull Requests 익스텐션을 활용하면 IDE 안에서 직접 PR을 확인하고 코멘트를 달 수 있다. 코드를 로컬에서 확인하고 싶으면 Warp에서 `gh pr checkout 123`으로 바로 해당 브랜치를 체크아웃할 수 있다. GitHub CLI(`gh`)와 Warp의 AI 제안이 결합되면, Git 관련 작업의 효율이 크게 올라간다.

### Obsidian + Daily Note + Toggl Track 연동

매일 아침 Obsidian에서 Daily Note를 생성하고, 그날의 작업 계획을 적는다. 각 작업을 시작할 때 Toggl Track의 타이머를 시작한다. 퇴근 전에 Toggl의 기록을 보면서 Daily Note에 실제 소요 시간과 회고를 추가한다. 이 루틴을 3개월간 유지했더니, 시간 추정 능력이 눈에 띄게 향상됐다. 예상 시간과 실제 시간의 차이가 처음에는 2배 이상이었는데, 지금은 20~30% 이내로 줄었다.

### Raycast + 모든 것

Raycast의 진정한 가치는 다른 도구들의 허브 역할이다. 내가 설정해둔 Raycast 명령어를 몇 가지 소개한다.

단축키 `Cmd + Shift + G`로 GitHub 저장소를 검색하고 바로 열 수 있다. `Cmd + Shift + J`로 최근 Jira 이슈를 조회할 수 있다. `Cmd + Shift + T`로 Toggl Track의 타이머를 시작하거나 정지할 수 있다. 이런 단축키를 설정해두면 앱을 전환하는 시간이 크게 줄어든다. 모든 작업의 시작점이 Raycast가 되는 셈이다.

### 커스텀 스크립트로 자동화

Raycast Script Commands나 Alfred Workflows를 활용하면 반복적인 작업을 자동화할 수 있다. 예를 들어 나는 "새 프로젝트 시작" 스크립트를 만들어두었는데, 이 스크립트를 실행하면 GitHub에 새 저장소 생성, 로컬에 클론, VS Code에서 열기, 기본 디렉토리 구조 생성이 자동으로 이루어진다. 이전에는 이 과정에 10분이 걸렸는데, 지금은 30초면 끝난다.

## 도구 정리 팁: 도구 과잉을 피하는 방법

도구를 많이 알게 되면 오히려 도구 과잉에 빠질 수 있다. 비슷한 기능의 도구를 여러 개 사용하면서 오히려 복잡성이 증가하는 경우다. 내가 적용하는 원칙은 "하나의 문제에 하나의 도구"이다.

메모 앱을 예로 들면, Notion, Obsidian, Bear, Apple Notes를 동시에 사용하면 "이 메모를 어디에 적었더라?"라는 문제가 생긴다. 하나를 정하고 모든 메모를 거기에 집중하는 것이 훨씬 효율적이다.

분기마다 한 번씩 "도구 정리의 날"을 가져보는 것도 추천한다. 현재 사용 중인 모든 구독 서비스와 도구를 나열하고, 지난 3개월간 실제로 사용하지 않은 것은 과감히 해지하거나 삭제한다. 나도 이 과정에서 월 $15 정도의 불필요한 구독을 정리한 적이 있다.

## 마치며

도구는 수단이지 목적이 아니다. 좋은 도구를 설치하는 것보다 중요한 것은, 그 도구를 습관적으로 잘 활용하는 것이다. 한 번에 모든 도구를 바꾸려 하지 말고, 하나씩 시도해보면서 자신의 워크플로우에 맞는 것을 찾아가시기 바란다.

이 글에서 소개한 도구 외에도 좋은 도구가 있다면 댓글로 공유해주시기 바란다. 서로의 경험을 나누는 것이 가장 좋은 학습이다.
