TypeScript는 JavaScript에 정적 타입 시스템을 추가한 프로그래밍 언어입니다. Microsoft가 2012년에 발표한 이후 꾸준히 성장하여, 2026년 현재 대부분의 프론트엔드 프로젝트에서 사실상 표준으로 사용되고 있습니다.

하지만 "TypeScript를 써야 한다"는 것과 "실제로 팀에 도입하는 것"은 전혀 다른 문제입니다. 이 글에서는 단순히 "왜 써야 하는가"를 넘어서, 실제로 팀 프로젝트에서 JavaScript에서 TypeScript로 전환한 경험과 그 과정에서 겪은 시행착오를 공유합니다.

## 우리 팀이 JS에서 TS로 전환한 이야기

### 전환 전 상황

2024년 초, 저희 팀은 React + JavaScript로 작성된 약 3만 줄 규모의 어드민 대시보드를 운영하고 있었습니다. 코드베이스가 커지면서 몇 가지 문제가 심각해졌습니다:

- **월평균 타입 관련 런타임 에러**: 12-15건 (Sentry 기준)
- **새 팀원의 온보딩 기간**: 평균 3주 (코드 구조 파악에 대부분의 시간 소요)
- **API 응답 구조 변경 시 놓치는 곳**: 매번 1-2곳에서 장애 발생
- **코드 리뷰에서 "이 함수 파라미터가 뭔가요?" 질문**: 주 5회 이상

CTO가 "타입스크립트 도입하자"고 했을 때, 솔직히 반대 의견도 있었습니다. "지금도 잘 돌아가는데 왜 바꿔요?" "학습 비용이 너무 크지 않나요?"

### 전환 과정 (3개월)

**1단계: 점진적 도입 결정 (1주차)**

전체 코드를 한 번에 바꾸는 Big Bang 방식이 아니라, 점진적으로 전환하기로 했습니다.

```json
// tsconfig.json - 초기 설정 (느슨하게 시작)
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "strict": false,          // 처음에는 strict 끔
    "allowJs": true,          // JS 파일 허용
    "checkJs": false,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "jsx": "react-jsx"
  },
  "include": ["src/**/*"]
}
```

**2단계: 새 파일은 무조건 .tsx (1-4주차)**

기존 파일은 건드리지 않고, 새로 만드는 파일만 TypeScript로 작성했습니다. 이 단계에서 팀원들이 TypeScript에 익숙해졌습니다.

**3단계: 공통 타입 정의 (2-3주차)**

API 응답 타입을 먼저 정의했습니다. 이것만으로도 효과가 즉각적이었습니다.

```typescript
// types/api.ts - 실제 프로덕션에서 사용한 타입 정의
interface ApiResponse<T> {
  success: boolean;
  data: T;
  error?: {
    code: string;
    message: string;
    details?: Record<string, string[]>;
  };
  meta?: {
    page: number;
    totalPages: number;
    totalItems: number;
  };
}

interface User {
  id: number;
  email: string;
  name: string;
  role: 'admin' | 'manager' | 'viewer';
  department: string;
  createdAt: string;    // ISO 8601
  lastLoginAt: string | null;
  permissions: Permission[];
}

type Permission =
  | 'dashboard:read'
  | 'dashboard:write'
  | 'users:read'
  | 'users:write'
  | 'reports:read'
  | 'reports:export';

// API 함수에 타입 적용
async function fetchUsers(
  page: number,
  filters?: { role?: User['role']; department?: string }
): Promise<ApiResponse<User[]>> {
  const params = new URLSearchParams({
    page: String(page),
    ...(filters?.role && { role: filters.role }),
    ...(filters?.department && { department: filters.department }),
  });
  const response = await fetch(`/api/users?${params}`);
  return response.json();
}
```

**4단계: 핵심 모듈 전환 (4-8주차)**

가장 버그가 많았던 모듈부터 전환했습니다: 결제 처리, 권한 관리, 데이터 시각화 순서.

**5단계: strict 모드 활성화 (8-12주차)**

점차 strict 옵션을 하나씩 켰습니다.

```json
// tsconfig.json - 최종 설정
{
  "compilerOptions": {
    "strict": true,                    // 전체 strict 활성화
    "noUncheckedIndexedAccess": true,  // 배열/객체 인덱스 접근 안전성
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "exactOptionalPropertyTypes": true
  }
}
```

### 전환 후 결과 (6개월 뒤 측정)

숫자로 보면 효과가 명확했습니다:

| 지표 | JS 시절 | TS 전환 후 | 변화 |
|------|---------|-----------|------|
| 월평균 타입 관련 런타임 에러 | 12-15건 | 1-2건 | **87% 감소** |
| 새 팀원 온보딩 기간 | 3주 | 1.5주 | **50% 단축** |
| API 변경 시 누락 장애 | 매번 발생 | 0건 | **100% 제거** |
| 코드 리뷰 "이게 뭔가요?" 질문 | 주 5회+ | 주 1회 미만 | **80% 감소** |
| 평균 PR 리뷰 시간 | 45분 | 25분 | **44% 단축** |

코드 리뷰 시간이 줄어든 것은 의외의 효과였습니다. 타입 정의가 문서 역할을 하니까, 리뷰어가 "이 함수가 뭘 하는 건지" 파악하는 시간이 크게 줄었습니다.

## 왜 TypeScript인가?

### 버그를 미리 잡아줍니다

JavaScript의 가장 큰 문제점 중 하나는 런타임에서야 에러를 발견할 수 있다는 것입니다.

```javascript
// JavaScript - 실행해봐야 에러를 발견
function greet(user) {
  return `Hello, ${user.name.toUpperCase()}!`;
}

greet({ nama: "Jay" }); // 오타! 런타임 에러 발생
```

TypeScript에서는 이 오류를 코드 작성 시점에 발견합니다.

```typescript
interface User {
  name: string;
  email: string;
}

function greet(user: User): string {
  return `Hello, ${user.name.toUpperCase()}!`;
}

greet({ nama: "Jay" }); // 컴파일 에러! 'nama'는 'User' 타입에 없음
```

에디터에서 빨간 줄로 바로 표시되므로 실행하기 전에 수정할 수 있습니다.

### 자동완성이 강력해집니다

TypeScript의 타입 정보 덕분에 에디터의 자동완성이 매우 정확해집니다. 객체의 속성, 함수의 파라미터, 반환값을 자동으로 제안해주므로 문서를 매번 확인할 필요가 줄어듭니다.

### 리팩토링이 안전해집니다

함수 이름을 변경하거나, 파라미터를 추가하거나, 데이터 구조를 수정할 때 TypeScript가 영향 받는 모든 곳을 알려줍니다. 대규모 코드베이스에서 리팩토링할 때 "어딘가에서 깨질 수 있다"는 불안감 없이 작업할 수 있습니다.

### 문서 역할을 합니다

타입 정의 자체가 코드의 문서 역할을 합니다.

```typescript
interface BlogPost {
  title: string;
  description: string;
  date: string;
  category: "tech" | "review" | "tutorial" | "life";
  tags: string[];
  published: boolean;
  thumbnail?: string;  // 선택적 필드
}
```

이 타입 정의만 보면 블로그 글이 어떤 구조인지 바로 이해할 수 있습니다.

## TypeScript 시작하기

### 프로젝트에 TypeScript 추가

기존 JavaScript 프로젝트에 TypeScript를 추가하는 것은 간단합니다.

```bash
npm install -D typescript @types/node
npx tsc --init
```

`tsconfig.json` 파일이 생성됩니다. 주요 설정을 확인합니다.

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src/**/*"]
}
```

### Next.js에서는 더 쉽습니다

Next.js 프로젝트에서는 별도 설정 없이 바로 TypeScript를 사용할 수 있습니다.

```bash
npx create-next-app@latest my-app --typescript
```

## 고급 패턴 — 프로덕션에서 진짜 쓰는 것들

기본 문법을 넘어서, 실전에서 자주 사용하는 고급 패턴들을 소개합니다. 이것들을 알면 TypeScript의 진가를 느낄 수 있습니다.

### Discriminated Unions (판별 유니온)

API 응답이나 상태 관리에서 가장 많이 쓰이는 패턴입니다. 저희 팀에서 가장 효과가 좋았던 패턴이기도 합니다.

```typescript
// Before: 모든 상태를 하나의 인터페이스에 우겨넣음
// → isLoading이 true인데 data에 접근하는 실수 발생
interface BadState {
  isLoading: boolean;
  data?: User[];
  error?: string;
}

// After: 판별 유니온으로 상태를 명확히 분리
type UserListState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: User[]; totalCount: number }
  | { status: 'error'; error: string; retryCount: number };

function renderUserList(state: UserListState) {
  switch (state.status) {
    case 'idle':
      return <p>검색 버튼을 눌러주세요</p>;
    case 'loading':
      return <Spinner />;
    case 'success':
      // 이 블록에서 state.data와 state.totalCount가 자동 추론됨!
      return <UserTable users={state.data} total={state.totalCount} />;
    case 'error':
      // 이 블록에서 state.error와 state.retryCount가 자동 추론됨!
      return <ErrorMessage message={state.error} retries={state.retryCount} />;
  }
}
```

이 패턴을 적용한 후 "undefined is not an object" 에러가 거의 사라졌습니다.

### Template Literal Types (템플릿 리터럴 타입)

CSS-in-JS나 이벤트 이름 같은 문자열 패턴을 타입으로 강제할 수 있습니다.

```typescript
// 이벤트 핸들러 이름을 타입으로 제한
type EventName = `on${Capitalize<'click' | 'hover' | 'focus' | 'blur'>}`;
// 결과: 'onClick' | 'onHover' | 'onFocus' | 'onBlur'

// CSS 단위를 포함하는 값
type CSSLength = `${number}${'px' | 'rem' | 'em' | '%' | 'vh' | 'vw'}`;

function setWidth(width: CSSLength) {
  // ...
}

setWidth('100px');  // OK
setWidth('2.5rem'); // OK
setWidth('100');    // 에러! 단위가 없음
setWidth('100kg');  // 에러! 유효하지 않은 단위

// API 엔드포인트 패턴
type ApiEndpoint = `/api/${'users' | 'posts' | 'comments'}${'' | `/${number}`}`;
// 결과: '/api/users' | '/api/users/123' | '/api/posts' | ...
```

### Conditional Types (조건부 타입)

입력 타입에 따라 출력 타입이 달라지는 경우에 사용합니다.

```typescript
// 배열이면 요소 타입을, 아니면 그대로 반환
type UnpackArray<T> = T extends (infer U)[] ? U : T;

type A = UnpackArray<string[]>;   // string
type B = UnpackArray<number>;     // number

// 실전 예시: API 응답에서 단일/복수 자동 추론
type ApiMethod = 'getOne' | 'getMany';

type ApiResult<M extends ApiMethod, T> =
  M extends 'getOne' ? T :
  M extends 'getMany' ? { items: T[]; total: number } :
  never;

// 사용
function fetchData<M extends ApiMethod>(
  method: M,
  endpoint: string
): Promise<ApiResult<M, User>> {
  // ...
}

// fetchData('getOne', '/users/1')의 반환 타입은 Promise<User>
// fetchData('getMany', '/users')의 반환 타입은 Promise<{ items: User[]; total: number }>
```

### Branded Types (브랜드 타입)

같은 string이나 number지만 의미가 다른 값을 구분할 때 사용합니다. 이것은 실수로 잘못된 ID를 넘기는 버그를 막아줍니다.

```typescript
// 같은 number지만 의미가 다른 ID들
type UserId = number & { readonly __brand: 'UserId' };
type PostId = number & { readonly __brand: 'PostId' };
type CommentId = number & { readonly __brand: 'CommentId' };

// 생성 함수
function toUserId(id: number): UserId { return id as UserId; }
function toPostId(id: number): PostId { return id as PostId; }

// 사용
function deleteUser(userId: UserId) { /* ... */ }
function deletePost(postId: PostId) { /* ... */ }

const userId = toUserId(42);
const postId = toPostId(99);

deleteUser(userId);  // OK
deleteUser(postId);  // 컴파일 에러! PostId를 UserId에 넣을 수 없음
```

저희 팀에서는 이 패턴 도입 후 "잘못된 ID를 넘겨서 다른 리소스가 삭제되는" 사고가 완전히 사라졌습니다.

## 실전 패턴

### 타입 가드

런타임에서 타입을 좁혀나가는 패턴입니다.

```typescript
function processInput(input: string | number) {
  if (typeof input === "string") {
    // 이 블록에서 input은 string
    return input.toUpperCase();
  }
  // 여기서 input은 number
  return input * 2;

```

### 커스텀 타입 가드 (프로덕션 예시)

```typescript
// API 에러 응답을 구분하는 타입 가드
interface ApiError {
  success: false;
  error: { code: string; message: string };
}

interface ApiSuccess<T> {
  success: true;
  data: T;
}

type ApiResponse<T> = ApiSuccess<T> | ApiError;

function isApiError<T>(response: ApiResponse<T>): response is ApiError {
  return !response.success;
}

// 사용
const response = await fetchUsers();
if (isApiError(response)) {
  // response.error가 자동 추론됨
  showToast(response.error.message);
  return;
}
// 여기서 response.data가 자동 추론됨
renderUsers(response.data);
```

TypeScript 내장 유틸리티 타입을 활용하면 타입을 변환할 수 있습니다.

```typescript
interface User {

```

## TypeScript 흔한 실수 10가지 — 제가 다 해본 것들

### 실수 1: `any`를 남용하기

```typescript
// 나쁜 예: any는 TypeScript의 의미를 없앤다
function processData(data: any) {
  return data.foo.bar.baz; // 런타임 에러 가능성 100%
}

// 좋은 예: unknown을 쓰고 타입을 좁히기
function processData(data: unknown) {
  if (typeof data === 'object' && data !== null && 'foo' in data) {
    // 안전하게 접근
  }
}

// 더 좋은 예: 제네릭 사용
function processData<T extends { foo: { bar: { baz: string } } }>(data: T) {
  return data.foo.bar.baz; // 타입 안전
}
```

저희 팀에서는 ESLint의 `@typescript-eslint/no-explicit-any` 규칙을 "error"로 설정했습니다. 처음에 팀원들이 불만이 있었지만, 2주 후에는 모두 동의했습니다.

### 실수 2: 타입 단언(as) 남용하기

```typescript
// 나쁜 예
const user = response.data as User; // 실제로 User가 아닐 수 있음

// 좋은 예: 런타임 검증을 추가
function isUser(data: unknown): data is User {
  return (
    typeof data === 'object' &&
    data !== null &&
    'name' in data &&
    'email' in data &&
    typeof (data as any).name === 'string'
  );
}

const data = response.data;
if (isUser(data)) {
  // 안전하게 User로 사용
}
```

### 실수 3: 옵셔널 체이닝의 과도한 사용

```typescript
// 나쁜 예: 모든 곳에 ?를 붙이면 타입 시스템이 무력화됨
const name = user?.profile?.name?.first?.trim();

// 좋은 예: 타입으로 null 가능성을 명확히 하기
interface User {
  profile: UserProfile;     // 항상 존재
  settings?: UserSettings;  // 없을 수 있음
}

const name = user.profile.name.first.trim(); // profile은 항상 있으니까
const theme = user.settings?.theme ?? 'light'; // settings는 없을 수 있으니까
```

### 실수 4: Enum 대신 Union Type을 쓰지 않기

```typescript
// 비추천: enum은 번들 크기를 증가시키고 트리 셰이킹이 안 됨
enum Status {
  Active = 'active',
  Inactive = 'inactive',
  Pending = 'pending',
}

// 추천: const assertion + union type
const STATUS = {
  Active: 'active',
  Inactive: 'inactive',
  Pending: 'pending',
} as const;

type Status = typeof STATUS[keyof typeof STATUS];
// 결과: 'active' | 'inactive' | 'pending'
```

### 실수 5: 제네릭에 제약 조건 안 넣기

```typescript
// 나쁜 예: T가 무엇이든 될 수 있어서 .id에 접근 불가
function findById<T>(items: T[], id: number): T | undefined {
  return items.find(item => item.id === id); // 에러!
}

// 좋은 예: 제약 조건으로 T가 id를 가지고 있음을 보장
function findById<T extends { id: number }>(items: T[], id: number): T | undefined {
  return items.find(item => item.id === id); // OK
}
```

### 실수 6: 비동기 함수 반환 타입 생략

```typescript
// 나쁜 예: 반환 타입이 불명확
async function fetchUser(id: number) {
  const response = await fetch(`/api/users/${id}`);
  return response.json(); // 반환 타입: Promise<any>
}

// 좋은 예: 명시적 반환 타입
async function fetchUser(id: number): Promise<ApiResponse<User>> {
  const response = await fetch(`/api/users/${id}`);
  return response.json();
}
```

### 실수 7: 인덱스 시그니처 남용

```typescript
// 나쁜 예: 아무 키나 허용 → 타입 안전성 0
interface Config {
  [key: string]: any;
}

// 좋은 예: Record와 알려진 키를 조합
interface Config {
  apiUrl: string;
  timeout: number;
  retryCount: number;
  features: Record<string, boolean>; // 확장 가능한 부분만 유연하게
}
```

### 실수 8: 타입과 인터페이스 구분 없이 사용

```typescript
// 규칙: 객체 구조 → interface, 유니온/인터섹션/매핑 → type
interface UserProps {      // 객체 구조
  name: string;
  age: number;
}

type Theme = 'light' | 'dark';              // 유니온
type WithTimestamp<T> = T & { createdAt: string }; // 인터섹션/매핑
```

### 실수 9: Non-null assertion(!) 남용

```typescript
// 나쁜 예
const element = document.getElementById('root')!;
element.innerHTML = 'Hello'; // element가 null이면 런타임 에러

// 좋은 예
const element = document.getElementById('root');
if (!element) {
  throw new Error('Root element not found');
}
element.innerHTML = 'Hello'; // 안전
```

### 실수 10: TypeScript를 너무 복잡하게 쓰기

```typescript
// 나쁜 예: 읽을 수 없는 타입 체조
type DeepReadonly<T> = T extends (infer U)[]
  ? ReadonlyArray<DeepReadonly<U>>
  : T extends object
  ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
  : T;

// 좋은 예: 단순하게, 필요할 때만 복잡하게
// 대부분의 경우 Readonly<T>면 충분합니다
type ImmutableConfig = Readonly<Config>;
```

TypeScript의 타입 시스템은 튜링 완전하기 때문에 이론적으로 무한히 복잡한 타입을 만들 수 있습니다. 하지만 팀원들이 읽을 수 없는 타입은 `any`보다 나쁩니다. 복잡한 타입은 주석을 달아서 의도를 설명해주세요.

## 성능 고려사항

TypeScript는 컴파일 타임에만 존재하므로 런타임 성능에는 영향이 없습니다. 하지만 개발 경험에는 영향을 미칩니다.

### 타입 체크 속도 최적화

프로젝트가 커지면 `tsc` 타입 체크가 느려질 수 있습니다.

```bash
# 타입 체크 시간 측정
time npx tsc --noEmit

# 저희 프로젝트 기준 (3만 줄):
# 최적화 전: 18초
# 최적화 후: 7초
```

제가 적용한 최적화들:

1. **Project References 사용**: 모노레포에서 패키지별로 분리 빌드

```json
// tsconfig.json
{
  "references": [
    { "path": "./packages/shared" },
    { "path": "./packages/ui" },
    { "path": "./packages/api" }
  ]
}
```

2. **`skipLibCheck: true` 유지**: node_modules 타입 체크를 건너뜀
3. **불필요한 `include` 범위 줄이기**: `"include": ["src/**/*"]`만 포함
4. **`isolatedModules: true`**: 파일 단위 빌드가 가능해져 속도 향상

### 번들 크기에 미치는 영향

TypeScript는 컴파일 후 완전히 사라지므로 번들 크기에 직접적인 영향은 없습니다. 다만 주의할 점:

- `enum`은 JavaScript 코드로 변환되어 번들에 포함됩니다. `const enum` 또는 유니온 타입을 사용하세요.
- 클래스 데코레이터(@)는 추가 런타임 코드를 생성합니다.
- `namespace`는 레거시 패턴이므로 사용하지 마세요.

## 마무리

TypeScript의 학습 곡선은 생각보다 가파르지 않습니다. JavaScript를 알고 있다면 기본 타입 문법만 추가로 배우면 됩니다. 타입 시스템의 고급 기능은 필요할 때 하나씩 배워가면 됩니다.

저희 팀이 TS 전환을 결정한 것은 2024년 최고의 기술 결정 중 하나였습니다. 물론 전환 과정에서 3개월간의 투자가 필요했지만, 그 이후 절약되는 디버깅 시간과 온보딩 시간을 생각하면 충분히 가치 있었습니다.

가장 중요한 것은 시작하는 것입니다. 완벽한 타입을 처음부터 작성할 필요 없습니다. `strict: false`에서 시작해서 점차 엄격하게 가면 됩니다. 간단한 개인 프로젝트에서 TypeScript를 써보면서 타입의 편리함을 직접 경험해보세요. 한번 익숙해지면 다시 순수 JavaScript로 돌아가기 어려울 것입니다.

그리고 한 가지 당부: **TypeScript를 쓴다고 테스트를 안 써도 되는 것은 아닙니다.** 타입 시스템은 "이 함수에 잘못된 타입이 전달되는 것"을 막아주지만, "이 함수의 비즈니스 로직이 올바른지"는 검증하지 않습니다. 타입과 테스트는 상호 보완적입니다.
