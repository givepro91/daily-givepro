REST API는 현대 웹 개발의 근간입니다. 프론트엔드와 백엔드를 분리하고, 마이크로서비스 간 통신을 처리하고, 외부 서비스와 연동하는 모든 과정에서 API 설계가 핵심입니다. 그런데 5년간 백엔드 개발을 하면서 느낀 점은, 대부분의 팀이 비슷한 실수를 반복한다는 것입니다.

이 글에서는 제가 직접 저질렀거나, 코드 리뷰에서 반복적으로 발견한 REST API 설계 실수 10가지를 정리합니다. 각 실수에 대해 잘못된 예시, 왜 문제인지, 올바른 해결법, 그리고 실제 프로덕션에서 어떤 결과를 초래했는지를 공유합니다.

## 1. 동사를 URL에 넣기

가장 흔한 실수이자 가장 고치기 쉬운 실수입니다.

### 잘못된 예시

```javascript
// Express.js - 이렇게 하지 마세요
app.get('/getUsers', (req, res) => { ... });
app.post('/createUser', (req, res) => { ... });
app.post('/deleteUser/:id', (req, res) => { ... });
app.get('/fetchOrdersByUser/:userId', (req, res) => { ... });
```

### 왜 문제인가

REST에서 URL은 **리소스(명사)**를 나타내고, HTTP 메서드가 **행위(동사)**를 나타냅니다. URL에 동사를 넣으면 HTTP 메서드의 의미가 중복되고, URL이 예측 불가능해집니다. `/getUsers`인지 `/fetchUsers`인지 `/retrieveUsers`인지 일관성이 없어지면, API를 사용하는 프론트엔드 개발자가 매번 문서를 확인해야 합니다.

### 올바른 예시

```javascript
// Express.js - RESTful하게
app.get('/users', (req, res) => { ... });        // 사용자 목록 조회
app.post('/users', (req, res) => { ... });       // 사용자 생성
app.delete('/users/:id', (req, res) => { ... }); // 사용자 삭제
app.get('/users/:userId/orders', (req, res) => { ... }); // 특정 사용자의 주문 조회
```

### 실무 팁

리소스 이름은 항상 복수형을 사용하세요. `/user/:id`보다 `/users/:id`가 일관적입니다. 컬렉션 전체를 다루는 `/users`와 개별 리소스를 다루는 `/users/:id`가 자연스럽게 연결됩니다.

## 2. HTTP 상태 코드 무시 — 모든 응답에 200 OK

이 실수는 실제 프로덕션 장애로 이어진 경험이 있습니다.

### 잘못된 예시

```javascript
// 모든 응답을 200으로 보내는 안티패턴
app.post('/users', async (req, res) => {
  try {
    const user = await createUser(req.body);
    res.status(200).json({ success: true, data: user });
  } catch (error) {
    // 에러도 200으로 보냄!
    res.status(200).json({ success: false, error: error.message });
  }
});
```

### 프로덕션 사고 경험

2023년, 제가 참여한 프로젝트에서 결제 API가 모든 응답에 200을 반환하고 있었습니다. 모니터링 시스템(Datadog)은 HTTP 5xx 상태 코드를 기준으로 알람을 설정해 두었는데, 실제로 결제 실패가 발생해도 상태 코드가 200이었기 때문에 알람이 울리지 않았습니다. 약 3시간 동안 결제 실패가 감지되지 않았고, 약 120건의 결제가 누락되었습니다. 이 사고 이후 팀 전체가 HTTP 상태 코드 가이드라인을 만들었습니다.

### 올바른 예시

```javascript
app.post('/users', async (req, res) => {
  try {
    const { email, name } = req.body;

    if (!email || !name) {
      return res.status(400).json({
        error: { code: 'VALIDATION_ERROR', message: '이메일과 이름은 필수입니다.' }
      });
    }

    const existingUser = await findUserByEmail(email);
    if (existingUser) {
      return res.status(409).json({
        error: { code: 'DUPLICATE_EMAIL', message: '이미 등록된 이메일입니다.' }
      });
    }

    const user = await createUser({ email, name });
    res.status(201).json({ data: user });
  } catch (error) {
    console.error('사용자 생성 실패:', error);
    res.status(500).json({
      error: { code: 'INTERNAL_ERROR', message: '서버 내부 오류가 발생했습니다.' }
    });
  }
});
```

### 자주 쓰는 HTTP 상태 코드 정리

| 코드 | 의미 | 사용 시점 |
|------|------|----------|
| 200 | OK | 조회 성공, 수정 성공 |
| 201 | Created | 리소스 생성 성공 |
| 204 | No Content | 삭제 성공 (응답 본문 없음) |
| 400 | Bad Request | 요청 데이터 유효성 검증 실패 |
| 401 | Unauthorized | 인증 실패 (토큰 없음/만료) |
| 403 | Forbidden | 인가 실패 (권한 부족) |
| 404 | Not Found | 리소스가 존재하지 않음 |
| 409 | Conflict | 중복 데이터, 충돌 |
| 422 | Unprocessable Entity | 문법은 맞지만 처리 불가 |
| 429 | Too Many Requests | Rate Limit 초과 |
| 500 | Internal Server Error | 서버 내부 오류 |

## 3. 에러 응답 형식 불일치

프론트엔드 개발자가 가장 싫어하는 것 중 하나입니다.

### 잘못된 예시

```javascript
// 에러 응답이 엔드포인트마다 전부 다름
// GET /users/:id → 404
{ "message": "User not found" }

// POST /orders → 400
{ "error": "Invalid order data", "details": ["amount is required"] }

// PUT /products/:id → 422
{ "err": { "msg": "Price must be positive", "code": 1001 } }
```

### 왜 문제인가

프론트엔드에서 에러 처리 로직을 일반화할 수 없습니다. 각 API마다 에러 구조가 다르면, `try-catch`에서 어떤 필드를 읽어야 하는지 매번 달라지고, 이는 에러 핸들러 코드의 복잡성을 기하급수적으로 높입니다.

### 올바른 예시

```javascript
// 통일된 에러 응답 미들웨어
class ApiError extends Error {
  constructor(statusCode, code, message, details = null) {
    super(message);
    this.statusCode = statusCode;
    this.code = code;
    this.details = details;
  }
}

// 에러 핸들링 미들웨어
app.use((err, req, res, next) => {
  if (err instanceof ApiError) {
    return res.status(err.statusCode).json({
      error: {
        code: err.code,
        message: err.message,
        ...(err.details && { details: err.details }),
        timestamp: new Date().toISOString(),
        path: req.originalUrl,
      }
    });
  }

  // 예상치 못한 에러
  console.error('Unhandled error:', err);
  res.status(500).json({
    error: {
      code: 'INTERNAL_ERROR',
      message: '서버 내부 오류가 발생했습니다.',
      timestamp: new Date().toISOString(),
      path: req.originalUrl,
    }
  });
});

// 사용 예시
app.get('/users/:id', async (req, res, next) => {
  try {
    const user = await findUser(req.params.id);
    if (!user) {
      throw new ApiError(404, 'USER_NOT_FOUND', '사용자를 찾을 수 없습니다.');
    }
    res.json({ data: user });
  } catch (error) {
    next(error);
  }
});
```

이렇게 하면 프론트엔드에서는 항상 `response.error.code`와 `response.error.message`를 읽으면 됩니다.

## 4. 페이지네이션 없는 목록 API

### 잘못된 예시

```javascript
// 테이블의 모든 레코드를 한 번에 반환
app.get('/products', async (req, res) => {
  const products = await Product.find({});
  res.json({ data: products });
});
```

### 프로덕션 사고 경험

서비스 초기에는 상품이 50개 정도라 문제가 없었습니다. 6개월 뒤 상품이 15,000개가 되었을 때, 이 API를 호출하면 응답 크기가 약 12MB에 달했고, 응답 시간이 8초까지 증가했습니다. 모바일 사용자는 사실상 앱을 사용할 수 없게 되었고, 서버 메모리도 급증해 OOM(Out of Memory)으로 Pod가 재시작되는 현상까지 발생했습니다.

### 올바른 예시

```javascript
app.get('/products', async (req, res) => {
  const page = parseInt(req.query.page) || 1;
  const limit = Math.min(parseInt(req.query.limit) || 20, 100); // 최대 100개 제한
  const skip = (page - 1) * limit;

  const [products, total] = await Promise.all([
    Product.find({}).sort({ createdAt: -1 }).skip(skip).limit(limit),
    Product.countDocuments({}),
  ]);

  res.json({
    data: products,
    pagination: {
      page,
      limit,
      total,
      totalPages: Math.ceil(total / limit),
      hasNext: page < Math.ceil(total / limit),
      hasPrev: page > 1,
    }
  });
});
```

### 커서 기반 페이지네이션

데이터가 매우 많거나 실시간으로 추가되는 경우, 오프셋 기반보다 커서 기반 페이지네이션이 더 적합합니다.

```javascript
app.get('/feed', async (req, res) => {
  const limit = Math.min(parseInt(req.query.limit) || 20, 50);
  const cursor = req.query.cursor; // 마지막으로 본 항목의 ID

  const query = cursor ? { _id: { $lt: cursor } } : {};
  const items = await Post.find(query)
    .sort({ _id: -1 })
    .limit(limit + 1); // 1개 더 가져와서 다음 페이지 존재 여부 확인

  const hasNext = items.length > limit;
  if (hasNext) items.pop();

  res.json({
    data: items,
    pagination: {
      nextCursor: hasNext ? items[items.length - 1]._id : null,
      hasNext,
    }
  });
});
```

## 5. 버전 관리 부재

### 잘못된 예시

```javascript
// 버전 없이 API를 운영하다가 스키마를 바꾸면...
// 기존: { "name": "홍길동" }
// 변경: { "firstName": "길동", "lastName": "홍" }
// → 기존 클라이언트 전부 깨짐
app.get('/users/:id', handler); // 버전 정보 없음
```

### 올바른 예시

```javascript
// URL 경로에 버전 포함 (가장 일반적)
app.use('/api/v1/users', usersV1Router);
app.use('/api/v2/users', usersV2Router);

// v1 라우터 - 기존 클라이언트 호환
const usersV1Router = express.Router();
usersV1Router.get('/:id', async (req, res) => {
  const user = await findUser(req.params.id);
  res.json({
    data: {
      name: `${user.lastName}${user.firstName}`, // v1 형식 유지
      email: user.email,
    }
  });
});

// v2 라우터 - 새로운 형식
const usersV2Router = express.Router();
usersV2Router.get('/:id', async (req, res) => {
  const user = await findUser(req.params.id);
  res.json({
    data: {
      firstName: user.firstName,
      lastName: user.lastName,
      email: user.email,
    }
  });
});
```

버전 관리를 처음부터 하지 않으면, 나중에 API를 변경할 때 모든 클라이언트를 동시에 업데이트해야 하는 상황에 빠집니다. 모바일 앱의 경우 사용자가 업데이트를 하지 않을 수 있기 때문에, 이전 버전 API를 최소 6개월~1년 이상 유지해야 합니다.

## 6. N+1 쿼리를 유발하는 API 구조

### 잘못된 예시

```javascript
// 게시글 목록 API가 작성자 정보를 포함하지 않음
app.get('/posts', async (req, res) => {
  const posts = await Post.find({}).limit(20);
  res.json({ data: posts }); // authorId만 포함
});

// 프론트엔드에서 각 게시글마다 작성자 정보를 별도 요청
// → 20개 게시글 × 1번 요청 = 총 21번의 API 호출
```

### 왜 문제인가

프론트엔드가 게시글 목록을 표시할 때 각 게시글의 작성자 이름을 보여줘야 한다면, 목록 API 1번 + 작성자 조회 N번 = 총 N+1번의 요청이 발생합니다. 네트워크 왕복 시간(RTT)이 50ms라면, 20개 게시글을 표시하는 데만 1초 이상 걸립니다.

### 올바른 예시

```javascript
// 관련 데이터를 함께 반환 (populate/join)
app.get('/posts', async (req, res) => {
  const posts = await Post.find({})
    .populate('author', 'name avatar')  // 작성자 정보를 함께 조회
    .limit(20);

  res.json({
    data: posts.map(post => ({
      id: post._id,
      title: post.title,
      content: post.content,
      createdAt: post.createdAt,
      author: {
        id: post.author._id,
        name: post.author.name,
        avatar: post.author.avatar,
      }
    }))
  });
});
```

### 필드 선택 파라미터 제공

모든 관련 데이터를 항상 포함할 필요는 없습니다. 클라이언트가 필요한 필드를 선택할 수 있게 해주면 됩니다.

```javascript
// ?include=author,comments 파라미터 지원
app.get('/posts', async (req, res) => {
  const includes = (req.query.include || '').split(',').filter(Boolean);

  let query = Post.find({}).limit(20);

  if (includes.includes('author')) {
    query = query.populate('author', 'name avatar');
  }
  if (includes.includes('comments')) {
    query = query.populate({
      path: 'comments',
      options: { limit: 3, sort: { createdAt: -1 } },
      populate: { path: 'author', select: 'name' }
    });
  }

  const posts = await query;
  res.json({ data: posts });
});
```

## 7. 인증(Authentication)과 인가(Authorization) 혼동

많은 주니어 개발자가 이 두 개념을 혼동합니다. 결과적으로 보안 취약점이 생깁니다.

### 잘못된 예시

```javascript
// 인증만 확인하고 인가는 확인하지 않는 실수
app.delete('/users/:id', authMiddleware, async (req, res) => {
  // authMiddleware가 토큰 유효성만 검증
  // 하지만 "이 사용자가 해당 리소스를 삭제할 권한이 있는지"는 확인하지 않음!
  await User.findByIdAndDelete(req.params.id);
  res.status(204).send();
});

// → 일반 사용자가 다른 사용자의 계정을 삭제할 수 있는 치명적 취약점
```

### 올바른 예시

```javascript
// 인증 미들웨어 - "누구인가?"
const authMiddleware = async (req, res, next) => {
  try {
    const token = req.headers.authorization?.replace('Bearer ', '');
    if (!token) {
      return res.status(401).json({
        error: { code: 'AUTH_REQUIRED', message: '인증이 필요합니다.' }
      });
    }
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = await User.findById(decoded.userId);
    if (!req.user) {
      return res.status(401).json({
        error: { code: 'USER_NOT_FOUND', message: '유효하지 않은 토큰입니다.' }
      });
    }
    next();
  } catch (error) {
    res.status(401).json({
      error: { code: 'INVALID_TOKEN', message: '토큰이 만료되었거나 유효하지 않습니다.' }
    });
  }
};

// 인가 미들웨어 - "무엇을 할 수 있는가?"
const authorize = (...roles) => (req, res, next) => {
  if (!roles.includes(req.user.role)) {
    return res.status(403).json({
      error: { code: 'FORBIDDEN', message: '이 작업을 수행할 권한이 없습니다.' }
    });
  }
  next();
};

// 소유자 확인 미들웨어 - "자신의 리소스인가?"
const ownerOrAdmin = (paramName = 'id') => async (req, res, next) => {
  const resourceOwnerId = req.params[paramName];
  if (req.user._id.toString() !== resourceOwnerId && req.user.role !== 'admin') {
    return res.status(403).json({
      error: { code: 'FORBIDDEN', message: '자신의 리소스만 수정할 수 있습니다.' }
    });
  }
  next();
};

// 적용
app.delete('/users/:id', authMiddleware, ownerOrAdmin('id'), async (req, res) => {
  await User.findByIdAndDelete(req.params.id);
  res.status(204).send();
});

// 관리자 전용 API
app.get('/admin/stats', authMiddleware, authorize('admin'), async (req, res) => {
  const stats = await getAdminStats();
  res.json({ data: stats });
});
```

핵심은 **인증(401)은 "당신이 누구인지 모른다"**, **인가(403)은 "당신이 누구인지는 알지만 권한이 없다"**입니다. 이 구분이 명확해야 프론트엔드에서도 적절한 UI를 보여줄 수 있습니다(로그인 페이지로 이동 vs 권한 부족 안내).

## 8. 과도한 중첩 URL

### 잘못된 예시

```
GET /companies/:companyId/departments/:deptId/teams/:teamId/members/:memberId/tasks/:taskId
```

### 왜 문제인가

URL이 길어지면 가독성이 떨어지고, 프론트엔드에서 API를 호출할 때 여러 ID를 모두 알아야 합니다. 대부분의 경우 `taskId`만 있으면 해당 태스크를 조회할 수 있음에도 불구하고, 상위 리소스의 ID를 모두 전달해야 합니다. 또한 리소스의 소속 관계가 변경되면(예: 팀 이동) URL 자체가 바뀌어 버립니다.

### 올바른 예시

```javascript
// 2단계 이하로 중첩을 제한
GET /tasks/:taskId                    // 태스크 직접 조회
GET /teams/:teamId/members            // 팀의 멤버 목록
GET /members/:memberId/tasks          // 멤버의 태스크 목록

// 필터링은 쿼리 파라미터로
GET /tasks?teamId=123&status=active&assignee=456
```

일반적으로 `/리소스/:id/하위리소스`까지만 중첩하고, 그 이상은 쿼리 파라미터나 독립 엔드포인트로 분리하는 것이 좋습니다.

## 9. PATCH vs PUT 오용

### 잘못된 예시

```javascript
// PUT을 부분 업데이트에 사용하는 실수
app.put('/users/:id', async (req, res) => {
  // 이름만 바꾸려는데 PUT을 사용
  // PUT은 리소스 전체 교체를 의미함
  const user = await User.findByIdAndUpdate(req.params.id, req.body);
  res.json({ data: user });
});

// 클라이언트가 { name: "새이름" }만 보내면
// email, avatar 등 다른 필드가 사라질 수 있음!
```

### 올바른 예시

```javascript
// PUT - 리소스 전체 교체 (모든 필드 필수)
app.put('/users/:id', async (req, res) => {
  const { name, email, avatar, bio } = req.body;

  // 모든 필수 필드가 있는지 검증
  if (!name || !email) {
    return res.status(400).json({
      error: { code: 'VALIDATION_ERROR', message: 'PUT 요청은 모든 필수 필드를 포함해야 합니다.' }
    });
  }

  const user = await User.findByIdAndReplace(req.params.id, {
    name, email, avatar, bio
  }, { new: true });
  res.json({ data: user });
});

// PATCH - 부분 업데이트 (변경할 필드만)
app.patch('/users/:id', async (req, res) => {
  const allowedFields = ['name', 'avatar', 'bio'];
  const updates = {};

  for (const field of allowedFields) {
    if (req.body[field] !== undefined) {
      updates[field] = req.body[field];
    }
  }

  if (Object.keys(updates).length === 0) {
    return res.status(400).json({
      error: { code: 'NO_UPDATES', message: '업데이트할 필드가 없습니다.' }
    });
  }

  const user = await User.findByIdAndUpdate(
    req.params.id,
    { $set: updates },
    { new: true }
  );
  res.json({ data: user });
});
```

실무에서는 대부분 부분 업데이트가 필요하므로 PATCH를 더 자주 사용합니다. PUT은 설정값처럼 항상 전체를 저장하는 경우에 적합합니다.

## 10. 날짜/시간 포맷 불일치

이 실수는 글로벌 서비스에서 특히 치명적입니다.

### 잘못된 예시

```javascript
// 각 API마다 날짜 형식이 다름
GET /users/1     → { "createdAt": "2026-03-01T09:00:00Z" }       // ISO 8601
GET /orders/1    → { "orderDate": "2026/03/01 18:00:00" }        // 커스텀 형식
GET /posts/1     → { "publishedAt": 1709276400 }                  // Unix timestamp
GET /events/1    → { "startDate": "01-03-2026" }                  // DD-MM-YYYY??
```

### 왜 문제인가

프론트엔드에서 날짜를 파싱하는 로직이 API마다 달라야 합니다. 타임존 처리가 일관되지 않으면, 한국 사용자가 보는 날짜와 미국 사용자가 보는 날짜가 하루 차이날 수 있습니다. 특히 주문 시간, 예약 시간 등 비즈니스 로직에 민감한 데이터에서 이 문제가 발생하면 고객 불만으로 직결됩니다.

### 올바른 예시

```javascript
// 모든 날짜는 ISO 8601 + UTC로 통일
// 직렬화 미들웨어
const serializeDates = (obj) => {
  if (obj instanceof Date) {
    return obj.toISOString(); // "2026-03-01T09:00:00.000Z"
  }
  if (Array.isArray(obj)) {
    return obj.map(serializeDates);
  }
  if (obj && typeof obj === 'object') {
    const result = {};
    for (const [key, value] of Object.entries(obj)) {
      result[key] = serializeDates(value);
    }
    return result;
  }
  return obj;
};

// 응답 시 적용
app.get('/orders/:id', async (req, res) => {
  const order = await Order.findById(req.params.id).lean();
  res.json({ data: serializeDates(order) });
});

// 결과: 모든 날짜가 일관된 형식
// {
//   "data": {
//     "id": "abc123",
//     "orderDate": "2026-03-01T09:00:00.000Z",
//     "deliveryDate": "2026-03-05T14:30:00.000Z",
//     "createdAt": "2026-02-28T23:15:00.000Z"
//   }
// }
```

**규칙**: 서버는 항상 UTC로 저장하고, UTC로 응답합니다. 사용자의 로컬 시간대 변환은 프론트엔드의 책임입니다.

## 보너스: OpenAPI (Swagger) 문서 작성

위의 실수들을 방지하는 가장 좋은 방법 중 하나는 API 스펙을 먼저 정의하는 것입니다.

```yaml
openapi: 3.0.3
info:
  title: User API
  version: 1.0.0
paths:
  /api/v1/users:
    get:
      summary: 사용자 목록 조회
      parameters:
        - name: page
          in: query
          schema:
            type: integer
            default: 1
            minimum: 1
        - name: limit
          in: query
          schema:
            type: integer
            default: 20
            maximum: 100
      responses:
        '200':
          description: 성공
          content:
            application/json:
              schema:
                type: object
                properties:
                  data:
                    type: array
                    items:
                      $ref: '#/components/schemas/User'
                  pagination:
                    $ref: '#/components/schemas/Pagination'
        '401':
          $ref: '#/components/responses/Unauthorized'
    post:
      summary: 사용자 생성
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreateUserRequest'
      responses:
        '201':
          description: 생성 성공
        '400':
          $ref: '#/components/responses/BadRequest'
        '409':
          description: 이메일 중복

components:
  schemas:

      type: object
      properties:
        id:
          type: string
        name:
          type: string
        email:
          type: string
          format: email
        createdAt:
          type: string
          format: date-time
    Pagination:
      type: object
      properties:
        page:
          type: integer
        limit:
          type: integer
        total:
          type: integer
        totalPages:
          type: integer
  responses:

      description: 인증 필요

          schema:
            type: object
            properties:

                    type: string

                    type: string
```

## 보안 팁: Rate Limiting과 CORS

API 설계가 아무리 좋아도 보안 설정이 없으면 무용지물입니다.

### Rate Limiting

```javascript
import rateLimit from 'express-rate-limit';

// 전역 Rate Limiter
const globalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15분
  max: 100, // IP당 최대 100회
  standardHeaders: true,
  legacyHeaders: false,
  message: {
    error: { code: 'RATE_LIMIT_EXCEEDED', message: '요청이 너무 많습니다. 잠시 후 다시 시도해주세요.' }
  }
});

// 로그인 API에는 더 엄격하게
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5, // 15분에 5회만 허용
  message: {
    error: { code: 'TOO_MANY_LOGIN_ATTEMPTS', message: '로그인 시도 횟수를 초과했습니다.' }
  }
});

app.use('/api', globalLimiter);
app.post('/api/auth/login', loginLimiter, loginHandler);
```

### CORS 설정

```javascript
import cors from 'cors';

// 개발 환경과 프로덕션 환경의 CORS 설정을 분리
const corsOptions = {
  origin: process.env.NODE_ENV === 'production'
    ? ['https://myapp.com', 'https://admin.myapp.com']
    : ['http://localhost:3000', 'http://localhost:5173'],
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  credentials: true,
  maxAge: 86400, // preflight 캐시 24시간
};

app.use(cors(corsOptions));
```

## 마치며

REST API 설계는 한 번 배우면 끝나는 것이 아니라, 프로젝트를 진행하면서 계속 개선해 나가야 하는 영역입니다. 위에 나열한 10가지 실수는 제가 실제로 경험한 것들이고, 대부분 사전에 알았더라면 쉽게 피할 수 있었던 것들입니다.

API를 설계할 때 다음 세 가지 질문을 항상 던져보세요.

1. **이 API를 처음 보는 개발자가 직관적으로 이해할 수 있는가?**
2. **데이터가 10배, 100배 늘어나도 성능에 문제가 없는가?**
3. **에러가 발생했을 때 클라이언트가 적절하게 대응할 수 있는가?**

이 세 가지만 만족하면 대부분의 실수를 피할 수 있습니다. API는 한 번 공개하면 바꾸기 어렵기 때문에, 처음부터 신중하게 설계하는 것이 결국 팀 전체의 생산성을 높이는 길입니다.
