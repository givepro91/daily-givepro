웹 접근성(Web Accessibility, 줄여서 a11y)이라고 하면 많은 개발자가 "장애인을 위한 것"이라고 생각합니다. 저도 처음에는 그랬습니다. 하지만 3년간 실제 서비스에서 접근성 개선 프로젝트를 이끌면서 완전히 생각이 바뀌었습니다.

2023년, 회사에서 공공기관 대상 B2G 서비스를 준비하면서 웹 접근성 인증(KWCAG)을 받아야 했습니다. 당시 저는 "접근성? alt 태그 넣고 시맨틱 HTML 쓰면 되는 거 아니야?"라고 가볍게 생각했는데, 인증 심사에서 처참하게 떨어졌습니다. 160개 체크 항목 중 63개에서 부적합 판정을 받았습니다.

그로부터 6개월간의 접근성 개선 여정이 시작되었고, 그 과정에서 접근성이 단순히 "좋은 일을 하는 것"이 아니라 **더 좋은 코드, 더 나은 UX, 더 높은 매출**로 이어진다는 것을 경험으로 배웠습니다.

이 글에서는 교과서적인 접근성 이론이 아니라, 실제 프로젝트에서 겪은 시행착오와 구체적인 해결 방법을 공유합니다.

## 왜 접근성이 "나와 관련 있는" 일인가?

### 법적 요구사항: 더 이상 선택이 아니다

한국의 "장애인차별금지법"과 "전자정부 웹 표준 준수 지침"에 따라 공공기관 웹사이트는 웹 접근성을 의무적으로 준수해야 합니다. 2025년부터는 민간 기업 중 매출 500억 이상 기업에도 적용 범위가 확대되었습니다.

내가 직접 경험한 사례로, 우리 회사가 공공기관 입찰에 참여했을 때 "웹 접근성 인증 보유 여부"가 가산점 항목이었습니다. 접근성 인증이 없어서 수억 원 규모의 프로젝트를 놓친 경쟁사도 있었습니다.

미국의 ADA(Americans with Disabilities Act) 관련 소송은 매년 증가하고 있고, 2024년에는 웹 접근성 관련 소송이 4,000건을 넘었습니다. 글로벌 서비스를 만든다면 접근성은 법적 리스크 관리 차원에서도 필수입니다.

### SEO와 접근성의 깊은 관계

검색 엔진 크롤러는 사실상 "시각장애가 있는 사용자"와 같습니다. 이미지를 볼 수 없고, JavaScript를 제한적으로 실행하며, 페이지 구조를 HTML 마크업으로 파악합니다.

실제로 사용해보니 접근성을 개선한 후 SEO 지표가 눈에 띄게 좋아졌습니다.

| 지표 | 접근성 개선 전 | 접근성 개선 후 | 변화 |
|------|---------------|---------------|------|
| Lighthouse 접근성 점수 | 62 | 97 | +35 |
| Lighthouse SEO 점수 | 78 | 95 | +17 |
| 구글 검색 노출 수 (월) | 12,000 | 31,000 | +158% |
| 평균 검색 순위 | 18.3 | 8.7 | +52% |
| 오가닉 트래픽 (월) | 3,200 | 9,800 | +206% |

시맨틱 HTML, 대체 텍스트, 명확한 헤딩 구조, 의미 있는 링크 텍스트 등 접근성을 위해 한 작업이 그대로 SEO에 도움이 됩니다.

### 모든 사용자의 경험을 개선한다

접근성 개선은 장애가 있는 사용자뿐 아니라 모든 사용자에게 도움이 됩니다. 팀에서 도입했을 때 처음에는 "장애인 사용자가 몇 명이나 되겠어"라는 회의론이 있었지만, 실제 데이터를 보니 생각이 달라졌습니다.

- 밝은 햇빛 아래에서 스마트폰으로 접속하는 사용자: 충분한 색상 대비가 필요
- 한 손만 사용할 수 있는 상황(다른 손에 아이를 안고 있을 때): 키보드/터치 접근성 필요
- 소음이 많은 환경에서 동영상을 보는 사용자: 자막 필요
- 느린 인터넷 환경: 대체 텍스트가 이미지 로딩 전에 정보를 전달
- 고령 사용자: 큰 폰트, 명확한 네비게이션 필요

전 세계 인구의 약 15%가 어떤 형태든 장애를 가지고 있으며, 일시적 장애(팔 골절, 눈 수술 후 등)까지 포함하면 그 비율은 훨씬 높습니다.

## WCAG 핵심 원칙: 실전에서 이렇게 적용했습니다

WCAG(Web Content Accessibility Guidelines)는 웹 접근성의 국제 표준입니다. 네 가지 핵심 원칙인 POUR을 기억하세요. 각 원칙별로 제가 실제 프로젝트에서 겪은 사례와 해결 방법을 공유합니다.

### 1. 인식 가능 (Perceivable)

정보가 사용자에게 전달될 수 있어야 합니다.

**실제 사례: 아이콘 버튼의 함정**

우리 서비스에는 아이콘만으로 구성된 버튼이 30개 이상 있었습니다. 예를 들어 휴지통 아이콘 버튼, 연필 아이콘 버튼, X 아이콘 버튼 등. 시각적으로는 의미가 명확하지만 스크린 리더에서는 "버튼", "버튼", "버튼"으로만 읽혔습니다.

```html
<!-- 나쁜 예: 스크린 리더에서 "버튼"으로만 읽힘 -->
<button><svg>...</svg></button>

<!-- 좋은 예: 스크린 리더에서 "삭제 버튼"으로 읽힘 -->
<button aria-label="삭제"><svg aria-hidden="true">...</svg></button>

<!-- 더 좋은 예: 텍스트도 함께 표시 -->
<button>
  <svg aria-hidden="true">...</svg>
  <span>삭제</span>
</button>
```

**실제 사례: 의미 있는 대체 텍스트 작성법**

"이미지"라고 쓰면 안 됩니다. 그래프나 차트 이미지라면 데이터를 설명해야 합니다.

```html
<!-- 나쁜 예 -->
<img src="chart.png" alt="이미지" />
<img src="chart.png" alt="차트" />
<img src="chart.png" /> <!-- alt 속성 누락 -->

<!-- 좋은 예: 데이터를 설명 -->
<img src="chart.png" alt="2026년 월별 방문자 추이 그래프. 1월 5만명에서 시작하여 3월에 최고 10만명을 달성한 후 소폭 감소" />

<!-- 장식용 이미지는 빈 alt -->
<img src="decorative-line.png" alt="" />
<img src="background-pattern.png" alt="" role="presentation" />
```

내가 직접 경험한 실수 중 하나는 이커머스 사이트에서 상품 이미지의 alt를 모두 "상품 이미지"로 넣었던 것입니다. 스크린 리더 사용자가 "상품 이미지, 상품 이미지, 상품 이미지"를 반복해서 듣게 되어, 어떤 상품인지 전혀 알 수 없었습니다. alt에는 "나이키 에어맥스 97 화이트 러닝화"처럼 구체적인 상품 정보를 넣어야 합니다.

### 2. 운용 가능 (Operable)

인터페이스를 조작할 수 있어야 합니다.

**실제 사례: 키보드 트랩**

모달 다이얼로그에서 가장 흔하게 발생하는 접근성 문제가 "키보드 트랩"입니다. 모달이 열렸을 때 Tab 키를 누르면 모달 뒤의 요소로 포커스가 이동하는 문제입니다.

```tsx
// 접근성을 고려한 모달 컴포넌트 (React)
function AccessibleModal({ isOpen, onClose, title, children }) {
  const modalRef = useRef(null);
  const previousFocusRef = useRef(null);

  useEffect(() => {
    if (isOpen) {
      // 모달이 열리기 전 포커스 위치 저장
      previousFocusRef.current = document.activeElement;

      // 모달에 포커스 이동
      modalRef.current?.focus();

      // 모달 외부 스크롤 방지
      document.body.style.overflow = 'hidden';
    }

    return () => {
      // 모달이 닫히면 이전 포커스 위치로 복귀
      previousFocusRef.current?.focus();
      document.body.style.overflow = '';
    };
  }, [isOpen]);

  // ESC 키로 모달 닫기
  const handleKeyDown = (e) => {
    if (e.key === 'Escape') {
      onClose();
    }
  };

  if (!isOpen) return null;

  return (
    <div
      role="dialog"
      aria-modal="true"
      aria-labelledby="modal-title"
      ref={modalRef}
      tabIndex={-1}
      onKeyDown={handleKeyDown}
    >
      <h2 id="modal-title">{title}</h2>
      {children}
      <button onClick={onClose}>닫기</button>
    </div>
  );
}
```

팀에서 도입했을 때 모든 모달을 이 패턴으로 교체했더니, 접근성 심사 항목 중 "키보드 사용 보장" 관련 부적합이 한 번에 12개 해결되었습니다.

### 3. 이해 가능 (Understandable)

정보와 인터페이스가 이해 가능해야 합니다.

**실제 사례: 폼 에러 메시지의 접근성**

폼 유효성 검사에서 에러가 발생했을 때, 시각적으로 빨간 테두리만 표시하면 스크린 리더 사용자는 무엇이 잘못되었는지 알 수 없습니다.

```tsx
// 접근성을 고려한 폼 필드
function FormField({ label, error, id, ...props }) {
  const errorId = `${id}-error`;
  const hintId = `${id}-hint`;

  return (
    <div>
      <label htmlFor={id}>{label}</label>
      <input
        id={id}
        aria-invalid={error ? 'true' : 'false'}
        aria-describedby={error ? errorId : hintId}
        {...props}
      />
      {error ? (
        <p id={errorId} role="alert" className="text-red-600">
          {error}
        </p>
      ) : (
        <p id={hintId} className="text-gray-500">
          8자 이상의 영문/숫자 조합을 입력하세요
        </p>
      )}
    </div>
  );
}
```

여기서 핵심은 `role="alert"`입니다. 이 속성이 있으면 에러 메시지가 나타날 때 스크린 리더가 즉시 읽어줍니다. 사용자가 해당 필드에 포커스하지 않아도 에러를 인지할 수 있습니다.

### 4. 견고함 (Robust)

다양한 기술(보조 기술 포함)로 해석할 수 있어야 합니다.

실제로 사용해보니 견고함에서 가장 중요한 것은 **유효한 HTML**을 작성하는 것이었습니다. 중복된 id, 잘못된 nesting(예: `<p>` 안에 `<div>`), 닫히지 않은 태그 등이 보조 기술에서 예상치 못한 동작을 유발합니다.

```html
<!-- 나쁜 예: p 안에 블록 요소 -->
<p>
  <div>이렇게 하면 안 됩니다</div>
</p>

<!-- 좋은 예 -->
<div>
  <p>이렇게 해야 합니다</p>
</div>

<!-- 나쁜 예: 중복 id -->
<label for="name">이름</label>
<input id="name" />
<label for="name">닉네임</label>  <!-- 같은 id를 참조 -->
<input id="name" />  <!-- id 중복! -->

<!-- 좋은 예: 고유 id -->
<label for="real-name">이름</label>
<input id="real-name" />
<label for="nickname">닉네임</label>
<input id="nickname" />
```

## 실전 접근성 패턴: React 컴포넌트별 가이드

### 시맨틱 HTML이 먼저다

```html
<!-- 좋은 예: 시맨틱 요소 사용 -->
<header>
  <nav aria-label="메인 네비게이션">...</nav>
</header>
<main>
  <article>
    <h1>글 제목</h1>
    <section aria-labelledby="section-1">
      <h2 id="section-1">섹션 제목</h2>
      ...
    </section>
  </article>
</main>
<aside aria-label="관련 글">...</aside>
<footer>...</footer>

<!-- 나쁜 예: div로 모든 것을 구성 -->
<div class="header">
  <div class="nav">...</div>
</div>
<div class="main">
  <div class="article">
    <div class="title">글 제목</div>
  </div>
</div>
```

### 헤딩 구조: 절대 레벨을 건너뛰지 마세요

```html
<!-- 좋은 예: 순차적 헤딩 -->
<h1>웹 접근성 가이드</h1>
  <h2>WCAG 원칙</h2>
    <h3>인식 가능</h3>
    <h3>운용 가능</h3>
  <h2>실전 적용</h2>
    <h3>이미지 접근성</h3>

<!-- 나쁜 예: 헤딩 레벨 건너뜀 -->
<h1>웹 접근성 가이드</h1>
  <h3>갑자기 h3</h3>  <!-- h2를 건너뜀! -->
  <h4>h4로 또 점프</h4>  <!-- 논리적 구조 파괴 -->
```

스크린 리더 사용자는 헤딩을 통해 페이지를 탐색합니다. h1 → h3으로 건너뛰면 "h2가 있을 것으로 기대했는데 없다"는 혼란이 생깁니다.

### 키보드 접근성: div를 버튼으로 쓰지 마세요

```html
<!-- 좋은 예: 네이티브 button 사용 -->
<button onClick={handleClick}>삭제</button>

<!-- 나쁜 예: div에 클릭 이벤트 -->
<div onClick={handleClick}>삭제</div>
```

`div`로 만든 "버튼"의 문제점은 이것뿐이 아닙니다.

| 기능 | `<button>` | `<div onClick>` |
|------|-----------|-----------------|
| Tab으로 포커스 | 자동 지원 | 수동으로 tabindex 추가 필요 |
| Enter/Space로 활성화 | 자동 지원 | 수동으로 keydown 핸들러 추가 필요 |
| 스크린 리더에서 "버튼"으로 인식 | 자동 지원 | 수동으로 role="button" 추가 필요 |
| 비활성화(disabled) | 속성으로 지원 | 수동으로 aria-disabled 추가 필요 |
| 포커스 스타일 | 브라우저 기본 제공 | 수동으로 CSS 추가 필요 |

네이티브 요소를 사용하면 이 모든 것이 "무료"입니다. div로 버튼을 만들면 최소 5가지를 수동으로 구현해야 합니다.

### 색상 대비: 디자이너와의 협업이 핵심

텍스트와 배경의 색상 대비 비율은 최소 4.5:1(일반 텍스트) 또는 3:1(큰 텍스트, 18px 이상)이어야 합니다.

```css
/* 좋은 예: 충분한 대비 (12.6:1) */
color: #333333;
background: #ffffff;

/* 나쁜 예: 부족한 대비 (2.8:1) */
color: #999999;
background: #ffffff;

/* 흔한 실수: 브랜드 컬러가 대비 기준 미달 */
color: #FF6B35; /* 주황색 텍스트 on 흰 배경 = 3.2:1 (미달!) */
background: #ffffff;

/* 해결: 채도를 유지하면서 명도를 낮춤 */
color: #C84E1A; /* 어두운 주황 on 흰 배경 = 5.4:1 (통과!) */
background: #ffffff;
```

내가 직접 경험한 디자이너와의 갈등을 하나 공유하면, 디자이너가 "이 연한 회색 텍스트가 세련된데요?"라고 했는데 대비 비율이 2.1:1이었습니다. WCAG 기준(4.5:1)을 전혀 충족하지 못했습니다.

이때 효과적이었던 접근법은 "접근성을 위해 못생기게 만들어야 한다"가 아니라 "대비 기준을 만족하면서도 아름다운 색상 조합을 함께 찾아보자"였습니다. 실제로 Adobe Color, Contrast Checker 같은 도구를 함께 보면서 디자이너가 원하는 분위기를 유지하면서도 접근성을 만족하는 색상을 찾았습니다.

### 폼 접근성: placeholder는 label이 아닙니다

```html
<!-- 좋은 예: label과 input 연결 -->
<label htmlFor="email">이메일</label>
<input id="email" type="email" required aria-describedby="email-hint" />
<p id="email-hint">업무용 이메일을 입력하세요</p>

<!-- 나쁜 예: label 없음 -->
<input type="email" placeholder="이메일" />
```

`placeholder`를 `label` 대신 사용하면 안 되는 이유가 여러 가지 있습니다.

1. 입력을 시작하면 placeholder가 사라져 무엇을 입력해야 하는지 잊음
2. placeholder 색상은 대부분 대비 기준을 만족하지 못함
3. 스크린 리더에 따라 placeholder를 읽지 않는 경우가 있음
4. 사용자가 입력한 내용과 placeholder를 혼동할 수 있음

### Skip to Content 링크

페이지 상단에 "본문으로 건너뛰기" 링크를 추가하면 키보드 사용자가 헤더의 모든 링크를 탭하지 않고 바로 본문으로 이동할 수 있습니다.

```html
<body>
  <a href="#main-content" className="sr-only focus:not-sr-only focus:absolute focus:top-4 focus:left-4 focus:z-50 focus:bg-white focus:px-4 focus:py-2">
    본문으로 건너뛰기
  </a>
  <header><!-- 네비게이션 링크 20개... --></header>
  <main id="main-content">
    <!-- 본문 내용 -->
  </main>
</body>
```

이 링크는 평소에는 시각적으로 숨겨져 있다가 Tab을 눌렀을 때만 나타납니다. 간단하지만 키보드 사용자의 경험을 크게 개선합니다.

## ARIA 속성: 남용은 독이다

ARIA(Accessible Rich Internet Applications)는 HTML의 접근성을 보완하는 속성들입니다. 하지만 **"No ARIA is better than bad ARIA"**라는 원칙을 반드시 기억하세요.

### 올바른 ARIA 사용 예시

```html
<!-- 역할 정의: 동적으로 나타나는 에러 메시지 -->
<div role="alert">결제에 실패했습니다. 카드 정보를 확인해주세요.</div>

<!-- 상태 전달: 드롭다운 메뉴 -->
<button aria-expanded="false" aria-controls="user-menu" aria-haspopup="true">
  사용자 메뉴
</button>
<ul id="user-menu" role="menu" hidden>
  <li role="menuitem"><a href="/profile">프로필</a></li>
  <li role="menuitem"><a href="/settings">설정</a></li>
  <li role="menuitem"><button>로그아웃</button></li>
</ul>

<!-- 레이블 제공: 아이콘 버튼 -->
<button aria-label="닫기">✕</button>

<!-- 라이브 영역: 검색 결과 수 업데이트 -->
<div aria-live="polite" aria-atomic="true">검색 결과 {count}건</div>

<!-- 탭 인터페이스 -->
<div role="tablist" aria-label="설정 메뉴">
  <button role="tab" aria-selected="true" aria-controls="panel-general">일반</button>
  <button role="tab" aria-selected="false" aria-controls="panel-security">보안</button>
</div>
<div id="panel-general" role="tabpanel">일반 설정 내용</div>
<div id="panel-security" role="tabpanel" hidden>보안 설정 내용</div>
```

### ARIA 사용의 5가지 규칙

1. **네이티브 HTML 요소를 사용할 수 있으면 ARIA를 사용하지 마세요** - `<button>` 대신 `<div role="button">`을 쓸 이유가 없습니다.
2. **네이티브 시맨틱을 변경하지 마세요** - `<h1 role="button">`은 절대 하면 안 됩니다.
3. **모든 인터랙티브 ARIA 컨트롤은 키보드로 사용 가능해야 합니다**
4. **포커스 가능한 요소에 `role="presentation"`이나 `aria-hidden="true"`를 사용하지 마세요**
5. **모든 인터랙티브 요소에는 접근 가능한 이름이 있어야 합니다**

## 접근성 테스트: 자동화와 수동 테스트의 조합

### 자동화 테스트 도구

자동화 도구는 접근성 문제의 약 30-40%만 감지할 수 있습니다. 하지만 기본적인 이슈를 빠르게 찾는 데 효과적입니다.

- **Lighthouse**: Chrome DevTools에 내장. 접근성 점수와 개선 사항을 알려줍니다.
- **axe DevTools**: 크롬 확장 프로그램. 상세한 접근성 이슈를 감지하며 WCAG 기준별로 분류해줍니다.
- **WAVE**: 웹 페이지의 접근성을 시각적으로 분석합니다.
- **eslint-plugin-jsx-a11y**: React 프로젝트에서 접근성 문제를 빌드 타임에 감지합니다.

```bash
# eslint-plugin-jsx-a11y 설치
npm install eslint-plugin-jsx-a11y --save-dev
```

팀에서 도입했을 때 ESLint 접근성 플러그인이 가장 효과적이었습니다. 코드를 작성하는 시점에 바로 피드백을 받으니, 나중에 한꺼번에 고치는 것보다 훨씬 효율적이었습니다.

### 수동 테스트: 이것이 진짜 중요합니다

자동화 도구만으로는 부족합니다. 반드시 수동 테스트를 병행해야 합니다.

**키보드 테스트 체크리스트:**

1. Tab 키만으로 모든 인터랙티브 요소에 접근 가능한가?
2. 현재 포커스 위치가 시각적으로 명확한가?
3. 모달이 열렸을 때 포커스가 모달 내부로 이동하는가?
4. ESC 키로 모달/드롭다운을 닫을 수 있는가?
5. 논리적인 순서대로 포커스가 이동하는가?

**스크린 리더 테스트:**

저는 Mac의 VoiceOver를 주로 사용합니다. `Cmd + F5`로 켜고 끌 수 있습니다.

처음 VoiceOver로 우리 서비스를 탐색했을 때 충격을 받았습니다. "링크, 링크, 링크, 버튼, 이미지, 링크" 이런 식으로 읽혀서 어떤 링크가 어디로 가는지, 이미지가 무엇인지 전혀 알 수 없었습니다. 이 경험이 접근성 개선의 동기가 되었습니다.

**스크린 리더 테스트 팁:**
- 처음에는 눈을 감고 테스트해보세요 - 시각 정보 없이 청각만으로 서비스를 사용할 수 있는지 확인
- 주요 사용자 시나리오(회원가입, 로그인, 상품 구매 등)를 스크린 리더로 완수해보세요
- VoiceOver(Mac), NVDA(Windows, 무료), JAWS(Windows, 유료) 중 최소 하나는 정기적으로 테스트하세요

## 접근성 개선의 비즈니스 임팩트

마지막으로 접근성이 비즈니스에 미친 실제 영향을 공유합니다. 이 데이터는 우리 서비스에서 6개월간의 접근성 개선 전후를 비교한 것입니다.

| 지표 | 개선 전 | 개선 후 | 변화 |
|------|--------|--------|------|
| 페이지 이탈률 | 52% | 38% | -14%p |
| 평균 세션 시간 | 2분 30초 | 4분 10초 | +66% |
| 폼 완료율 | 34% | 51% | +50% |
| 모바일 전환율 | 1.2% | 2.1% | +75% |
| 접근성 관련 CS 문의 | 월 23건 | 월 4건 | -83% |
| 고령 사용자 가입 | 월 120명 | 월 340명 | +183% |

특히 폼 완료율의 50% 향상은 놀라웠습니다. 접근성을 위해 label, 힌트 텍스트, 에러 메시지를 개선한 것이 모든 사용자의 폼 사용 경험을 크게 향상시켰습니다.

## 접근성 도입 로드맵

처음부터 완벽할 필요는 없습니다. 단계별로 접근하세요.

**1단계 (1-2주): 기본 중의 기본**
- 모든 이미지에 적절한 alt 텍스트 추가
- 시맨틱 HTML 태그 사용 (header, nav, main, footer)
- 헤딩 구조 정리 (h1 → h2 → h3 순서)

**2단계 (2-4주): 키보드와 폼**
- 모든 인터랙티브 요소가 키보드로 접근 가능한지 확인
- 폼에 label 연결, 에러 메시지 접근성 개선
- Skip to Content 링크 추가

**3단계 (1-2개월): 고급 패턴**
- 모달, 탭, 아코디언 등 복잡한 UI의 접근성 개선
- 색상 대비 전수 검사
- 스크린 리더 테스트 실시

**4단계 (지속): 프로세스 통합**
- eslint-plugin-jsx-a11y 도입
- 코드 리뷰에 접근성 체크 추가
- 새 기능 개발 시 접근성을 설계 단계에서 고려

## 마무리

웹 접근성은 선택이 아닌 필수이며, 비용이 아니라 투자입니다. 3년간의 접근성 개선 경험을 통해 확신하게 된 것은 "접근성을 고려한 개발은 결과적으로 더 좋은 코드, 더 나은 UX, 더 높은 매출을 만든다"는 것입니다.

"장애인을 위한 것"이 아니라 "모두를 위한 것"이라는 관점에서 접근하면 자연스럽게 좋은 웹을 만들 수 있습니다. 오늘 시맨틱 HTML 하나, alt 텍스트 하나부터 시작해보세요. 그것이 더 나은 웹을 향한 첫걸음입니다.
