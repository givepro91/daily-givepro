웹 접근성(Web Accessibility, 줄여서 a11y)은 장애가 있는 사용자를 포함하여 모든 사람이 웹 콘텐츠에 접근하고 사용할 수 있도록 하는 것입니다. 전 세계 인구의 약 15%가 어떤 형태든 장애를 가지고 있으며, 이들도 웹을 동등하게 이용할 권리가 있습니다.

접근성은 단순히 "좋은 일을 하는 것"이 아닙니다. 법적 요구사항이기도 하고, SEO에도 긍정적인 영향을 미치며, 결과적으로 더 많은 사용자에게 도달할 수 있게 합니다.

## 왜 접근성이 중요한가?

### 법적 요구사항

한국의 "장애인차별금지법"과 "전자정부 웹 표준 준수 지침"에 따라 공공기관 웹사이트는 웹 접근성을 의무적으로 준수해야 합니다. 민간 기업도 점차 적용 범위가 확대되고 있습니다.

### SEO 향상

검색 엔진 크롤러는 시각적 요소를 볼 수 없습니다. 접근성을 고려해 만든 시맨틱 HTML, 대체 텍스트, 명확한 링크 텍스트 등은 크롤러가 콘텐츠를 더 잘 이해하도록 돕습니다. 즉, 접근성이 좋은 웹사이트는 SEO 점수도 높습니다.

### 사용자 경험 전반 향상

접근성 개선은 장애가 있는 사용자뿐 아니라 모든 사용자에게 도움이 됩니다. 밝은 햇빛 아래에서 화면을 보는 사람, 한 손만 사용할 수 있는 상황, 느린 인터넷 환경 등에서도 접근 가능한 웹사이트는 더 나은 경험을 제공합니다.

## 핵심 원칙 (WCAG)

WCAG(Web Content Accessibility Guidelines)는 웹 접근성의 국제 표준입니다. 네 가지 핵심 원칙인 POUR을 기억하세요.

### 1. 인식 가능 (Perceivable)

정보가 사용자에게 전달될 수 있어야 합니다.

- 이미지에 대체 텍스트 제공
- 동영상에 자막 제공
- 충분한 색상 대비
- 텍스트 크기 조절 가능

### 2. 운용 가능 (Operable)

인터페이스를 조작할 수 있어야 합니다.

- 키보드만으로 모든 기능 사용 가능
- 충분한 시간 제공
- 깜빡이는 콘텐츠 제한
- 명확한 네비게이션

### 3. 이해 가능 (Understandable)

정보와 인터페이스가 이해 가능해야 합니다.

- 읽기 쉬운 텍스트
- 예측 가능한 동작
- 입력 오류 수정 도움

### 4. 견고함 (Robust)

다양한 기술(보조 기술 포함)로 해석할 수 있어야 합니다.

- 유효한 HTML
- 보조 기술과의 호환성
- 미래 기술과의 호환성

## 실전 접근성 체크리스트

### 이미지

```html
<!-- 좋은 예: 의미 있는 대체 텍스트 -->
<img src="chart.png" alt="2026년 월별 방문자 추이 그래프, 3월에 최고 10만 명 달성" />

<!-- 좋은 예: 장식용 이미지는 빈 alt -->
<img src="decorative-line.png" alt="" />

<!-- 나쁜 예 -->
<img src="chart.png" alt="이미지" />
<img src="chart.png" /> <!-- alt 속성 누락 -->
```

### 시맨틱 HTML

```html
<!-- 좋은 예: 시맨틱 요소 사용 -->
<header>
  <nav aria-label="Main navigation">...</nav>
</header>
<main>
  <article>
    <h1>글 제목</h1>
    <section>...</section>
  </article>
</main>
<footer>...</footer>

<!-- 나쁜 예: div로 모든 것을 구성 -->
<div class="header">
  <div class="nav">...</div>
</div>
```

### 헤딩 구조

```html
<!-- 좋은 예: 순차적 헤딩 -->
<h1>페이지 제목</h1>
  <h2>섹션 1</h2>
    <h3>하위 섹션</h3>
  <h2>섹션 2</h2>

<!-- 나쁜 예: 헤딩 레벨 건너뜀 -->
<h1>페이지 제목</h1>
  <h3>갑자기 h3</h3>  <!-- h2를 건너뜀 -->
```

### 키보드 접근성

모든 인터랙티브 요소(링크, 버튼, 폼)는 키보드로 접근 가능해야 합니다.

```html
<!-- 좋은 예: 네이티브 button 사용 -->
<button onClick={handleClick}>삭제</button>

<!-- 나쁜 예: div에 클릭 이벤트 -->
<div onClick={handleClick}>삭제</div>
```

`div`로 만든 "버튼"은 Tab 키로 포커스할 수 없고, Enter/Space 키로 활성화할 수 없으며, 스크린 리더가 버튼으로 인식하지 못합니다.

### 색상 대비

텍스트와 배경의 색상 대비 비율은 최소 4.5:1 (일반 텍스트) 또는 3:1 (큰 텍스트)이어야 합니다.

```css
/* 좋은 예: 충분한 대비 */
color: #333333; /* 진한 회색 텍스트 */
background: #ffffff; /* 흰 배경 */
/* 대비 비율: 12.6:1 */

/* 나쁜 예: 부족한 대비 */
color: #999999; /* 밝은 회색 텍스트 */
background: #ffffff; /* 흰 배경 */
/* 대비 비율: 2.8:1 */
```

### 폼 접근성

```html
<!-- 좋은 예: label과 input 연결 -->
<label htmlFor="email">이메일</label>
<input id="email" type="email" required aria-describedby="email-hint" />
<p id="email-hint">업무용 이메일을 입력하세요</p>

<!-- 나쁜 예: label 없음 -->
<input type="email" placeholder="이메일" />
```

`placeholder`는 `label`을 대체할 수 없습니다. 입력을 시작하면 placeholder가 사라져 무엇을 입력해야 하는지 알 수 없게 됩니다.

### Skip to Content 링크

페이지 상단에 "본문으로 건너뛰기" 링크를 추가하면 키보드 사용자가 헤더의 모든 링크를 탭하지 않고 바로 본문으로 이동할 수 있습니다.

```html
<a href="#main-content" className="sr-only focus:not-sr-only">
  본문으로 건너뛰기
</a>
```

## ARIA 속성

ARIA(Accessible Rich Internet Applications)는 HTML의 접근성을 보완하는 속성들입니다.

```html
<!-- 역할 정의 -->
<div role="alert">에러가 발생했습니다.</div>

<!-- 상태 전달 -->
<button aria-expanded="false" aria-controls="menu">메뉴</button>
<ul id="menu" hidden>...</ul>

<!-- 레이블 제공 -->
<button aria-label="닫기">✕</button>

<!-- 라이브 영역 -->
<div aria-live="polite">검색 결과 10건</div>
```

### ARIA 사용 규칙

1. 네이티브 HTML 요소를 사용할 수 있으면 ARIA를 사용하지 마세요
2. 네이티브 시맨틱을 변경하지 마세요
3. 모든 인터랙티브 ARIA 컨트롤은 키보드로 사용 가능해야 합니다

## 접근성 테스트 도구

- **Lighthouse**: Chrome DevTools에 내장. 접근성 점수와 개선 사항을 알려줍니다
- **axe DevTools**: 크롬 확장 프로그램. 상세한 접근성 이슈 감지
- **WAVE**: 웹 페이지의 접근성을 시각적으로 분석
- **스크린 리더 테스트**: VoiceOver(Mac), NVDA(Windows)로 직접 테스트

## 마무리

웹 접근성은 선택이 아닌 필수입니다. 처음부터 완벽할 필요는 없습니다. 시맨틱 HTML 사용, 이미지 대체 텍스트, 키보드 접근성, 충분한 색상 대비, 이 네 가지만 먼저 챙겨도 접근성이 크게 개선됩니다.

접근성을 고려한 개발은 결과적으로 더 좋은 코드, 더 나은 SEO, 더 넓은 사용자층을 만듭니다. "장애인을 위한 것"이 아니라 "모두를 위한 것"이라는 관점에서 접근하면 자연스럽게 좋은 웹을 만들 수 있습니다.
