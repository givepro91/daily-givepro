VS Code는 전 세계에서 가장 많이 사용되는 코드 에디터입니다. Stack Overflow의 2025년 설문조사에 따르면 개발자의 약 73%가 VS Code를 주력 에디터로 사용하고 있습니다. 하지만 기본 설정만으로 사용하고 있다면 이 강력한 도구의 절반도 활용하지 못하는 것과 같습니다.

저는 5년 넘게 VS Code를 주력 에디터로 사용하면서 수많은 설정과 확장 프로그램을 시도해봤습니다. IntelliJ, Sublime Text, Vim 등 다른 에디터도 사용해봤지만 결국 VS Code로 돌아왔는데, 그 이유는 "적절한 투자로 최고의 생산성"을 낼 수 있기 때문입니다.

이 글에서는 제가 매일 사용하면서 검증한 단축키, 확장 프로그램, 설정 팁을 체계적으로 정리합니다. 모든 설정을 한번에 적용하기보다는 하루에 1-2개씩 익히는 것을 추천합니다.

## 꼭 알아야 할 핵심 단축키: 생산성의 기본

단축키를 외우는 것만으로도 개발 속도가 눈에 띄게 빨라집니다. 아래 단축키는 Mac 기준이며, Windows에서는 `Cmd`을 `Ctrl`로 바꾸면 됩니다.

처음부터 모든 단축키를 외우려 하지 마세요. 저는 포스트잇에 "이번 주 익힐 단축키 3개"를 적어서 모니터에 붙여놓고 의식적으로 사용했습니다. 1주일이면 자연스럽게 손에 익습니다.

### 파일 탐색과 이동 - 가장 많이 쓰는 단축키들

파일 사이를 빠르게 이동하는 것은 생산성의 기본입니다. 실무에서 하루에 수십 번 사용하는 단축키들입니다.

- **`Cmd + P`**: 빠른 파일 열기. 이것 하나만 기억하세요. 프로젝트 내 모든 파일을 이름으로 검색할 수 있습니다. 파일명의 일부만 입력해도 퍼지 매칭으로 찾아줍니다. 저는 사이드바를 거의 사용하지 않고 이 단축키로 파일을 엽니다. `Cmd + P` 후에 `@`을 입력하면 현재 파일의 심볼(함수, 변수) 목록을 볼 수 있고, `:`을 입력하면 특정 라인으로 이동할 수 있습니다.

- **`Cmd + Shift + P`**: 명령 팔레트. VS Code의 모든 명령에 접근할 수 있는 만능 도구입니다. 설정 변경, 확장 프로그램 실행, 테마 변경, Git 명령, 터미널 관련 작업 등 거의 모든 작업이 가능합니다. "어떤 기능이 있는지 모르겠으면 명령 팔레트에서 검색하라"는 것이 VS Code의 철학입니다.

- **`Cmd + B`**: 사이드바 토글. 코딩에 집중할 때 사이드바를 숨기면 화면을 넓게 사용할 수 있습니다. 저는 코드를 작성할 때는 거의 항상 사이드바를 숨기고, 파일 구조를 확인할 때만 잠깐 켭니다.

- **`Ctrl + Tab`**: 열린 탭 사이 전환. 최근 사용한 파일 순서대로 정렬되어 있어 빠르게 이동할 수 있습니다.

- **`Cmd + Shift + E`**: 탐색기(Explorer) 패널로 이동.

- **`Cmd + G`**: 특정 라인으로 이동. 에러 메시지에 표시된 라인 번호로 바로 점프할 수 있습니다. 디버깅 시 매우 유용합니다.

- **`F12`**: 정의로 이동(Go to Definition). 함수나 변수의 정의 위치로 바로 점프합니다. TypeScript 프로젝트에서 특히 강력합니다.

- **`Alt + F12`**: 정의 미리보기(Peek Definition). 현재 파일을 떠나지 않고 작은 인라인 창에서 정의를 확인할 수 있습니다. 이 기능을 알고 나서 파일 이동 횟수가 절반으로 줄었습니다.

- **`Shift + F12`**: 모든 참조 찾기(Find All References). 특정 함수나 변수가 프로젝트 어디에서 사용되는지 한눈에 볼 수 있습니다. 리팩토링 전에 영향 범위를 파악할 때 필수입니다.

### 코드 편집 단축키 - 타이핑 시간을 절반으로

코드를 작성하고 수정하는 과정에서 가장 많이 사용하는 단축키들입니다. 이것들을 익히면 코드 작성 속도가 체감될 만큼 빨라집니다.

- **`Cmd + D`**: 현재 선택한 단어와 같은 다음 단어를 추가 선택. 변수명을 한꺼번에 바꿀 때 매우 유용합니다. 제가 가장 자주 사용하는 단축키 TOP 3 안에 들어갑니다. 예를 들어 `userName`이라는 변수를 `userEmail`로 바꾸고 싶으면, `userName` 하나를 선택한 후 `Cmd + D`를 여러 번 눌러 모든 `userName`을 선택하고 한꺼번에 타이핑하면 됩니다.

- **`Cmd + Shift + L`**: 현재 선택한 단어와 같은 모든 단어를 한번에 선택. `Cmd + D`의 강화 버전입니다. 파일 내 모든 동일 단어를 일괄 변경에 효과적입니다.

- **`Alt + Up/Down`**: 현재 줄을 위아래로 이동. 코드 순서를 바꿀 때 잘라내기/붙여넣기보다 훨씬 빠릅니다. import 문 순서 정리할 때도 유용합니다.

- **`Cmd + Shift + K`**: 현재 줄 삭제. 불필요한 코드를 빠르게 제거할 수 있습니다. `Ctrl + X`(잘라내기)와 비슷하지만 클립보드를 오염시키지 않는 장점이 있습니다.

- **`Alt + Shift + Up/Down`**: 현재 줄 복제. 비슷한 코드를 반복적으로 작성할 때 시간을 절약합니다.

- **`Cmd + /`**: 주석 토글. 선택한 영역이나 현재 줄을 한번에 주석 처리/해제합니다.

- **`Cmd + Shift + [` / `]`**: 코드 접기/펼치기. 긴 파일에서 구조를 파악할 때 유용합니다.

- **`Cmd + Enter`**: 현재 줄 아래에 빈 줄 삽입. 커서 위치에 상관없이 다음 줄에서 작업을 시작할 수 있습니다.

- **`Cmd + Shift + Enter`**: 현재 줄 위에 빈 줄 삽입.

- **`Cmd + L`**: 현재 줄 전체 선택. 여러 번 누르면 연속된 줄을 계속 선택합니다.

### 멀티 커서 - VS Code의 숨은 강력 기능

멀티 커서는 VS Code에서 제가 가장 사랑하는 기능입니다. 한번 익히면 "이걸 어떻게 안 쓰고 살았지?"라고 느끼게 됩니다.

- **`Alt + Click`**: 클릭한 위치에 커서 추가. 여러 줄에서 동시에 타이핑할 수 있습니다.
- **`Cmd + Alt + Up/Down`**: 위/아래에 커서 추가. 연속된 줄에서 같은 위치에 커서를 만들 때 편리합니다.
- **`Cmd + Shift + L`** 후 편집: 선택한 단어의 모든 인스턴스에 커서가 생성됩니다.

**실전 예시**: JSON 데이터를 TypeScript 인터페이스로 변환해야 하는 상황을 생각해보세요.

```json
{
  "name": "string",
  "age": "number",
  "email": "string"
}
```

이것을 TypeScript 인터페이스로 바꿀 때, 멀티 커서를 사용하면 각 줄의 `"`를 동시에 편집하여 몇 초 만에 변환할 수 있습니다. 멀티 커서 없이 하면 3분, 멀티 커서로 하면 10초입니다.

### 검색과 바꾸기 - 리팩토링의 핵심

프로젝트 전체에서 코드를 찾고 수정하는 작업은 일상적으로 발생합니다.

- **`Cmd + F`**: 현재 파일 내 검색. `Cmd + G`로 다음 결과로 이동할 수 있습니다.
- **`Cmd + H`**: 현재 파일 내 찾기 및 바꾸기.
- **`Cmd + Shift + F`**: 프로젝트 전체 검색. 특정 함수가 어디서 호출되는지 추적할 때 필수적입니다. 정규표현식 검색도 지원합니다.
- **`Cmd + Shift + H`**: 프로젝트 전체에서 찾기 및 바꾸기. API 엔드포인트 변경 등 대규모 수정 작업에 사용합니다.

**실전 팁**: 프로젝트 전체 검색(`Cmd + Shift + F`)에서 "파일 포함" 필드를 활용하면 특정 폴더나 파일 타입만 검색할 수 있습니다. 예를 들어 `src/**/*.tsx`로 범위를 제한하면 불필요한 node_modules 결과를 제외할 수 있습니다.

## 생산성을 높여주는 필수 확장 프로그램: 실사용 리뷰

VS Code의 진정한 강점은 풍부한 확장 프로그램 생태계에 있습니다. 수만 개의 확장 프로그램 중에서 제가 실제로 2년 이상 사용하며 검증한 필수 확장 프로그램들을 소개합니다. 설치만 해놓고 사용하지 않는 확장 프로그램은 리스트에서 제외했습니다.

### 코드 품질 관련 (필수 중의 필수)

**1. ESLint**
JavaScript/TypeScript 코드의 문법 오류와 스타일 문제를 실시간으로 검사합니다. 팀 프로젝트에서 코드 일관성을 유지하는 데 필수적입니다. 저희 팀에서는 ESLint 경고가 있는 코드는 PR 머지가 불가능하도록 CI에서 체크합니다.

**2. Prettier**
코드 포매팅을 자동화합니다. 저장할 때 자동으로 코드 스타일을 정리해주므로 "탭이냐 스페이스냐" 같은 스타일 논쟁에서 완전히 해방됩니다. ESLint와 함께 사용할 때는 `eslint-config-prettier`로 충돌을 방지해야 합니다.

**3. Error Lens**
에러와 경고를 코드 옆에 인라인으로 표시합니다. 이 확장이 없으면 Problems 패널을 열어야만 에러를 확인할 수 있는데, Error Lens가 있으면 코드를 작성하면서 실시간으로 에러를 볼 수 있습니다. 제가 개인적으로 "이거 없이 어떻게 코딩했지?"라고 느끼는 확장 프로그램 1위입니다.

### Git 관련

**4. GitLens**
Git 히스토리를 시각적으로 보여줍니다. 각 줄의 마지막 커밋 정보(누가, 언제, 왜 변경했는지), blame 정보, 파일 히스토리 등을 에디터 안에서 바로 확인할 수 있습니다. 코드 리뷰나 버그 추적 시 "이 코드를 왜 이렇게 작성했지?"라는 질문에 커밋 메시지로 답을 찾을 수 있습니다.

저희 팀에서 한번 프로덕션 버그가 발생했을 때, GitLens의 line blame으로 해당 코드의 변경 히스토리를 추적해서 원인 커밋을 5분 만에 찾은 적이 있습니다. GitLens 없이 `git log`로 했다면 30분은 걸렸을 겁니다.

**5. Git Graph**
Git 브랜치 구조를 그래프로 시각화합니다. 복잡한 브랜치 전략을 사용하는 프로젝트에서 현재 상태를 한눈에 파악할 수 있습니다.

### 개발 효율성

**6. Thunder Client**
VS Code 안에서 HTTP 요청을 보내고 응답을 확인할 수 있는 API 클라이언트입니다. Postman을 별도로 실행할 필요 없이 에디터 안에서 API 테스트가 가능합니다. 컬렉션 기능도 있어서 자주 사용하는 API 요청을 저장해둘 수 있습니다.

**7. Auto Rename Tag**
HTML/JSX에서 여는 태그를 수정하면 닫는 태그가 자동으로 함께 수정됩니다. React 개발 시 `<div>`를 `<section>`으로 바꿀 때 닫는 태그를 까먹는 실수를 방지합니다.

**8. Path Intellisense**
파일 경로를 입력할 때 자동 완성을 제공합니다. import 구문 작성 시 오타를 방지할 수 있습니다.

**9. TODO Highlight**
코드 내 TODO, FIXME, HACK, BUG 등의 주석을 시각적으로 강조합니다. 나중에 처리해야 할 작업을 놓치지 않도록 도와줍니다.

**10. Console Ninja**
`console.log`의 결과를 코드 옆에 인라인으로 표시합니다. 브라우저 콘솔을 확인하러 가지 않아도 됩니다. 디버깅 효율이 체감될 만큼 향상됩니다. 2025년에 알게 된 확장인데, 이후 디버깅 방식이 완전히 바뀌었습니다.

### 시각적 편의성

**11. Indent Rainbow**
들여쓰기 레벨별로 색상을 다르게 표시합니다. Python, YAML 같이 들여쓰기가 중요한 언어에서 특히 유용하고, JSX의 깊은 중첩을 파악할 때도 도움됩니다.

**12. Material Icon Theme**
파일 확장자에 따라 아이콘을 표시합니다. `.tsx`, `.test.ts`, `.config.ts` 등을 한눈에 구분할 수 있어 파일 탐색 속도가 빨라집니다.

### 확장 프로그램 성능 관리

확장 프로그램을 너무 많이 설치하면 VS Code가 느려집니다. 제 경험상 15-20개가 적정선이며, 정기적으로 사용하지 않는 확장을 정리하는 것이 중요합니다.

확장 프로그램의 성능 영향을 확인하는 방법: `Cmd + Shift + P` -> "Developer: Show Running Extensions"를 실행하면 각 확장의 활성화 시간을 확인할 수 있습니다. 500ms 이상 걸리는 확장이 있다면 대안을 찾아보는 것이 좋습니다.

## 최적의 settings.json 설정: 제 실제 설정 공개

VS Code의 설정 파일을 직접 편집하면 더 세밀한 커스터마이징이 가능합니다. `Cmd + Shift + P`로 명령 팔레트를 열고 "Open Settings (JSON)"을 검색하면 직접 편집할 수 있습니다.

아래는 제가 실제로 사용하는 settings.json의 핵심 부분입니다. 각 설정에 대한 상세 설명을 함께 달았습니다.

```json
{
  // === 에디터 기본 설정 ===
  "editor.fontSize": 14,
  "editor.lineHeight": 1.8,
  "editor.minimap.enabled": false,
  "editor.wordWrap": "on",
  "editor.tabSize": 2,
  "editor.linkedEditing": true,
  "editor.cursorBlinking": "smooth",
  "editor.cursorSmoothCaretAnimation": "on",
  "editor.smoothScrolling": true,
  "editor.stickyScroll.enabled": true,
  "editor.bracketPairColorization.enabled": true,
  "editor.guides.bracketPairs": true,
  "editor.suggestSelection": "first",
  "editor.inlineSuggest.enabled": true,

  // === 자동 포매팅 ===
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit",
    "source.organizeImports": "explicit"
  },

  // === 파일 관리 ===
  "files.autoSave": "afterDelay",
  "files.autoSaveDelay": 1000,
  "files.trimTrailingWhitespace": true,
  "files.insertFinalNewline": true,
  "files.exclude": {
    "**/.git": true,
    "**/.DS_Store": true,
    "**/node_modules": true,
    "**/.next": true
  },

  // === 워크벤치 ===
  "workbench.startupEditor": "none",
  "workbench.tree.indent": 16,
  "workbench.editor.enablePreview": false,
  "workbench.colorTheme": "One Dark Pro",

  // === 터미널 ===
  "terminal.integrated.fontSize": 13,
  "terminal.integrated.defaultProfile.osx": "zsh",

  // === 탐색기 ===
  "explorer.confirmDelete": false,
  "explorer.confirmDragAndDrop": false,
  "explorer.compactFolders": false,

  // === TypeScript ===
  "typescript.preferences.importModuleSpecifier": "non-relative",
  "typescript.suggest.autoImports": true,
  "typescript.updateImportsOnFileMove.enabled": "always"
}
```

### 주요 설정 상세 설명

- **`editor.minimap.enabled: false`**: 미니맵은 대부분의 경우 불필요합니다. 화면 공간만 차지하고 실제로 사용하는 개발자는 드뭅니다. 비활성화하면 코드 영역이 넓어집니다.

- **`editor.stickyScroll.enabled: true`**: 스크롤할 때 현재 위치의 상위 스코프(함수명, 클래스명)가 에디터 상단에 고정되어 표시됩니다. 긴 파일에서 "지금 어느 함수 안에 있지?"라는 고민을 해결해줍니다.

- **`editor.formatOnSave: true`**: 파일을 저장할 때 자동으로 Prettier가 코드를 포매팅합니다. 수동으로 포매팅할 필요가 없어집니다.

- **`source.organizeImports: "explicit"`**: 저장 시 import 문을 자동으로 정렬하고 사용하지 않는 import를 제거합니다. 코드 리뷰 때 "불필요한 import 제거해주세요"라는 코멘트를 할 일이 없어집니다.

- **`files.trimTrailingWhitespace: true`**: 저장 시 줄 끝의 불필요한 공백을 자동 제거합니다. git diff에서 공백 변경이 노이즈로 나타나는 것을 방지합니다.

- **`workbench.editor.enablePreview: false`**: 파일을 클릭하면 미리보기가 아닌 정식 탭으로 엽니다. 미리보기 모드에서는 다른 파일을 클릭하면 현재 탭이 대체되는데, 이것이 혼란스러운 분들은 이 설정을 false로 하세요.

- **`explorer.compactFolders: false`**: 하위 폴더가 하나뿐인 경우 폴더를 한 줄에 합쳐서 표시하는 기능을 비활성화합니다. `src/components/ui`가 한 줄에 표시되면 직관성이 떨어지므로 꺼두는 것을 추천합니다.

## 워크스페이스 활용 팁: 프로 개발자의 환경 관리

### 멀티 루트 워크스페이스

여러 프로젝트를 하나의 VS Code 창에서 관리할 수 있습니다. `File > Add Folder to Workspace`로 폴더를 추가하면 됩니다. 프론트엔드와 백엔드를 동시에 작업하거나, 모노레포 구조에서 여러 패키지를 한꺼번에 편집할 때 특히 유용합니다.

### 프로필 기능 - 프로젝트 유형별 최적 환경

VS Code의 프로필 기능은 생산성에 큰 차이를 만듭니다. 저는 4개의 프로필을 사용합니다.

| 프로필 | 용도 | 특징 |
|--------|------|------|
| React Dev | React/Next.js 프로젝트 | Tailwind IntelliSense, ESLint, React 관련 확장 |
| Python Dev | Python/ML 프로젝트 | Python, Jupyter, Black 포매터 |
| Writing | 블로그 글 작성 | Markdown 관련 확장, 집중 모드 테마 |
| Minimal | 간단한 스크립트 | 최소 확장만 (빠른 실행) |

프로필을 분리하면 불필요한 확장 프로그램의 로딩을 방지할 수 있어 VS Code 실행 속도도 빨라집니다.

### 스니펫 커스터마이징 - 반복 코드를 10초로

자주 반복하는 코드 패턴을 스니펫으로 등록해두면 시간을 크게 절약할 수 있습니다. `Cmd + Shift + P`에서 "Configure User Snippets"를 검색하면 직접 만들 수 있습니다.

제가 실제로 사용하는 스니펫들:

```json
{
  "React Server Component": {
    "prefix": "rsc",
    "body": [
      "export default async function ${1:ComponentName}() {",
      "  return (",
      "    <div>",
      "      $0",
      "    </div>",
      "  );",
      "}"
    ],
    "description": "React Server Component"
  },
  "React Client Component": {
    "prefix": "rcc",
    "body": [
      "\"use client\";",
      "",
      "import { useState } from \"react\";",
      "",
      "export default function ${1:ComponentName}() {",
      "  return (",
      "    <div>",
      "      $0",
      "    </div>",
      "  );",
      "}"
    ],
    "description": "React Client Component with use client directive"
  },
  "Console Log with Label": {
    "prefix": "cll",
    "body": ["console.log(\"${1:label}:\", ${2:value});"],
    "description": "Console log with descriptive label"
  },
  "Try Catch Block": {
    "prefix": "trycatch",
    "body": [
      "try {",
      "  $0",
      "} catch (error) {",
      "  console.error(\"${1:Error context}:\", error);",
      "  throw error;",
      "}"
    ]
  }
}
```

`rsc`를 입력하고 Tab을 누르면 서버 컴포넌트 보일러플레이트가 자동으로 생성됩니다. 하루에 수십 번 반복하는 코드가 있다면 반드시 스니펫으로 만드세요.

## 터미널 활용법: 에디터 안에서 모든 것을 해결

VS Code의 통합 터미널은 외부 터미널 앱을 사용하는 것보다 편리한 점이 많습니다.

- **`` Ctrl + ` ``**: 통합 터미널 토글
- **`` Ctrl + Shift + ` ``**: 새 터미널 생성
- **`Cmd + Shift + [` / `]`**: 터미널 탭 전환
- 터미널을 에디터 옆에 배치하면 코드와 터미널 출력을 동시에 볼 수 있습니다

### 터미널 분할 활용법

저는 보통 3개의 터미널을 동시에 열어둡니다.

1. **개발 서버**: `npm run dev` 실행 중
2. **Git 작업**: commit, push, branch 관리
3. **임시 명령**: 패키지 설치, 파일 조작 등

터미널 이름을 지정할 수 있으므로(터미널 탭 오른쪽 클릭 > Rename) "Dev Server", "Git", "Tasks"처럼 이름을 붙여두면 구분이 쉽습니다.

## 디버깅 활용: console.log를 졸업하자

VS Code의 내장 디버거는 `console.log`보다 훨씬 강력합니다. 저도 처음에는 "console.log로 충분하지 않나?"라고 생각했지만, 디버거를 제대로 활용하기 시작하면서 버그 추적 시간이 절반 이하로 줄었습니다.

### 기본 디버깅 단축키

- **F5**: 디버깅 시작
- **F9**: 브레이크포인트 토글
- **F10**: 스텝 오버 (함수 안으로 들어가지 않고 다음 줄로)
- **F11**: 스텝 인 (함수 안으로 들어가기)
- **Shift + F11**: 스텝 아웃 (현재 함수에서 나오기)
- **Shift + F5**: 디버깅 중지

### 조건부 브레이크포인트

일반 브레이크포인트는 해당 줄에 도달할 때마다 멈추지만, 조건부 브레이크포인트는 특정 조건이 참일 때만 멈춥니다. 예를 들어 루프에서 특정 항목일 때만 디버깅하고 싶을 때 유용합니다.

브레이크포인트를 오른쪽 클릭 > "Edit Breakpoint" > 조건 입력 (예: `item.id === "abc123"`)

### Logpoint - console.log의 진화

브레이크포인트 대신 Logpoint를 사용하면 코드를 멈추지 않고 변수 값을 출력할 수 있습니다. console.log를 코드에 삽입하지 않아도 되므로, 디버깅 후 정리할 필요가 없습니다.

브레이크포인트 오른쪽 클릭 > "Edit Breakpoint" > "Log Message" 선택 > `Variable value: {variableName}` 형식으로 입력

## 실전 워크플로우 예시: 제 하루 일과

마지막으로 제가 실제로 VS Code를 사용하는 하루 워크플로우를 공유합니다.

**아침 (코드 리뷰)**:
1. GitLens로 PR의 변경 내용 확인
2. `Cmd + Shift + F`로 관련 코드 검색
3. 인라인 코멘트 작성

**오전 (기능 개발)**:
1. `Cmd + P`로 파일 빠르게 열기
2. 스니펫(`rsc`, `rcc`)으로 컴포넌트 생성
3. 코드 작성 후 저장 시 자동 포매팅+린팅
4. 터미널에서 개발 서버 확인

**오후 (디버깅)**:
1. 버그 리포트의 라인 번호로 `Cmd + G` 이동
2. 브레이크포인트 설정 후 F5로 디버깅
3. GitLens blame으로 변경 히스토리 확인
4. 수정 후 `Cmd + Shift + F`로 영향 범위 확인

## 마무리: 작은 변화의 복리 효과

VS Code는 단순한 텍스트 에디터가 아니라 강력한 개발 환경입니다. 오늘 소개한 단축키와 설정들을 하나씩 적용해 보세요. 처음에는 익숙하지 않더라도 일주일만 꾸준히 사용하면 자연스럽게 손에 익게 됩니다.

가장 중요한 것은 자신의 워크플로우에 맞게 커스터마이징하는 것입니다. 모든 설정을 한번에 바꾸기보다는 하루에 하나씩 새로운 단축키를 익히는 것을 추천합니다. 하루에 1분을 절약하는 단축키를 하나 배우면, 1년이면 4시간을 절약합니다. 그런 단축키를 50개 배우면 연간 200시간, 즉 25일치 근무 시간을 절약하는 셈입니다.

이 글에서 소개한 내용 중 하나라도 여러분의 개발 생산성에 도움이 되었으면 좋겠습니다. 자신만의 생산성 팁이 있다면 댓글로 공유해주세요.
