코드를 push할 때마다 수동으로 테스트를 실행하고, 빌드하고, 서버에 SSH로 접속해서 `git pull`하고, `npm run build`하고, PM2를 재시작하는... 이 과정을 매번 반복하고 있었습니다. 하루에 3-4번 배포하면 매번 20분씩, 한 달이면 40시간 이상을 배포에 쏟고 있었습니다.

내가 직접 경험한 최악의 수동 배포 사고를 말씀드리면, 금요일 오후 5시에 급하게 핫픽스를 배포하다가 프로덕션 서버에서 `npm run build` 대신 `npm run dev`를 실행했습니다. 개발 모드로 프로덕션이 돌아가면서 소스맵이 노출되고, 번들 최적화가 안 된 상태로 서비스되었습니다. 고객사에서 "사이트가 엄청 느려요"라는 연락이 와서야 발견했습니다.

이 사건 이후 GitHub Actions로 CI/CD 파이프라인을 구축했고, 그 후로 배포 관련 사고가 단 한 번도 발생하지 않았습니다.

이 글에서는 CI/CD의 기본 개념부터, 제가 3개의 실제 프로젝트(Next.js 웹앱, Express API 서버, 모노레포)에서 구축한 GitHub Actions 워크플로우를 단계별로 공유합니다.

## CI/CD란? 왜 필요한가?

### CI (Continuous Integration)

코드 변경이 있을 때마다 자동으로 빌드하고 테스트하는 것입니다. 여러 개발자가 동시에 작업할 때 코드 충돌이나 버그를 빠르게 발견할 수 있습니다.

팀에서 도입했을 때 가장 큰 변화는 "PR을 올리면 자동으로 테스트가 돌아간다"는 것이었습니다. 이전에는 "테스트 돌려봤어요?"라는 질문을 코드 리뷰에서 매번 했는데, CI를 도입한 후에는 테스트가 통과하지 않으면 머지 버튼 자체가 비활성화되니 이 질문이 사라졌습니다.

### CD (Continuous Deployment / Delivery)

테스트를 통과한 코드를 자동으로 프로덕션 환경에 배포하는 것입니다.

- **Continuous Delivery**: 테스트 통과 후 배포 "준비"까지 자동화. 실제 배포는 수동 승인 후 진행.
- **Continuous Deployment**: 테스트 통과 후 프로덕션 배포까지 완전 자동화.

실제로 사용해보니 처음에는 Continuous Delivery로 시작해서 자신감이 생긴 후에 Continuous Deployment로 전환하는 것이 안전합니다. 저희 팀도 3개월간 수동 승인 방식을 유지하다가, 테스트 커버리지가 80%를 넘은 시점에서 완전 자동 배포로 전환했습니다.

### CI/CD 도입 전후 비교

| 항목 | CI/CD 도입 전 | CI/CD 도입 후 |
|------|-------------|-------------|
| 배포 소요 시간 | 20분 (수동) | 5분 (자동) |
| 배포 실수 | 월 2-3회 | 0회 |
| 배포 빈도 | 주 2-3회 | 하루 3-5회 |
| 테스트 실행 | "나중에 돌려볼게요" | 자동 (PR마다) |
| 린트 체크 | 개인에게 의존 | 자동 (강제) |
| 배포 권한 | 특정 인원만 | 자동화 (누구나 PR 머지) |

## GitHub Actions 기본 개념

### 핵심 용어

- **Workflow**: 자동화할 전체 프로세스. `.github/workflows/` 디렉토리에 YAML 파일로 정의합니다.
- **Event (Trigger)**: 워크플로우를 실행하는 이벤트. `push`, `pull_request`, `schedule` 등.
- **Job**: 워크플로우 내의 작업 단위. 같은 러너에서 실행되는 스텝의 모음.
- **Step**: Job 내의 개별 작업. 명령어를 실행하거나 액션을 사용합니다.
- **Action**: 재사용 가능한 작업 단위. GitHub Marketplace에서 다양한 액션을 찾을 수 있습니다.
- **Runner**: 워크플로우가 실행되는 서버. GitHub에서 제공하는 호스팅 러너를 무료로 사용할 수 있습니다.

### 첫 번째 워크플로우: 기본 CI

가장 기본적인 CI 워크플로우부터 시작하겠습니다.

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

      - name: Run tests
        run: npm test

      - name: Build
        run: npm run build
```

이 워크플로우는 `main` 브랜치에 push하거나 PR을 생성할 때 자동으로 실행됩니다.

## 실전 워크플로우 구성: 단계별 발전

### Level 1: 린트 + 타입 체크 + 테스트 + 빌드

```yaml
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  lint-and-type-check:
    name: Lint & Type Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"
      - run: npm ci
      - run: npm run lint
      - run: npx tsc --noEmit

  test:
    name: Unit Tests
    runs-on: ubuntu-latest
    needs: lint-and-type-check
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"
      - run: npm ci
      - run: npm test -- --coverage
      - name: Upload coverage
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage/

  build:
    name: Build
    runs-on: ubuntu-latest
    needs: [lint-and-type-check, test]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"
      - run: npm ci
      - run: npm run build
```

### Job 간 의존성 설계

`needs` 키워드로 Job의 실행 순서를 지정합니다. 위 예시에서 실행 흐름은 이렇습니다.

```
lint-and-type-check (1분)
    ├─→ test (2분)          ← lint 성공 후 실행
    └─→ build (1분 30초)    ← lint와 test 모두 성공 후 실행
```

내가 직접 경험한 팁: lint를 가장 먼저 실행하세요. lint는 30초 만에 끝나는데, 린트 에러가 있으면 테스트와 빌드를 실행할 필요가 없습니다. 이렇게 하면 실패 시 피드백을 빠르게 받을 수 있어서 무료 한도도 절약됩니다.

### Level 2: 캐싱 최적화로 빌드 시간 60% 단축

`actions/setup-node`의 `cache: "npm"` 옵션만으로도 충분하지만, Next.js의 `.next/cache`까지 캐싱하면 빌드 시간을 크게 줄일 수 있습니다.

```yaml
steps:
  - uses: actions/checkout@v4

  - uses: actions/setup-node@v4
    with:
      node-version: 20
      cache: "npm"

  - name: Cache Next.js build
    uses: actions/cache@v4
    with:
      path: |
        .next/cache
      key: nextjs-${{ hashFiles('**/package-lock.json') }}-${{ hashFiles('**/*.ts', '**/*.tsx') }}
      restore-keys: |
        nextjs-${{ hashFiles('**/package-lock.json') }}-

  - run: npm ci
  - run: npm run build
```

실제 수치로 비교하면 이렇습니다.

| 단계 | 캐싱 없음 | npm 캐싱만 | npm + Next.js 캐싱 |
|------|----------|-----------|-------------------|
| npm ci | 45초 | 12초 | 12초 |
| npm run build | 90초 | 90초 | 35초 |
| **총 시간** | **135초** | **102초** | **47초** |

### Level 3: 환경 변수와 시크릿 관리

API 키, 배포 토큰 등 민감한 정보는 GitHub 시크릿에 저장합니다. Repository Settings > Secrets and variables > Actions에서 설정합니다.

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production  # 환경별 시크릿 분리
    steps:
      - name: Deploy to production
        env:
          DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
          SENTRY_AUTH_TOKEN: ${{ secrets.SENTRY_AUTH_TOKEN }}
        run: |
          echo "Deploying with token..."
```

**시크릿 관리 팁:**

- 환경(Environment)별로 시크릿을 분리하세요 (staging, production)
- `GITHUB_TOKEN`은 자동으로 제공되므로 별도 설정이 필요 없습니다
- 시크릿은 로그에 `***`로 마스킹됩니다
- 시크릿이 포함된 값을 `echo`로 출력하면 자동 마스킹되지만, base64 인코딩 등으로 변환하면 노출될 수 있으니 주의하세요

환경별로 시크릿을 분리하는 것이 중요합니다. 팀에서 도입했을 때 처음에는 모든 시크릿을 Repository 레벨에 넣었다가, staging 배포 시 production DB 비밀번호가 사용되는 사고가 발생했습니다.

## 실전 배포 워크플로우

### Vercel 자동 배포 (테스트 통과 조건부)

Vercel은 GitHub 연동만으로 자동 배포가 가능합니다. 하지만 테스트를 통과한 후에만 배포하고 싶다면 Vercel의 자동 배포를 끄고 GitHub Actions에서 제어합니다.

```yaml
name: Deploy to Vercel

on:
  push:
    branches: [main]

env:
  VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
  VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}

jobs:
  test:
    name: Test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"
      - run: npm ci
      - run: npm run lint
      - run: npm test

  deploy:
    name: Deploy to Production
    needs: test
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4

      - name: Install Vercel CLI
        run: npm install --global vercel@latest

      - name: Pull Vercel Environment
        run: vercel pull --yes --environment=production --token=${{ secrets.VERCEL_TOKEN }}

      - name: Build
        run: vercel build --prod --token=${{ secrets.VERCEL_TOKEN }}

      - name: Deploy
        run: vercel deploy --prebuilt --prod --token=${{ secrets.VERCEL_TOKEN }}

      - name: Notify Slack
        if: success()
        run: |
          curl -X POST ${{ secrets.SLACK_WEBHOOK_URL }} \
            -H 'Content-Type: application/json' \
            -d '{"text":"Production 배포 완료! 커밋: ${{ github.sha }}"}'
```

### Preview 배포 (PR마다 자동 생성)

PR을 올리면 자동으로 Preview URL이 생성되어, 코드 리뷰어가 실제 동작을 확인할 수 있습니다.

```yaml
name: Preview Deploy

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  deploy-preview:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Vercel CLI
        run: npm install --global vercel@latest

      - name: Deploy Preview
        id: deploy
        run: |
          URL=$(vercel deploy --token=${{ secrets.VERCEL_TOKEN }})
          echo "url=$URL" >> $GITHUB_OUTPUT

      - name: Comment PR with Preview URL
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `Preview 배포 완료! ${{ steps.deploy.outputs.url }}`
            })
```

팀에서 이것을 도입했을 때 코드 리뷰의 질이 크게 향상되었습니다. "이 부분 동작 확인해주세요"라고 말하면서 Preview URL을 첨부하니, 리뷰어가 직접 클릭해서 확인할 수 있었습니다.

## 유용한 워크플로우 패턴

### 자동 라벨링

PR의 변경 파일에 따라 자동으로 라벨을 추가합니다.

```yaml
name: Auto Label

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  label:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/labeler@v5
        with:
          repo-token: ${{ secrets.GITHUB_TOKEN }}
```

`.github/labeler.yml` 파일도 함께 생성합니다.

```yaml
frontend:
  - changed-files:
    - any-glob-to-any-file: ['src/components/**', 'src/pages/**', '*.css']

backend:
  - changed-files:
    - any-glob-to-any-file: ['src/api/**', 'src/services/**']

docs:
  - changed-files:
    - any-glob-to-any-file: ['docs/**', '*.md']

config:
  - changed-files:
    - any-glob-to-any-file: ['*.config.*', '.github/**']
```

### 스케줄 실행: 주간 의존성 업데이트 체크

```yaml
name: Dependency Check

on:
  schedule:
    - cron: "0 9 * * 1"  # 매주 월요일 오전 9시 (UTC)

jobs:
  check-deps:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - name: Check outdated packages
        run: |
          npm outdated --json > outdated.json || true
          if [ -s outdated.json ]; then
            echo "Outdated packages found"
            cat outdated.json
          fi
      - name: Security audit
        run: npm audit --json > audit.json || true
```

### Matrix 전략: 여러 환경에서 동시 테스트

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20, 22]
      fail-fast: false  # 하나가 실패해도 나머지 계속 실행
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci
      - run: npm test
```

`fail-fast: false`를 설정하는 것이 중요합니다. 기본값은 `true`인데, 이 경우 하나의 Node.js 버전에서 실패하면 나머지 버전의 테스트가 즉시 취소됩니다. 모든 버전의 결과를 보고 싶다면 `false`로 설정하세요.

### 조건부 실행: 변경된 파일에 따라 다른 Job 실행

모노레포에서 특히 유용한 패턴입니다. 프론트엔드 코드만 변경됐을 때 백엔드 테스트를 실행할 필요가 없습니다.

```yaml
jobs:
  changes:
    runs-on: ubuntu-latest
    outputs:
      frontend: ${{ steps.filter.outputs.frontend }}
      backend: ${{ steps.filter.outputs.backend }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            frontend:
              - 'packages/web/**'
              - 'packages/shared/**'
            backend:
              - 'packages/api/**'
              - 'packages/shared/**'

  test-frontend:
    needs: changes
    if: needs.changes.outputs.frontend == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:frontend

  test-backend:
    needs: changes
    if: needs.changes.outputs.backend == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:backend
```

실제로 사용해보니 이 패턴으로 모노레포의 CI 실행 시간이 평균 60% 줄었습니다. 프론트엔드만 수정했을 때 백엔드 테스트 5분을 건너뛸 수 있으니까요.

## 실전 트러블슈팅: 제가 겪은 문제들

### 문제 1: 워크플로우가 무한 루프에 빠졌다

CI 워크플로우에서 자동으로 코드를 수정하고 커밋하는 로직을 넣었더니, 그 커밋이 다시 워크플로우를 트리거해서 무한 루프가 발생했습니다.

```yaml
# 해결: [skip ci] 또는 특정 토큰 사용
- name: Auto-fix and commit
  run: |
    npm run lint:fix
    git config user.name "github-actions[bot]"
    git config user.email "github-actions[bot]@users.noreply.github.com"
    git add -A
    git diff --staged --quiet || git commit -m "style: auto-fix lint errors [skip ci]"
    git push
```

`[skip ci]` 커밋 메시지를 사용하면 해당 커밋은 워크플로우를 트리거하지 않습니다.

### 문제 2: 테스트가 로컬에서는 통과하는데 CI에서 실패

대부분 타임존 차이가 원인이었습니다. 로컬은 KST(UTC+9)인데 CI 러너는 UTC입니다.

```yaml
# 해결: CI에서 타임존 설정
jobs:
  test:
    runs-on: ubuntu-latest
    env:
      TZ: Asia/Seoul
    steps:
      - run: npm test
```

### 문제 3: 무료 한도를 초과했다

GitHub Actions는 private 저장소에서 월 2,000분의 무료 한도가 있습니다. 여러 프로젝트에서 동시에 사용하면 금방 소진됩니다.

**무료 한도 절약 팁:**

1. **캐싱 적극 활용**: npm, Next.js 빌드 캐시를 반드시 설정
2. **paths 필터**: README만 수정했을 때 전체 CI가 돌지 않도록
3. **조건부 실행**: 변경된 파일에 따라 필요한 Job만 실행
4. **timeout 설정**: 무한 빌드 방지
5. **concurrency 설정**: 같은 브랜치의 이전 실행 자동 취소

```yaml
# 같은 브랜치에서 새 push가 오면 이전 실행을 취소
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
```

이 설정 하나로 월 사용량이 30% 줄었습니다. 특히 PR에서 push를 연속으로 할 때 이전 실행이 자동 취소되니 불필요한 실행이 사라집니다.

## 워크플로우 보안 가이드

### 서드파티 Action 사용 시 주의사항

```yaml
# 나쁜 예: 브랜치를 참조 (악의적 업데이트 가능)
- uses: some-action/deploy@main

# 좋은 예: 특정 커밋 SHA를 참조 (변경 불가)
- uses: some-action/deploy@a1b2c3d4e5f6

# 적절한 타협: 메이저 버전 태그 참조
- uses: actions/checkout@v4
```

### Pull Request에서의 시크릿 보호

Fork에서 생성된 PR은 시크릿에 접근할 수 없습니다. 이것은 보안을 위한 설계입니다. `pull_request_target` 이벤트를 사용하면 시크릿에 접근할 수 있지만, Fork의 코드를 실행하므로 매우 위험합니다.

```yaml
# 위험: Fork에서 악의적 코드를 실행하면 시크릿 유출 가능
on:
  pull_request_target:  # 주의! 시크릿에 접근 가능

# 안전: Fork PR에서는 시크릿 없이 테스트만 실행
on:
  pull_request:  # Fork PR에서 시크릿 접근 불가 (안전)
```

## 마무리: CI/CD는 프로젝트 초기에 설정하세요

GitHub Actions는 진입 장벽이 낮으면서도 강력한 CI/CD 도구입니다. YAML 문법만 알면 복잡한 자동화 파이프라인을 구축할 수 있습니다.

가장 중요한 조언은 **CI/CD를 프로젝트 초기에 설정하라**는 것입니다. 나중에 추가하려면 기존 코드의 린트 에러 200개, 테스트 부재, 타입 에러 50개를 한꺼번에 해결해야 합니다. 내가 직접 경험한 바로는, 프로젝트 시작 6개월 후에 CI를 도입하려다가 린트 에러 수정에만 3일이 걸렸습니다.

프로젝트 첫 커밋부터 이렇게 시작하세요.

1. **1일차**: 기본 린트 + 빌드 체크 워크플로우
2. **1주차**: 테스트 추가 (처음에는 1-2개라도)
3. **2주차**: 캐싱 최적화
4. **1개월차**: 자동 배포 (staging 먼저)
5. **2개월차**: 프로덕션 자동 배포 (수동 승인 포함)
6. **3개월차**: 완전 자동 배포 (테스트 커버리지 충분할 때)

CI/CD는 한 번 설정하면 매일 당신을 대신해서 일합니다. 처음 설정에 투자하는 4-8시간이 앞으로 매달 40시간을 절약해줍니다. 이보다 ROI가 좋은 투자가 어디 있을까요?
