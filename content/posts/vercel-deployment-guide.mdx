웹 프로젝트를 만들었다면 다음 단계는 배포입니다. 저는 이 블로그를 포함해서 총 6개의 Next.js 프로젝트를 Vercel에 배포해서 운영하고 있습니다. 처음 배포할 때는 "버튼 하나면 끝"이라는 말만 믿고 시작했다가, 환경변수 하나 때문에 3시간을 삽질한 적도 있습니다.

이 글에서는 **진짜 처음부터 끝까지** Vercel 배포 과정을 다룹니다. 단순히 "Deploy 버튼을 누르세요"가 아니라, 각 단계에서 **화면에 뭐가 보이는지**, **어디서 실수하기 쉬운지**, **배포 후 뭘 해야 하는지**까지 상세하게 공유합니다.

## 왜 Vercel인가?

### Next.js와의 완벽한 호환

Vercel은 Next.js를 만든 회사에서 운영하는 클라우드 플랫폼입니다. 당연히 Next.js의 모든 기능을 네이티브로 지원합니다. App Router, Server Components, Edge Runtime, 이미지 최적화, ISR(Incremental Static Regeneration) 등 별도 설정 없이 동작합니다.

다른 플랫폼(Netlify, Cloudflare Pages)에서 Next.js를 배포하면 일부 기능이 제한되거나 추가 설정이 필요합니다. 예를 들어 Netlify에서 Next.js의 서버 액션을 사용하려면 어댑터 설정이 필요한데, Vercel에서는 그냥 됩니다. 이 차이가 작아 보이지만, **트러블슈팅에 쏟는 시간을 생각하면 큰 차이**입니다.

### 자동 배포

GitHub 저장소를 연결하면 코드를 push할 때마다 자동으로 빌드하고 배포합니다. PR을 생성하면 프리뷰 배포가 자동으로 생성되어 변경사항을 실제 환경에서 확인할 수 있습니다.

### 글로벌 CDN

Vercel의 Edge Network는 전 세계에 분산되어 있어 어디서든 빠른 응답 속도를 제공합니다. 한국에서의 응답 속도도 우수합니다. 실제 제 블로그 기준으로 **정적 페이지의 TTFB(Time to First Byte)가 50-80ms** 수준입니다.

### 무료 제공 기능 (Hobby 플랜)

- HTTPS (SSL 인증서 자동 발급)
- 커스텀 도메인 연결
- 자동 CI/CD
- 프리뷰 배포
- 서버리스 함수 (월 100GB-Hours)
- Edge Functions
- Analytics (기본)
- 대역폭 100GB/월
- 빌드 시간 6,000분/월

개인 블로그나 포트폴리오에는 충분하고도 남는 수준입니다.

## 실전 배포: 처음부터 끝까지

실제 제가 이 블로그를 배포했던 과정을 그대로 재현합니다.

### 1단계: 프로젝트 준비

먼저 프로젝트가 로컬에서 정상 동작하는지 확인합니다. **이 단계를 건너뛰면 안 됩니다.** Vercel에서 빌드가 실패하면 원인을 찾기 어려우니까요.

```bash
# 의존성 설치
npm install

# 빌드 테스트 - 이것이 성공해야 Vercel에서도 성공합니다
npm run build

# 빌드된 결과물로 로컬 실행
npm start
```

`npm run build`에서 에러가 나면 Vercel에서도 반드시 에러가 납니다. **로컬 빌드 성공 = Vercel 배포 성공의 전제 조건**입니다.

흔한 빌드 에러 원인:
- TypeScript 타입 에러 (`next build`는 타입 체크를 수행합니다)
- ESLint 에러 (기본 설정에서 lint도 실행됩니다)
- 이미지 import 경로 오류
- 환경 변수 누락 (이건 나중에 자세히 다룹니다)

### 2단계: GitHub 저장소에 Push

```bash
# 아직 git 초기화를 안 했다면
git init
git add .
git commit -m "Initial commit"

# GitHub에서 새 저장소 생성 후
git remote add origin https://github.com/username/my-blog.git
git branch -M main
git push -u origin main
```

**중요**: `.env.local` 파일이 `.gitignore`에 포함되어 있는지 반드시 확인하세요. API 키나 시크릿이 GitHub에 올라가면 큰일납니다. Next.js 프로젝트를 `create-next-app`으로 생성했다면 `.gitignore`에 기본 포함되어 있지만, 직접 확인하는 것이 좋습니다.

### 3단계: Vercel 가입

[vercel.com](https://vercel.com)에 접속합니다. "Sign Up" 버튼을 클릭하면 GitHub, GitLab, Bitbucket 계정으로 가입할 수 있습니다.

**화면에 보이는 것**: 가입 화면에서 "Continue with GitHub"를 선택하면 GitHub OAuth 인증 화면이 뜹니다. "Authorize Vercel" 버튼을 클릭하면 됩니다. Vercel이 GitHub 저장소에 접근할 수 있는 권한을 요청합니다.

가입이 완료되면 Vercel 대시보드로 이동합니다. 빈 화면에 "Add New..." 버튼이 보입니다.

### 4단계: 프로젝트 임포트

대시보드에서 **"Add New > Project"**를 클릭합니다.

**화면에 보이는 것**: "Import Git Repository" 섹션에 GitHub 저장소 목록이 나타납니다. 방금 push한 저장소를 찾아서 "Import" 버튼을 클릭합니다. 저장소가 보이지 않으면 "Adjust GitHub App Permissions" 링크를 클릭하여 Vercel의 GitHub 접근 권한을 확인하세요.

**화면에 보이는 것 (Configure Project)**: 프레임워크 감지 결과가 표시됩니다.

```
Framework Preset: Next.js (자동 감지)
Root Directory: ./ (모노레포가 아니면 기본값)
Build Command: next build (자동 설정)
Output Directory: .next (자동 설정)
Install Command: npm install (자동 설정)
```

대부분의 Next.js 프로젝트에서는 이 설정을 변경할 필요가 없습니다. **그냥 두세요.**

### 5단계: 환경 변수 설정 (여기서 실수하면 3시간 삽질합니다)

"Deploy" 버튼을 누르기 전에 **반드시** 환경 변수를 설정해야 합니다.

**화면에 보이는 것**: "Configure Project" 화면 하단에 "Environment Variables" 섹션이 있습니다. 여기서 Key-Value 쌍을 입력합니다.

```
NEXT_PUBLIC_SITE_URL=https://your-domain.com
NEXT_PUBLIC_SITE_NAME=My Blog
NEXT_PUBLIC_GA_MEASUREMENT_ID=G-XXXXXXXXXX
NEXT_PUBLIC_ADSENSE_CLIENT_ID=ca-pub-XXXXXXXXXX
DATABASE_URL=postgresql://...
```

#### 환경변수 관련 삽질 경험담

저는 처음 배포할 때 이 단계를 건너뛰었습니다. "나중에 설정하면 되겠지"라고 생각했거든요. 결과는?

**빌드는 성공했습니다.** 하지만 사이트에 접속하니 **Google Analytics가 동작하지 않고, AdSense 코드가 비어 있고, Open Graph 이미지 URL이 localhost를 가리키고 있었습니다.**

"왜 Analytics가 안 되지?" → 브라우저 콘솔 확인 → GA ID가 `undefined` → 아, 환경변수! → Vercel 대시보드에서 추가 → 재배포 → 여전히 안 됨 → **알고 보니 환경변수 이름에 오타** (`NEXT_PUBLC_GA_ID`로 적었음) → 수정 → 재배포 → 이번엔 Analytics는 되는데 사이트맵의 URL이 여전히 이상함 → `NEXT_PUBLIC_SITE_URL`을 빼먹음 → 추가 → 재배포...

**이렇게 3시간이 날아갔습니다.**

교훈:
1. **로컬의 `.env.local` 파일에 있는 모든 변수를 Vercel에도 설정하세요**
2. **변수 이름을 복사-붙여넣기 하세요** (오타 방지)
3. **`NEXT_PUBLIC_` 접두사를 빼먹지 마세요** (클라이언트에서 사용하는 변수는 반드시 이 접두사가 필요합니다)
4. **환경변수 설정 후에는 반드시 재배포해야 적용됩니다**

#### 환경별 환경변수 분리

Vercel에서는 환경변수를 **Production**, **Preview**, **Development** 3개 환경별로 다르게 설정할 수 있습니다.

예를 들어:
- Production: `DATABASE_URL`이 프로덕션 DB를 가리킴
- Preview: `DATABASE_URL`이 스테이징 DB를 가리킴
- Development: `DATABASE_URL`이 로컬 DB를 가리킴

이 기능을 활용하면 PR 프리뷰 배포에서 프로덕션 DB를 건드리는 사고를 방지할 수 있습니다. **반드시 설정하세요.**

### 6단계: 배포

"Deploy" 버튼을 클릭합니다.

**화면에 보이는 것**: 빌드 로그가 실시간으로 스크롤됩니다. 주요 단계:

```
Cloning github.com/username/my-blog (Branch: main, Commit: abc1234)
Installing dependencies...
npm install completed in 15s
Running build command: next build
Creating an optimized production build...
Compiled successfully
Generating static pages (10/10)
Build completed in 45s
Deploying to production...
```

**빌드 시간 참고**: 제 블로그(약 30개 포스트, MDX 사용) 기준 빌드 시간이 **약 40-50초**입니다. 정적 페이지가 많을수록 빌드 시간이 늘어납니다.

빌드가 완료되면 축하 화면과 함께 배포 URL이 표시됩니다. `https://my-blog-abc123.vercel.app` 같은 형태입니다.

## 커스텀 도메인 연결

### 도메인 구매

가비아, Namecheap, GoDaddy 등에서 도메인을 구매합니다. 한국 사이트의 경우 가비아가 편합니다.

**비용 참고**:
- `.com` 도메인: 연 1.5-2만 원
- `.dev` 도메인: 연 1.5-2만 원
- `.kr` 도메인: 연 2-3만 원
- `.io` 도메인: 연 5-7만 원

### DNS 설정

Vercel 대시보드의 **Settings > Domains**에서 도메인을 추가합니다.

**화면에 보이는 것**: 도메인 입력 필드에 `yourdomain.com`을 입력하면, Vercel이 DNS 설정 가이드를 보여줍니다. 두 가지 방법이 있습니다.

**방법 1: A/CNAME 레코드 추가 (간단)**

도메인 등록 업체의 DNS 관리에서 다음 레코드를 추가합니다:

```
A    @    76.76.21.21
CNAME www  cname.vercel-dns.com
```

**방법 2: Nameserver 변경 (권장)**

도메인의 네임서버를 Vercel로 변경합니다. 이 방법이 더 좋은 이유는 Vercel에서 DNS를 직접 관리할 수 있기 때문입니다.

```
ns1.vercel-dns.com
ns2.vercel-dns.com
```

DNS 전파에는 **최대 48시간**이 걸릴 수 있습니다. 하지만 실제로는 대부분 **30분-2시간** 이내에 완료됩니다.

SSL 인증서는 Vercel이 Let's Encrypt를 통해 자동 발급합니다. 갱신도 자동입니다.

## 배포 후 3시간 동안 삽질한 이유: 환경변수 하나

위에서 잠깐 언급했지만, 좀 더 자세한 이야기를 공유합니다. 이건 제가 두 번째 프로젝트를 배포할 때 겪은 일입니다.

프로젝트는 Next.js + Supabase를 사용하는 풀스택 앱이었습니다. 로컬에서 완벽하게 동작했고, Vercel에 배포도 성공했습니다. 하지만 배포된 사이트에서 로그인이 안 되었습니다.

```
Error: Invalid API key
```

"분명 환경변수 다 설정했는데?" → Vercel 대시보드에서 확인 → `NEXT_PUBLIC_SUPABASE_URL`과 `NEXT_PUBLIC_SUPABASE_ANON_KEY`가 있음 → 값도 맞음 → 그런데 왜?

**30분 후**: 혹시 빌드 타임에 환경변수가 주입되는 건 아닐까? → `.env.local`과 Vercel 환경변수를 비교 → 값이 동일함 → 다시 빌드해봄 → 여전히 같은 에러

**1시간 후**: Supabase 대시보드에서 API 키를 재발급 → Vercel에 업데이트 → 재배포 → 여전히 같은 에러 → 미치겠음

**2시간 후**: 빌드 로그를 자세히 보기 시작 → 환경변수 목록을 출력하는 디버그 코드를 추가 → 재배포 → **`SUPABASE_SERVICE_ROLE_KEY`가 누락**되어 있음을 발견

**알고 보니**: 서버 사이드에서 사용하는 `SUPABASE_SERVICE_ROLE_KEY`(NEXT_PUBLIC 접두사 없음)를 설정하지 않았던 것입니다. 클라이언트 쪽 환경변수만 신경 쓰고, 서버 쪽을 까먹었습니다.

**3시간 후**: 서버 쪽 환경변수 추가 → 재배포 → 드디어 동작!

### 환경변수 체크리스트

이 경험 이후, 저는 배포 전 항상 이 체크리스트를 확인합니다:

```bash
# .env.local의 모든 변수가 Vercel에도 있는지 확인
# 로컬 .env.local
cat .env.local | grep -v "^#" | grep "=" | cut -d'=' -f1 | sort > /tmp/local_vars.txt

# Vercel 환경변수 목록과 비교 (Vercel CLI 사용)
vercel env ls | sort > /tmp/vercel_vars.txt

# 차이 확인
diff /tmp/local_vars.txt /tmp/vercel_vars.txt
```

## 성능 최적화: 배포 전후 비교

배포만 하면 끝이 아닙니다. 성능 최적화를 하면 사용자 경험이 확 달라집니다.

### 최적화 전 (이 블로그 초기 배포)

| 지표 | 값 | 등급 |
|------|-----|------|
| LCP (Largest Contentful Paint) | 3.2초 | 나쁨 |
| FID (First Input Delay) | 180ms | 개선 필요 |
| CLS (Cumulative Layout Shift) | 0.15 | 개선 필요 |
| Lighthouse Performance | 62점 | 나쁨 |
| 페이지 크기 | 1.8MB | 과다 |

### 최적화 작업

**1. 이미지 최적화**

Next.js의 `Image` 컴포넌트로 모든 이미지를 교체했습니다.

```tsx
// Before: HTML img 태그 직접 사용
<img src="/images/hero.png" alt="히어로 이미지" />

// After: Next.js Image 컴포넌트
import Image from "next/image";
<Image
  src="/images/hero.png"
  alt="히어로 이미지"
  width={800}
  height={400}
  priority // LCP 이미지에만 적용
  placeholder="blur"
/>
```

효과: WebP/AVIF 자동 변환, 디바이스 맞춤 리사이징, lazy loading. **이것만으로 페이지 크기가 1.8MB → 680KB로 줄었습니다.**

**2. 폰트 최적화**

```tsx
// next/font를 사용한 폰트 최적화
import { Noto_Sans_KR } from "next/font/google";

const notoSansKr = Noto_Sans_KR({
  subsets: ["latin"],
  weight: ["400", "700"],
  display: "swap",
});
```

`next/font`를 사용하면 Google Fonts를 빌드 타임에 다운로드하여 셀프 호스팅합니다. **외부 네트워크 요청이 줄어들어 LCP가 개선됩니다.**

**3. 서드파티 스크립트 지연 로딩**

```tsx
import Script from "next/script";

// Google Analytics - 페이지 로드 후 로딩
<Script
  src={`https://www.googletagmanager.com/gtag/js?id=${gaId}`}
  strategy="afterInteractive"
/>

// AdSense - 유휴 시간에 로딩
<Script
  src="https://pagead2.googlesyndication.com/pagead/js/adsbygoogle.js"
  strategy="lazyOnload"
/>
```

**4. 번들 분석 및 코드 스플리팅**

```bash
# 번들 크기 분석
npm install @next/bundle-analyzer

# next.config.js에 분석기 추가 후
ANALYZE=true npm run build
```

번들 분석 결과 `moment.js`(300KB)가 포함되어 있었는데, 날짜 포맷팅에만 사용하고 있었습니다. `dayjs`(2KB)로 교체했습니다.

### 최적화 후

| 지표 | 전 | 후 | 개선율 |
|------|-----|-----|--------|
| LCP | 3.2초 | 1.1초 | 66% 개선 |
| FID | 180ms | 45ms | 75% 개선 |
| CLS | 0.15 | 0.02 | 87% 개선 |
| Lighthouse Performance | 62점 | 94점 | 52% 개선 |
| 페이지 크기 | 1.8MB | 420KB | 77% 감소 |

**체감 차이가 엄청납니다.** 특히 모바일에서 "뭔가 느리다"는 느낌이 완전히 사라졌습니다.

## CI/CD 파이프라인: GitHub Actions 연동

Vercel의 기본 자동 배포도 좋지만, **테스트와 린트를 통과한 코드만 배포**하고 싶다면 GitHub Actions를 추가 설정합니다.

### 기본 CI/CD 워크플로우

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"

      - run: npm ci

      - name: TypeScript 타입 체크
        run: npx tsc --noEmit

      - name: ESLint
        run: npm run lint

      - name: 단위 테스트
        run: npm test -- --coverage

      - name: 빌드 테스트
        run: npm run build
```

이 워크플로우를 설정하면:
1. PR을 올릴 때 자동으로 타입 체크, 린트, 테스트, 빌드를 실행합니다
2. 모든 체크를 통과해야 머지할 수 있도록 Branch Protection Rule을 설정할 수 있습니다
3. main에 머지되면 Vercel이 자동으로 프로덕션 배포를 수행합니다

### Vercel과 GitHub Actions 연동 팁

Vercel은 기본적으로 모든 push에 대해 배포를 트리거합니다. GitHub Actions와 함께 사용할 때는 **"CI가 통과한 경우에만 배포"**하도록 설정하는 것이 좋습니다.

Vercel 대시보드에서 **Settings > Git > Ignored Build Step**에 다음을 입력합니다:

```bash
# CI가 실패하면 배포를 건너뛰는 스크립트
git log -1 --pretty=%B | grep -q "\[skip deploy\]" && exit 0 || exit 1
```

또는 Vercel CLI를 GitHub Actions에서 직접 사용하는 방법도 있습니다:

```yaml
- name: Vercel 배포
  if: github.ref == 'refs/heads/main'
  run: |
    npm i -g vercel
    vercel pull --yes --environment=production --token=${{ secrets.VERCEL_TOKEN }}
    vercel build --prod --token=${{ secrets.VERCEL_TOKEN }}
    vercel deploy --prebuilt --prod --token=${{ secrets.VERCEL_TOKEN }}
```

## 모니터링과 알림 설정

배포 후 "잘 돌아가겠지"라고 방치하면 안 됩니다. 장애를 빨리 감지해야 합니다.

### Vercel Analytics 설정

```bash
npm install @vercel/analytics @vercel/speed-insights
```

```tsx
// app/layout.tsx
import { Analytics } from "@vercel/analytics/react";
import { SpeedInsights } from "@vercel/speed-insights/next";

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="ko">
      <body>
        {children}
        <Analytics />
        <SpeedInsights />
      </body>
    </html>
  );
}
```

이렇게 설정하면 Vercel 대시보드에서:
- **Web Vitals**: LCP, FID, CLS, TTFB 실시간 모니터링
- **페이지별 방문자 수**: 어떤 페이지가 인기 있는지
- **에러율**: 서버 에러(500) 발생 빈도

### UptimeRobot으로 다운타임 알림

Vercel 자체는 매우 안정적이지만, **내 코드의 버그로 인한 장애**는 별개입니다. [UptimeRobot](https://uptimerobot.com)을 설정하면 사이트가 다운되었을 때 즉시 알림을 받을 수 있습니다. 무료 플랜으로 5분 간격 모니터링이 가능합니다.

### Sentry로 런타임 에러 추적

```bash
npx @sentry/wizard@latest -i nextjs
```

Sentry를 설정하면 프로덕션에서 발생하는 JavaScript 에러를 실시간으로 추적할 수 있습니다. "어떤 사용자가 어떤 페이지에서 어떤 에러를 만났는지" 상세 정보를 확인할 수 있습니다.

## 비용 분석: 언제 유료 플랜으로 업그레이드할까?

### Hobby 플랜 (무료)의 한계

| 리소스 | Hobby (무료) | Pro ($20/월) |
|--------|-------------|-------------|
| 대역폭 | 100GB/월 | 1TB/월 |
| 빌드 시간 | 6,000분/월 | 24,000분/월 |
| 서버리스 함수 실행 | 100GB-Hours | 1,000GB-Hours |
| 이미지 최적화 | 1,000개/월 | 5,000개/월 |
| 팀 멤버 | 1명 | 무제한 |
| 상업적 사용 | 불가 | 가능 |

### 업그레이드 시점 판단 기준

제 경험상 다음 상황이 오면 업그레이드를 고려해야 합니다:

**1. 트래픽이 월 3만 PV를 넘을 때**
- 대역폭 100GB가 빠듯해지기 시작합니다
- 이미지 최적화 1,000개 제한에 걸릴 수 있습니다

**2. 상업적 목적으로 사용할 때**
- Hobby 플랜은 개인/비상업적 용도만 허용합니다
- 회사 프로젝트, 수익 창출 목적이면 Pro 이상이 필요합니다

**3. 서버리스 함수를 많이 사용할 때**
- API 라우트가 많고 트래픽이 있으면 100GB-Hours가 부족합니다
- 특히 DB 조회가 포함된 API는 실행 시간이 길어집니다

**4. 팀으로 개발할 때**
- Hobby 플랜은 1인 전용입니다
- 팀 협업이 필요하면 Pro 플랜이 필수입니다

### 비용 최적화 팁

- **정적 생성(SSG)을 최대한 활용하세요**: 서버리스 함수 호출이 줄어듭니다
- **ISR(Incremental Static Regeneration)로 서버 부하를 줄이세요**: 매 요청마다 렌더링하는 대신 일정 간격으로 재생성
- **이미지는 외부 CDN 사용도 고려하세요**: Cloudinary, imgix 등을 사용하면 Vercel의 이미지 최적화 횟수를 아낄 수 있습니다
- **Edge Runtime을 활용하세요**: 서버리스 함수보다 실행 비용이 낮습니다

| 기능 | Vercel | Netlify | Cloudflare Pages | Railway |

| 무료 대역폭 | 100GB/월 | 100GB/월 | 무제한 | 사용량 과금 |

### SaaS 프로젝트 배포 시 추가 고려사항

SaaS 프로젝트는 블로그와 다르게 **데이터베이스, 인증, 결제** 등 외부 서비스와의 연동이 필요합니다.

**필수 환경변수 체크리스트 (SaaS):**

```
# 데이터베이스
DATABASE_URL=postgresql://...
DIRECT_URL=postgresql://...  (Prisma 사용 시 필요)

# 인증
NEXTAUTH_URL=https://your-app.com
NEXTAUTH_SECRET=your-secret-key  (openssl rand -base64 32로 생성)
GOOGLE_CLIENT_ID=...
GOOGLE_CLIENT_SECRET=...

# 결제
STRIPE_SECRET_KEY=sk_live_...
STRIPE_WEBHOOK_SECRET=whsec_...
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=pk_live_...

# 이메일
RESEND_API_KEY=re_...

# 모니터링
SENTRY_DSN=https://...@sentry.io/...
```

**실수하기 쉬운 포인트**:
- Stripe webhook secret은 Vercel 배포 URL이 확정된 후에 Stripe 대시보드에서 설정해야 합니다
- `NEXTAUTH_URL`은 반드시 배포 도메인과 일치해야 합니다 (localhost 아님!)
- `DATABASE_URL`은 Connection Pooling URL을 사용해야 서버리스 환경에서 안정적입니다

### 모노레포 프로젝트 배포

모노레포(turborepo, nx 등)에서 특정 앱만 Vercel에 배포하려면 "Root Directory"를 설정해야 합니다.

예를 들어 다음 구조에서:
```
my-monorepo/
├── apps/
│   ├── web/        ← 이것만 배포
│   └── admin/
├── packages/
│   ├── ui/
│   └── config/
└── turbo.json
```

Vercel 프로젝트 설정에서:
- **Root Directory**: `apps/web`
- **Build Command**: `cd ../.. && npx turbo run build --filter=web`
- **Install Command**: `cd ../.. && npm install`

이 설정이 없으면 Vercel이 루트 디렉토리에서 빌드하려다가 실패합니다. 제가 처음 모노레포를 배포할 때 이것 때문에 1시간을 삽질했습니다.

## 트러블슈팅 모음

### 빌드 실패: "Module not found"

```
Module not found: Can't resolve './components/Button'
```

**원인**: 대소문자 불일치. macOS는 대소문자를 구분하지 않지만, Vercel의 빌드 서버(Linux)는 구분합니다.
**해결**: 파일명과 import 경로의 대소문자를 정확히 맞추세요.

### 빌드 실패: "Out of memory"

```
FATAL ERROR: Ineffective mark-compacts near heap limit Allocation failed
```

**원인**: 빌드 시 메모리 부족.
**해결**: `next.config.js`에서 `experimental.workerThreads: false` 설정하거나, Vercel 대시보드에서 빌드 메모리를 늘립니다.

### API 라우트에서 504 Timeout

**원인**: 서버리스 함수의 기본 타임아웃은 Hobby 플랜에서 10초입니다.
**해결**: 함수 실행 시간을 최적화하거나, Pro 플랜(최대 300초)으로 업그레이드합니다.

### 환경변수가 적용되지 않음

**원인**: 환경변수 추가 후 재배포를 하지 않았거나, `NEXT_PUBLIC_` 접두사를 빼먹은 경우.
**해결**: 환경변수 설정 후 반드시 재배포. 클라이언트에서 사용하는 변수는 `NEXT_PUBLIC_` 접두사 필수.

## 마무리

Vercel을 사용하면 배포 과정에 대한 걱정 없이 개발에 집중할 수 있습니다. GitHub에 push하면 자동으로 배포되고, HTTPS, CDN, 도메인 관리까지 모두 해결됩니다.

이 글에서 공유한 삽질 경험이 여러분의 시간을 아껴주었으면 좋겠습니다. 핵심을 정리하면:

1. **로컬에서 `npm run build` 성공을 먼저 확인하세요**
2. **환경변수는 복사-붙여넣기로 정확하게 설정하세요**
3. **배포 후 성능 최적화를 반드시 하세요** (Lighthouse 90점 이상 목표)
4. **모니터링을 설정해서 장애를 빨리 감지하세요**
5. **무료 플랜으로 시작하고, 필요할 때 업그레이드하세요**

무료 플랜으로도 개인 블로그나 포트폴리오 사이트를 충분히 운영할 수 있습니다. 저는 월 2만 PV 정도의 이 블로그를 아직 Hobby 플랜으로 운영하고 있습니다. 배포, 어렵지 않습니다. 시작해보세요.
