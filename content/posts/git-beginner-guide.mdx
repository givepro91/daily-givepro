Git은 모든 개발자가 반드시 알아야 할 버전 관리 도구입니다. 코드의 변경 이력을 추적하고, 여러 개발자가 동시에 작업할 수 있게 해주며, 실수로 코드를 망쳤을 때 이전 상태로 돌아갈 수 있게 해줍니다. 이 글에서는 Git의 기본 개념부터 실무에서 자주 사용하는 명령어까지 체계적으로 정리합니다.

## Git이란 무엇인가?

Git은 리누스 토르발즈가 2005년에 만든 분산 버전 관리 시스템(DVCS)입니다. "분산"이라는 단어가 의미하는 것은 모든 개발자가 전체 저장소의 복사본을 가지고 있다는 뜻입니다. 중앙 서버가 다운되어도 로컬에서 계속 작업할 수 있고, 인터넷 연결 없이도 커밋을 만들 수 있습니다.

### Git과 GitHub의 차이

많은 초보자가 Git과 GitHub를 혼동합니다.

- **Git**: 버전 관리 소프트웨어. 로컬 컴퓨터에서 실행됩니다.
- **GitHub**: Git 저장소를 호스팅하는 클라우드 서비스. 코드 공유, 협업, 이슈 관리 등의 추가 기능을 제공합니다.

Git 외에도 GitHub, GitLab, Bitbucket 등 여러 호스팅 서비스가 있지만, GitHub가 가장 널리 사용됩니다.

## Git 설치와 초기 설정

### 설치

Mac에서는 Xcode Command Line Tools를 설치하면 Git이 함께 설치됩니다.

```bash
xcode-select --install
```

Windows에서는 Git 공식 사이트에서 설치 프로그램을 다운로드합니다.

### 초기 설정

설치 후 사용자 정보를 설정합니다. 이 정보는 모든 커밋에 기록됩니다.

```bash
git config --global user.name "Your Name"
git config --global user.email "your@email.com"
```

## Git의 핵심 개념

### 세 가지 영역

Git에는 세 가지 주요 영역이 있습니다. 이 개념을 이해하는 것이 Git을 마스터하는 첫걸음입니다.

1. **Working Directory (작업 디렉토리)**: 실제 파일을 편집하는 공간. 아직 Git이 추적하지 않는 변경사항이 여기에 있습니다.
2. **Staging Area (스테이징 영역)**: 다음 커밋에 포함시킬 변경사항을 준비하는 공간. `git add`로 파일을 이곳으로 옮깁니다.
3. **Repository (저장소)**: 커밋된 변경사항이 영구적으로 저장되는 공간. `git commit`으로 스테이징 영역의 내용을 저장합니다.

### 커밋이란?

커밋은 특정 시점의 프로젝트 상태를 스냅샷으로 저장하는 것입니다. 각 커밋에는 고유한 해시 ID, 작성자, 날짜, 메시지가 포함됩니다. 커밋 메시지는 "무엇을 왜 변경했는지"를 설명해야 합니다.

## 필수 Git 명령어

### 저장소 초기화와 클론

```bash
# 새 저장소 만들기
git init

# 기존 저장소 복제하기
git clone https://github.com/user/repo.git
```

### 변경사항 추적

```bash
# 현재 상태 확인
git status

# 변경 내용 확인
git diff

# 스테이징에 추가
git add 파일명         # 특정 파일
git add .             # 모든 변경사항

# 커밋 만들기
git commit -m "변경 내용 설명"
```

### 히스토리 확인

```bash
# 커밋 히스토리 보기
git log

# 한 줄로 보기
git log --oneline

# 그래프로 보기
git log --oneline --graph --all
```

### 원격 저장소 작업

```bash
# 원격 저장소 추가
git remote add origin https://github.com/user/repo.git

# 변경사항 올리기
git push origin main

# 변경사항 가져오기
git pull origin main
```

## 브랜치 전략

브랜치는 Git의 가장 강력한 기능 중 하나입니다. 독립적인 작업 공간을 만들어 기능 개발, 버그 수정 등을 메인 코드에 영향을 주지 않고 진행할 수 있습니다.

### 기본 브랜치 명령어

```bash
# 브랜치 목록 확인
git branch

# 새 브랜치 생성
git branch feature/login

# 브랜치 전환
git switch feature/login

# 생성과 전환을 한번에
git switch -c feature/login

# 브랜치 병합
git switch main
git merge feature/login

# 브랜치 삭제
git branch -d feature/login
```

### 브랜치 이름 규칙

팀에서 일관된 브랜치 이름을 사용하면 관리가 쉬워집니다.

- `feature/기능명`: 새 기능 개발
- `fix/버그명`: 버그 수정
- `hotfix/긴급수정`: 프로덕션 긴급 수정
- `refactor/대상`: 코드 리팩토링
- `docs/문서명`: 문서 작업

## 충돌 해결하기

두 브랜치에서 같은 파일의 같은 부분을 수정하면 병합 시 충돌이 발생합니다. Git은 충돌 부분을 마커로 표시합니다.

```
<<<<<<< HEAD
현재 브랜치의 코드
=======
병합하려는 브랜치의 코드
>>>>>>> feature/login
```

충돌을 해결하려면 마커를 제거하고 최종적으로 남길 코드를 선택한 후 커밋하면 됩니다. VS Code 같은 에디터에서는 충돌 해결 UI를 제공하므로 더 쉽게 처리할 수 있습니다.

## 실수 되돌리기

### 스테이징 취소

```bash
# 스테이징된 파일을 작업 디렉토리로 되돌리기
git restore --staged 파일명
```

### 커밋 수정

```bash
# 마지막 커밋 메시지 수정 (아직 push하지 않은 경우)
git commit --amend -m "수정된 메시지"
```

### 커밋 되돌리기

```bash
# 특정 커밋의 변경사항을 되돌리는 새 커밋 생성
git revert 커밋해시
```

`git revert`는 히스토리를 유지하면서 변경을 되돌리므로 팀 프로젝트에서 안전합니다. `git reset`은 히스토리를 변경하므로 이미 push한 커밋에는 사용하지 마세요.

## .gitignore 설정

Git으로 추적하지 않을 파일을 `.gitignore`에 정의합니다.

```gitignore
# 의존성
node_modules/

# 빌드 결과물
.next/
dist/

# 환경 변수
.env
.env.local

# OS 파일
.DS_Store
Thumbs.db

# IDE 설정
.idea/
.vscode/settings.json
```

프로젝트 시작 시 `.gitignore`를 먼저 설정하세요. 한번 추적된 파일은 `.gitignore`에 추가해도 추적이 계속됩니다.

## 좋은 커밋 메시지 작성법

커밋 메시지는 미래의 자신과 동료를 위한 문서입니다.

### 좋은 예시

```
feat: 사용자 로그인 기능 추가

- 이메일/비밀번호 기반 로그인 구현
- JWT 토큰 발급 및 검증 로직 추가
- 로그인 실패 시 에러 메시지 표시
```

### 나쁜 예시

```
수정함
update
fix bug
asdf
```

Conventional Commits 규약을 따르면 일관된 메시지를 작성할 수 있습니다. `feat:`, `fix:`, `docs:`, `refactor:`, `test:` 등의 접두어를 사용합니다.

## 실전 워크플로우

### 개인 프로젝트

1. `main` 브랜치에서 새 기능 브랜치 생성
2. 기능 개발 후 커밋
3. `main`에 병합
4. 원격 저장소에 push

### 팀 프로젝트 (GitHub Flow)

1. `main`에서 기능 브랜치 생성
2. 커밋을 만들며 개발
3. GitHub에 push
4. Pull Request 생성
5. 코드 리뷰 진행
6. 승인 후 `main`에 병합

## 마무리

Git은 처음에 복잡해 보이지만 핵심 개념과 기본 명령어만 익히면 일상적인 개발 작업에 충분합니다. 매일 사용하면서 자연스럽게 익숙해지는 도구이니 너무 많은 것을 한번에 외우려 하지 마세요.

가장 중요한 것은 자주 커밋하고 의미 있는 메시지를 남기는 습관입니다. 작은 단위로 자주 커밋하면 문제 발생 시 원인을 찾기 쉽고, 되돌리기도 간편합니다.
