Git은 모든 개발자가 반드시 알아야 할 버전 관리 도구입니다. 하지만 많은 Git 튜토리얼이 `git add`, `git commit`, `git push` 수준에서 끝납니다. 실무에서는 이것만으로 부족합니다.

내가 직접 경험한 최악의 Git 사고를 먼저 말씀드리면, 신입 시절 merge 충돌을 해결하다가 동료가 3일간 작업한 코드를 날려버렸습니다. 충돌 마커(`<<<<<<<`, `=======`, `>>>>>>>`)를 보고 당황해서, 내 코드만 남기고 동료 코드를 전부 삭제한 채 커밋해버린 것입니다. 동료는 그날 밤 야근하면서 코드를 다시 작성했고, 저는 정말 미안해서 죽고 싶었습니다.

그 사건 이후 Git을 제대로 공부했습니다. 5년간 실무에서 Git을 사용하면서 배운 것들 -- 기본 개념부터, 위기 상황 대처법, 팀 협업 전략까지 -- 을 이 글에서 모두 공유합니다.

## Git이란 무엇인가?

Git은 리누스 토르발즈가 2005년에 만든 분산 버전 관리 시스템(DVCS)입니다. "분산"이라는 단어가 의미하는 것은 모든 개발자가 전체 저장소의 복사본을 가지고 있다는 뜻입니다. 중앙 서버가 다운되어도 로컬에서 계속 작업할 수 있고, 인터넷 연결 없이도 커밋을 만들 수 있습니다.

### Git과 GitHub의 차이

많은 초보자가 Git과 GitHub를 혼동합니다. 저도 처음에 그랬습니다.

- **Git**: 버전 관리 소프트웨어. 로컬 컴퓨터에서 실행됩니다. 리눅스 커널처럼 완전히 오프라인으로도 사용 가능합니다.
- **GitHub**: Git 저장소를 호스팅하는 클라우드 서비스. 코드 공유, 협업, 이슈 관리, CI/CD, 코드 리뷰 등의 추가 기능을 제공합니다.

비유하면, Git은 Word 프로그램이고 GitHub는 Google Drive입니다. Word 없이 Google Drive를 쓸 수 없듯이, Git 없이 GitHub를 쓸 수 없습니다.

Git 외에도 GitHub, GitLab, Bitbucket 등 여러 호스팅 서비스가 있지만, 2026년 현재 GitHub가 압도적으로 가장 많이 사용됩니다.

## Git 설치와 초기 설정

### 설치

Mac에서는 Xcode Command Line Tools를 설치하면 Git이 함께 설치됩니다.

```bash
xcode-select --install
```

Windows에서는 Git 공식 사이트(git-scm.com)에서 설치 프로그램을 다운로드합니다.

### 초기 설정: 반드시 해야 하는 것들

설치 후 사용자 정보를 설정합니다. 이 정보는 모든 커밋에 기록됩니다.

```bash
# 필수: 이름과 이메일 설정
git config --global user.name "홍길동"
git config --global user.email "gildong@example.com"

# 추천: 기본 브랜치 이름을 main으로 설정
git config --global init.defaultBranch main

# 추천: 기본 에디터를 VS Code로 설정
git config --global core.editor "code --wait"

# 추천: pull 전략을 rebase로 설정 (불필요한 merge 커밋 방지)
git config --global pull.rebase true

# macOS 추천: 한글 파일명이 깨지지 않도록
git config --global core.quotepath false
```

팀에서 도입했을 때 `pull.rebase true`가 가장 큰 변화를 가져왔습니다. 기본 설정(merge)을 사용하면 `git pull`할 때마다 불필요한 merge 커밋이 생겨서 히스토리가 지저분해지는데, rebase로 바꾸면 히스토리가 깔끔하게 유지됩니다.

## Git의 핵심 개념: 이것만 이해하면 90% 해결

### 세 가지 영역 (이것이 Git의 전부입니다)

Git에는 세 가지 주요 영역이 있습니다. 이 개념을 확실히 이해하는 것이 Git을 마스터하는 첫걸음입니다.

```
┌──────────────┐    git add     ┌──────────────┐   git commit   ┌──────────────┐
│   Working    │ ──────────────→│   Staging    │ ──────────────→│  Repository  │
│  Directory   │                │    Area      │                │   (.git)     │
│  (작업 공간)   │ ←─────────────│  (대기 공간)   │                │   (저장소)    │
│              │  git restore   │              │                │              │
└──────────────┘                └──────────────┘                └──────────────┘
```

1. **Working Directory (작업 디렉토리)**: 실제 파일을 편집하는 공간. 여기서 코드를 수정합니다.
2. **Staging Area (스테이징 영역)**: 다음 커밋에 포함시킬 변경사항을 "선별"하는 공간. `git add`로 파일을 이곳으로 옮깁니다.
3. **Repository (저장소)**: 커밋된 변경사항이 영구적으로 저장되는 공간. `git commit`으로 스테이징 영역의 내용을 저장합니다.

**왜 Staging Area가 필요한가?**

처음에는 "바로 커밋하면 되지 왜 Staging이 필요한 거야?"라고 생각했습니다. 하지만 실무에서 이유를 깨달았습니다.

예를 들어, 로그인 기능을 개발하면서 동시에 발견한 오타도 수정했다고 합시다. 이때 모든 변경사항을 한 커밋에 넣으면 나중에 "로그인 기능"만 되돌리고 싶을 때 오타 수정까지 함께 되돌아갑니다. Staging Area가 있으면 이렇게 할 수 있습니다.

```bash
# 로그인 관련 파일만 커밋
git add src/auth/login.ts src/auth/token.ts
git commit -m "feat: 이메일/비밀번호 로그인 기능 추가"

# 오타 수정은 별도 커밋
git add src/components/Header.tsx
git commit -m "fix: 헤더 오타 수정 (접속 → 접속하기)"
```

### 커밋이란?

커밋은 특정 시점의 프로젝트 상태를 스냅샷으로 저장하는 것입니다. 각 커밋에는 고유한 해시 ID(예: `a1b2c3d`), 작성자, 날짜, 메시지가 포함됩니다.

내가 직접 경험한 팁: 커밋을 "세이브 포인트"라고 생각하세요. 게임에서 보스전 전에 세이브하듯이, 중요한 변경을 하기 전에 커밋해두면 언제든 그 시점으로 돌아갈 수 있습니다.

## 필수 Git 명령어: 실무에서 매일 쓰는 것들

### 저장소 초기화와 클론

```bash
# 새 저장소 만들기 (빈 폴더에서)
git init

# 기존 저장소 복제하기 (가장 많이 쓰는 시작 방법)
git clone https://github.com/user/repo.git

# 특정 폴더 이름으로 클론
git clone https://github.com/user/repo.git my-project
```

### 변경사항 추적: 가장 자주 쓰는 명령어

```bash
# 현재 상태 확인 (가장 많이 치는 명령어!)
git status

# 변경 내용 확인 (무엇이 바뀌었는지)
git diff

# 스테이징된 변경 확인 (커밋될 내용 미리보기)
git diff --staged

# 특정 파일만 스테이징
git add src/auth/login.ts

# 여러 파일 한번에 스테이징
git add src/auth/login.ts src/auth/token.ts

# 모든 변경사항 스테이징 (주의: .gitignore에 없는 모든 파일 포함)
git add .

# 커밋 만들기
git commit -m "feat: 로그인 기능 추가"

# 커밋 메시지를 에디터에서 작성 (긴 메시지일 때)
git commit
```

**실무 팁: `git add .`보다 `git add <파일명>`을 쓰세요.** `git add .`를 습관적으로 쓰면 .env 파일이나 디버깅용 console.log가 포함된 파일이 실수로 커밋될 수 있습니다. 실제로 사용해보니, 파일 하나하나 확인하면서 add하는 것이 커밋 품질을 높이는 가장 좋은 습관이었습니다.

### 히스토리 확인

```bash
# 커밋 히스토리 보기
git log

# 한 줄로 보기 (가장 자주 쓰는 형태)
git log --oneline

# 그래프로 보기 (브랜치 흐름 확인)
git log --oneline --graph --all

# 특정 파일의 히스토리
git log --oneline src/auth/login.ts

# 최근 5개 커밋만
git log --oneline -5
```

### 원격 저장소 작업

```bash
# 원격 저장소 추가
git remote add origin https://github.com/user/repo.git

# 변경사항 올리기
git push origin main

# 변경사항 가져오기 + 병합 (pull = fetch + merge)
git pull origin main

# 가져오기만 (병합 안 함, 확인 후 수동 병합)
git fetch origin

# 현재 브랜치를 원격에 push하고 upstream 설정
git push -u origin feature/login
```

## 브랜치: Git의 가장 강력한 기능

브랜치는 독립적인 작업 공간을 만들어 기능 개발, 버그 수정 등을 메인 코드에 영향을 주지 않고 진행할 수 있게 합니다.

### 기본 브랜치 명령어

```bash
# 브랜치 목록 확인
git branch

# 원격 브랜치 포함 확인
git branch -a

# 새 브랜치 생성
git branch feature/login

# 브랜치 전환 (switch 권장, checkout도 가능)
git switch feature/login

# 생성과 전환을 한번에
git switch -c feature/login

# 브랜치 병합 (main에 feature/login을 병합)
git switch main
git merge feature/login

# 병합 완료 후 브랜치 삭제
git branch -d feature/login
```

### 실무에서 쓰는 브랜치 이름 규칙

팀에서 도입했을 때 일관된 브랜치 이름 규칙이 없으면 혼란이 생깁니다.

```
feature/기능명     → 새 기능 개발 (feature/social-login)
fix/버그명         → 버그 수정 (fix/login-redirect-error)
hotfix/긴급수정    → 프로덕션 긴급 수정 (hotfix/payment-crash)
refactor/대상      → 코드 리팩토링 (refactor/auth-module)
docs/문서명        → 문서 작업 (docs/api-guide)
chore/작업명       → 설정, 의존성 등 (chore/upgrade-node-20)
```

### Git Flow vs GitHub Flow: 어떤 전략을 쓸까?

**Git Flow** (복잡한 릴리즈 주기가 있는 프로젝트)
```
main (프로덕션)
 └─ develop (개발)
     ├─ feature/login
     ├─ feature/payment
     └─ release/1.2.0
```

**GitHub Flow** (대부분의 웹 서비스에서 추천)
```
main (프로덕션, 항상 배포 가능)
 ├─ feature/login
 ├─ fix/header-bug
 └─ feature/payment
```

실제로 사용해보니 GitHub Flow가 훨씬 간단하고 효과적이었습니다. Git Flow는 release 브랜치, hotfix 브랜치 등 관리해야 할 브랜치가 많아서 소규모 팀에서는 오버헤드가 큽니다. GitHub Flow는 `main`에서 브랜치 만들고 → 작업하고 → PR 올리고 → 리뷰 받고 → 머지하는 단순한 흐름입니다.

## 충돌 해결하기: 가장 무서운 순간

두 브랜치에서 같은 파일의 같은 부분을 수정하면 병합 시 충돌이 발생합니다.

```
<<<<<<< HEAD
const greeting = "안녕하세요";     ← 현재 브랜치 (내 코드)
=======
const greeting = "반갑습니다";     ← 병합하려는 브랜치 (동료 코드)
>>>>>>> feature/login
```

### 충돌 해결 프로세스

```bash
# 1. 충돌 발생 확인
git status
# Both modified: src/components/Header.tsx

# 2. 파일을 열어서 충돌 해결
# VS Code에서 "Accept Current", "Accept Incoming", "Accept Both" 버튼 활용

# 3. 해결한 파일 스테이징
git add src/components/Header.tsx

# 4. merge 커밋 완료
git commit
```

**충돌 방지를 위한 실무 팁:**

1. **작은 단위로 자주 머지**: 2주 동안 작업한 거대한 PR보다, 2-3일 단위의 작은 PR이 충돌 확률이 낮습니다.
2. **main을 자주 가져오기**: 작업 중에 `git pull origin main`을 자주 해서 최신 상태를 유지하세요.
3. **같은 파일 동시 작업 피하기**: 스프린트 플래닝 때 같은 파일을 수정할 가능성이 있으면 미리 조율하세요.

내가 직접 경험한 교훈: 충돌이 발생했을 때 가장 중요한 것은 **양쪽 코드를 모두 이해한 후에 해결**하는 것입니다. 이해하지 못한 채 한쪽을 선택하면 신입 시절의 저처럼 동료 코드를 날리는 사고가 발생합니다.

## 실수 되돌리기: 위기 상황별 대응

### "add를 잘못했어요" → 스테이징 취소

```bash
# 특정 파일의 스테이징 취소
git restore --staged src/auth/login.ts

# 모든 파일의 스테이징 취소
git restore --staged .
```

### "커밋 메시지를 잘못 썼어요" → 메시지 수정

```bash
# 마지막 커밋 메시지 수정 (아직 push 안 한 경우에만!)
git commit --amend -m "fix: 로그인 리다이렉트 버그 수정"
```

### "커밋에 파일을 빼먹었어요" → 마지막 커밋에 추가

```bash
# 빠진 파일을 스테이징
git add src/auth/forgot-password.ts

# 마지막 커밋에 추가 (메시지 유지)
git commit --amend --no-edit
```

### "커밋을 되돌리고 싶어요" → revert (안전) vs reset (위험)

```bash
# 안전한 방법: 되돌리는 새 커밋 생성 (push한 커밋에 사용)
git revert a1b2c3d

# 위험한 방법: 커밋 자체를 삭제 (아직 push 안 한 경우에만!)
git reset --soft HEAD~1   # 커밋만 취소, 변경사항은 스테이징에 유지
git reset --mixed HEAD~1  # 커밋 취소, 변경사항은 작업 디렉토리에 유지 (기본값)
git reset --hard HEAD~1   # 커밋 취소, 변경사항도 삭제! (복구 어려움)
```

**`reset --hard`는 정말 마지막 수단입니다.** 이미 push한 커밋에 `reset`을 쓰면 force push가 필요하고, 이는 동료의 작업을 망칠 수 있습니다. `revert`는 히스토리를 유지하면서 변경을 되돌리므로 팀 프로젝트에서 안전합니다.

### "작업 중인데 급하게 다른 브랜치로 가야 해요" → stash

```bash
# 현재 변경사항을 임시 저장
git stash

# 다른 브랜치에서 작업
git switch hotfix/urgent-bug
# ... 작업 후 커밋 ...

# 원래 브랜치로 돌아와서 임시 저장한 내용 복원
git switch feature/login
git stash pop

# stash 목록 확인
git stash list

# 특정 stash 적용
git stash apply stash@{1}
```

실제로 사용해보니 `stash`는 하루에 2-3번은 쓰는 필수 명령어입니다. 작업 중에 "이것 좀 급하게 봐주세요"라는 요청이 올 때, 커밋하기 애매한 상태의 코드를 임시로 저장하고 다른 작업을 처리한 후 다시 복원할 수 있습니다.

### "파일을 삭제/덮어썼는데 이전 버전으로 복구하고 싶어요"

```bash
# 특정 커밋의 특정 파일을 복원
git restore --source=a1b2c3d src/auth/login.ts

# 마지막 커밋 상태로 파일 복원 (작업 디렉토리의 변경 취소)
git restore src/auth/login.ts
```

### "모든 것이 꼬였어요" → reflog (최후의 보루)

```bash
# Git이 기록한 모든 HEAD 이동 히스토리 확인
git reflog

# 출력 예시:
# a1b2c3d HEAD@{0}: reset: moving to HEAD~3  ← reset으로 3개 커밋 삭제
# e4f5g6h HEAD@{1}: commit: feat: 결제 기능
# i7j8k9l HEAD@{2}: commit: feat: 장바구니 기능

# 특정 시점으로 복원
git reset --hard e4f5g6h
```

`reflog`는 `reset --hard`로 삭제한 커밋도 복구할 수 있습니다. Git은 30일간 모든 작업 히스토리를 보관합니다. 내가 직접 경험한 사례로, 실수로 `reset --hard`를 해서 3개의 커밋이 사라졌을 때 `reflog`로 복구했습니다. 이때의 안도감이란...

## .gitignore 설정: 프로젝트 시작의 첫 번째 파일

Git으로 추적하지 않을 파일을 `.gitignore`에 정의합니다.

```gitignore
# 의존성
node_modules/
vendor/

# 빌드 결과물
.next/
dist/
build/

# 환경 변수 (민감 정보!)
.env
.env.local
.env.production

# OS 파일
.DS_Store
Thumbs.db

# IDE 설정 (개인 설정은 공유하지 않음)
.idea/
.vscode/settings.json
*.swp

# 로그
*.log
npm-debug.log*

# 테스트 커버리지
coverage/
```

**중요: 프로젝트 시작 시 `.gitignore`를 먼저 설정하세요.** 한번 추적된 파일은 `.gitignore`에 추가해도 추적이 계속됩니다. 이미 추적 중인 파일을 무시하려면 이렇게 합니다.

```bash
# 추적 중인 파일을 Git에서 제거 (로컬 파일은 유지)
git rm --cached .env
git commit -m "chore: .env를 Git 추적에서 제거"
```

내가 직접 경험한 사고: .env 파일을 실수로 커밋한 적이 있습니다. GitHub에 push된 후 발견해서 급하게 삭제했지만, Git 히스토리에는 남아있었습니다. 결국 해당 API 키를 모두 교체해야 했습니다. **한 번이라도 커밋된 시크릿은 이미 노출된 것**으로 간주하고 반드시 교체하세요.

## 좋은 커밋 메시지 작성법

커밋 메시지는 미래의 자신과 동료를 위한 문서입니다. 6개월 후에 "이 커밋에서 왜 이렇게 변경했지?"를 알 수 있어야 합니다.

### Conventional Commits 규약

```
<type>(<scope>): <subject>

<body>

<footer>
```

| Type | 설명 | 예시 |
|------|------|------|
| feat | 새 기능 | feat: 소셜 로그인 기능 추가 |
| fix | 버그 수정 | fix: 로그인 시 리다이렉트 무한 루프 해결 |
| docs | 문서 변경 | docs: API 사용 가이드 업데이트 |
| style | 코드 스타일 (기능 변화 없음) | style: Prettier 적용 |
| refactor | 리팩토링 | refactor: 인증 모듈 구조 개선 |
| test | 테스트 추가/수정 | test: 로그인 유닛 테스트 추가 |
| chore | 빌드, 설정 변경 | chore: Node.js 20으로 업그레이드 |

### 좋은 커밋 메시지 vs 나쁜 커밋 메시지

```
# 좋은 예: 무엇을 왜 변경했는지 알 수 있음
feat: 사용자 로그인 기능 추가

- 이메일/비밀번호 기반 로그인 구현
- JWT 토큰 발급 및 검증 로직 추가
- 로그인 실패 시 에러 메시지 표시
- 5회 실패 시 30분 잠금 기능 추가

Closes #42

# 나쁜 예: 아무 정보도 없음
수정함
update
fix bug
asdf
ㅋㅋ
```

팀에서 도입했을 때 Conventional Commits와 함께 commitlint를 설정하면, 형식에 맞지 않는 커밋 메시지가 거부됩니다. 처음에는 "귀찮다"는 반응이 있었지만, 한 달 후 Git 히스토리를 볼 때 모든 팀원이 "이거 정말 좋다"고 했습니다.

## 실전 워크플로우: 팀에서 이렇게 일합니다

### GitHub Flow (우리 팀의 일상)

```bash
# 1. main에서 최신 코드 가져오기
git switch main
git pull origin main

# 2. 새 브랜치 생성
git switch -c feature/social-login

# 3. 코드 작성 → 커밋 (작은 단위로 자주!)
git add src/auth/kakao.ts
git commit -m "feat: 카카오 로그인 SDK 연동"

git add src/auth/naver.ts
git commit -m "feat: 네이버 로그인 SDK 연동"

git add src/auth/social-callback.ts
git commit -m "feat: 소셜 로그인 콜백 처리"

# 4. 원격에 push
git push -u origin feature/social-login

# 5. GitHub에서 Pull Request 생성
# → 코드 리뷰 진행
# → CI 테스트 통과 확인
# → 승인 후 main에 머지

# 6. 로컬 정리
git switch main
git pull origin main
git branch -d feature/social-login
```

### 코드 리뷰를 위한 커밋 전략

내가 직접 경험한 교훈: PR에 커밋이 1개만 있으면 리뷰어가 전체 변경사항을 한 번에 봐야 하므로 리뷰가 어렵습니다. 반대로 커밋이 너무 잘게 쪼개져 있으면("오타 수정", "빈 줄 제거" 같은 커밋) 의미 있는 변경을 파악하기 어렵습니다.

**추천 커밋 단위:**
- 하나의 논리적 변경 = 하나의 커밋
- 리뷰어가 커밋을 순서대로 읽었을 때 이야기가 되는 것처럼

```
커밋 1: feat: 소셜 로그인 DB 스키마 추가
커밋 2: feat: 카카오 로그인 API 연동
커밋 3: feat: 네이버 로그인 API 연동
커밋 4: feat: 소셜 로그인 UI 컴포넌트 구현
커밋 5: test: 소셜 로그인 유닛 테스트 추가
```

이렇게 하면 리뷰어가 "DB 스키마부터 보고 → API 연동 확인하고 → UI 확인하고 → 테스트 확인" 순서로 논리적으로 리뷰할 수 있습니다.

## 자주 하는 실수와 해결법

| 상황 | 해결 명령어 |
|------|-----------|
| 잘못된 브랜치에 커밋함 | `git stash` → `git switch 올바른-브랜치` → `git stash pop` → `git commit` |
| push한 커밋에 오류 발견 | `git revert 커밋해시` → `git push` |
| .env를 커밋해버림 | `git rm --cached .env` → API 키 교체 → `git commit` |
| merge 중 너무 복잡해짐 | `git merge --abort` (merge 취소 후 다시 시도) |
| rebase 중 꼬여버림 | `git rebase --abort` (rebase 취소 후 다시 시도) |

## 유용한 Git 별칭 (Alias) 설정

매일 쓰는 명령어를 짧게 줄이면 생산성이 올라갑니다.

```bash
git config --global alias.st "status"
git config --global alias.co "switch"
git config --global alias.br "branch"
git config --global alias.lg "log --oneline --graph --all"
git config --global alias.last "log -1 HEAD --format='%H %s'"
git config --global alias.unstage "restore --staged"
```

이제 `git st`, `git co feature/login`, `git lg` 등으로 짧게 사용할 수 있습니다.

## 마무리: Git은 매일 쓰면서 익히는 도구

Git은 처음에 복잡해 보이지만, 핵심 개념(세 가지 영역, 브랜치, 커밋)만 이해하면 일상적인 개발 작업에 충분합니다. 나머지는 필요할 때 찾아보면 됩니다.

5년간 실무에서 Git을 사용하면서 깨달은 가장 중요한 원칙 세 가지입니다.

1. **자주 커밋하세요**: 작은 단위로 자주 커밋하면 문제 발생 시 원인을 찾기 쉽고, 되돌리기도 간편합니다.
2. **의미 있는 메시지를 남기세요**: 6개월 후의 자신이 이해할 수 있는 커밋 메시지를 쓰세요.
3. **push 전에 한 번 더 확인하세요**: `git diff --staged`로 커밋될 내용을 확인하고, `git log --oneline -5`로 최근 히스토리를 확인하는 습관을 들이세요.

Git은 실수해도 대부분 복구할 수 있습니다. `reflog`가 30일간 모든 것을 기억하고 있으니까요. 두려워하지 말고 많이 써보세요. 실수하면서 배우는 것이 가장 빠른 길입니다.
