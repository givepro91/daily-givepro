"내 컴퓨터에서는 되는데?" 이 말은 개발자들 사이에서 유명한 밈이지만, 실제로 이 문제로 프로덕션 장애가 발생하면 웃을 수 없습니다.

내가 직접 경험한 최악의 케이스를 말씀드리면, Node.js 18에서 개발한 API를 배포했는데 서버에는 Node.js 16이 설치되어 있어서 optional chaining(?.) 문법에서 에러가 났습니다. 새벽 2시에 장애 알림을 받고, 서버에 접속해서 Node.js 버전을 확인하고, 급하게 nvm으로 버전을 변경하고, PM2를 재시작하고... 이 모든 과정이 2시간이 걸렸습니다.

Docker를 도입한 후에는 이런 일이 한 번도 발생하지 않았습니다. 개발 환경과 프로덕션 환경이 동일한 컨테이너에서 실행되기 때문입니다.

이 글에서는 Docker의 기본 개념부터, 제가 3개의 실제 프로젝트(Next.js 웹앱, Express API 서버, Python ML 서비스)를 Docker로 전환하면서 배운 실전 노하우를 공유합니다.

## Docker란 무엇인가? 왜 써야 하는가?

Docker는 애플리케이션을 컨테이너라는 격리된 환경에서 실행할 수 있게 해주는 도구입니다. 컨테이너 안에는 애플리케이션과 그에 필요한 모든 의존성(라이브러리, 런타임, 시스템 도구)이 함께 패키징됩니다.

### 컨테이너 vs 가상 머신: 실제 차이를 체감하다

이론적인 비교는 많은 글에서 다루니, 제가 실제로 체감한 차이를 말씀드리겠습니다.

이전 회사에서 VirtualBox로 개발 환경을 구성했을 때, VM 하나를 부팅하는 데 3분이 걸렸고, MacBook Pro 16인치에서 VM 2개를 동시에 돌리면 팬이 미친 듯이 돌아갔습니다. 메모리 사용량이 12GB를 넘어서 Chrome 탭을 몇 개 열면 시스템이 느려졌습니다.

Docker로 전환한 후, 같은 환경이 5초 만에 시작되었고, 메모리 사용량은 500MB 이하였습니다. 컨테이너 5개를 동시에 돌려도 시스템이 쾌적했습니다.

| 비교 항목 | 가상 머신 (VM) | Docker 컨테이너 |
|-----------|---------------|----------------|
| 크기 | 수 GB (OS 포함) | 수 MB ~ 수백 MB |
| 시작 시간 | 수 분 | 수 초 |
| 메모리 사용 | 각 VM마다 OS 메모리 | 커널 공유, 최소 메모리 |
| 격리 수준 | 완전 격리 (하드웨어 레벨) | 프로세스 레벨 격리 |
| 성능 오버헤드 | 10-20% | 거의 없음 |
| 동시 실행 | 2-3개가 한계 | 10개 이상 가능 |

### 핵심 용어 정리

- **이미지(Image)**: 컨테이너를 만들기 위한 설계도. 읽기 전용이며 레이어로 구성됩니다.
- **컨테이너(Container)**: 이미지를 기반으로 생성된 실행 가능한 인스턴스. 이미지 + 쓰기 가능 레이어.
- **Dockerfile**: 이미지를 빌드하기 위한 명세서. 레시피와 같습니다.
- **Docker Hub**: Docker 이미지를 공유하는 레지스트리. npm이 JavaScript 패키지 저장소인 것처럼 Docker Hub는 이미지 저장소입니다.
- **볼륨(Volume)**: 컨테이너의 데이터를 영구적으로 저장하는 방법. 컨테이너가 삭제되어도 데이터가 유지됩니다.
- **네트워크(Network)**: 컨테이너 간 통신을 관리합니다. 같은 네트워크에 있는 컨테이너끼리 서비스명으로 통신할 수 있습니다.

## Docker 설치와 초기 설정

### Mac에서 설치

Docker Desktop for Mac을 공식 사이트에서 다운로드합니다. Apple Silicon(M1/M2/M3/M4)과 Intel 모두 지원합니다.

```bash
# 설치 확인
docker --version
docker compose version
```

### 리소스 설정: 이것 때문에 삽질했습니다

Docker Desktop의 Settings > Resources에서 CPU, 메모리, 디스크 할당량을 조정할 수 있습니다.

내가 직접 경험한 실수: 기본 메모리 설정(2GB)으로 Next.js 빌드를 돌렸더니 `FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory` 에러가 발생했습니다.

**추천 리소스 설정:**

| 용도 | CPU | 메모리 | 디스크 |
|------|-----|--------|--------|
| 가벼운 개발 (API 1-2개) | 2코어 | 4GB | 20GB |
| 일반 개발 (앱 + DB) | 4코어 | 6GB | 40GB |
| 무거운 개발 (마이크로서비스) | 6코어 | 8GB+ | 60GB+ |

팀에서 도입했을 때는 README에 Docker 리소스 최소 요구사항을 명시해서, 새로운 팀원이 "빌드가 안 돼요"로 시간을 낭비하지 않도록 했습니다.

## Dockerfile 작성법: 초보에서 최적화까지

### 기본 Dockerfile

Node.js 애플리케이션용 Dockerfile을 단계별로 설명합니다.

```dockerfile
# 1. 베이스 이미지 선택
# alpine은 5MB 수준의 경량 리눅스. 일반 node 이미지(900MB)보다 훨씬 가벼움
FROM node:20-alpine

# 2. 작업 디렉토리 설정
WORKDIR /app

# 3. 의존성 파일을 먼저 복사 (레이어 캐싱 활용!)
COPY package.json package-lock.json ./

# 4. 의존성 설치
# npm ci는 package-lock.json 기반으로 정확한 버전을 설치 (재현 가능한 빌드)
RUN npm ci

# 5. 소스 코드 복사 (의존성 뒤에!)
COPY . .

# 6. 빌드
RUN npm run build

# 7. 포트 노출 (문서화 목적, 실제 포트 매핑은 docker run -p로)
EXPOSE 3000

# 8. 실행 명령
CMD ["npm", "start"]
```

### 레이어 캐싱: 빌드 시간을 70% 줄인 비법

Dockerfile의 각 명령어(FROM, COPY, RUN 등)는 레이어를 만듭니다. Docker는 변경되지 않은 레이어를 캐시에서 재사용합니다.

실제로 사용해보니 이 순서가 빌드 속도에 엄청난 차이를 만들었습니다.

```dockerfile
# 나쁜 예: 소스 코드를 먼저 복사
COPY . .                    # 소스 코드가 바뀔 때마다
RUN npm ci                  # 의존성을 다시 설치! (캐시 무효화)
RUN npm run build

# 좋은 예: 의존성 파일을 먼저 복사
COPY package.json package-lock.json ./   # 의존성 파일이 안 바뀌면
RUN npm ci                               # 이 레이어는 캐시 사용!
COPY . .                                 # 소스만 새로 복사
RUN npm run build                        # 빌드만 다시 실행
```

이 최적화 하나로 빌드 시간이 평균 3분 20초에서 55초로 줄었습니다. 의존성이 변경되지 않은 일반적인 코드 변경의 경우 `npm ci` 단계를 완전히 건너뛰기 때문입니다.

### .dockerignore: 반드시 설정하세요

빌드 컨텍스트에서 제외할 파일을 `.dockerignore`에 정의합니다. 이 파일이 없으면 `node_modules`(수백 MB)가 빌드 컨텍스트에 포함되어 빌드가 엄청나게 느려집니다.

```
node_modules
.next
.git
*.md
.env*
.DS_Store
coverage
.nyc_output
dist
```

내가 직접 경험한 실수: `.dockerignore`를 설정하지 않아서 로컬의 `node_modules`가 컨테이너에 복사되었고, Linux 컨테이너에서 Mac용 바이너리(예: esbuild)를 실행하려다 에러가 발생했습니다. 이 디버깅에 반나절이 걸렸습니다.

### 멀티 스테이지 빌드: 프로덕션 이미지 사이즈 80% 감소

멀티 스테이지 빌드는 빌드에 필요한 도구(TypeScript 컴파일러, 번들러 등)를 최종 이미지에서 제거하는 기법입니다.

```dockerfile
# === 스테이지 1: 빌드 ===
FROM node:20-alpine AS builder
WORKDIR /app

# 의존성 설치 (devDependencies 포함)
COPY package*.json ./
RUN npm ci

# 소스 복사 및 빌드
COPY . .
RUN npm run build

# === 스테이지 2: 프로덕션 ===
FROM node:20-alpine AS runner
WORKDIR /app

# 프로덕션 의존성만 설치
COPY --from=builder /app/package*.json ./
RUN npm ci --omit=dev

# 빌드 결과물만 복사
COPY --from=builder /app/.next ./.next
COPY --from=builder /app/public ./public

# 보안: 루트가 아닌 사용자로 실행
RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nextjs
USER nextjs

EXPOSE 3000
CMD ["npm", "start"]
```

이 방식으로 Next.js 프로젝트의 이미지 크기가 1.2GB에서 250MB로 줄었습니다.

| 방식 | 이미지 크기 | 포함된 내용 |
|------|-----------|------------|
| 단일 스테이지 | 1.2GB | 소스, node_modules(dev 포함), 빌드 도구, 빌드 결과 |
| 멀티 스테이지 | 250MB | 프로덕션 의존성, 빌드 결과만 |

## 필수 Docker 명령어: 실전에서 매일 쓰는 것들

### 이미지 관련

```bash
# 이미지 빌드 (현재 디렉토리의 Dockerfile 사용)
docker build -t my-app .

# 특정 Dockerfile로 빌드
docker build -f Dockerfile.dev -t my-app:dev .

# 빌드 캐시 없이 빌드 (문제 해결용)
docker build --no-cache -t my-app .

# 이미지 목록 확인
docker images

# 사용하지 않는 이미지 정리 (디스크 공간 확보)
docker image prune -a
```

### 컨테이너 관련

```bash
# 컨테이너 실행 (포그라운드)
docker run -p 3000:3000 my-app

# 백그라운드 실행 + 자동 삭제 + 이름 지정
docker run -d --rm --name my-app -p 3000:3000 my-app

# 실행 중인 컨테이너 확인
docker ps

# 모든 컨테이너 확인 (중지된 것 포함)
docker ps -a

# 컨테이너 중지
docker stop my-app

# 컨테이너 로그 확인 (실시간)
docker logs -f my-app

# 실행 중인 컨테이너에 쉘 접속 (디버깅용)
docker exec -it my-app sh
```

### 실전에서 자주 쓰는 옵션 정리

```bash
# 환경 변수 전달
docker run -e NODE_ENV=production -e DATABASE_URL=postgres://... my-app

# .env 파일로 환경 변수 전달
docker run --env-file .env.production my-app

# 볼륨 마운트 (로컬 파일을 컨테이너에 연결)
docker run -v $(pwd)/data:/app/data my-app

# 메모리 제한 (OOM 방지)
docker run -m 512m my-app

# 컨테이너 전체 정리 (주의! 모든 중지된 컨테이너 삭제)
docker system prune
```

## Docker Compose: 실전 멀티 컨테이너 환경

실제 애플리케이션은 여러 서비스로 구성됩니다. 제가 실제 프로젝트에서 사용한 Docker Compose 설정을 공유합니다.

### 개발 환경 구성 (Next.js + PostgreSQL + Redis)

```yaml
# docker-compose.yml
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile.dev
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgresql://devuser:devpass@db:5432/myapp
      - REDIS_URL=redis://cache:6379
      - NODE_ENV=development
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    volumes:
      - .:/app
      - /app/node_modules  # 호스트의 node_modules를 사용하지 않음
      - /app/.next          # 빌드 캐시도 컨테이너 것 사용
    command: npm run dev

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: devuser
      POSTGRES_PASSWORD: devpass
      POSTGRES_DB: myapp
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # 초기 데이터
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U devuser -d myapp"]
      interval: 5s
      timeout: 5s
      retries: 5

  cache:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data

  adminer:
    image: adminer
    ports:
      - "8080:8080"
    depends_on:
      - db

volumes:
  postgres_data:
  redis_data:
```

여기서 핵심 포인트들을 설명하겠습니다.

**`depends_on`과 `healthcheck`의 조합**: `depends_on`만으로는 DB가 "실행 중"인지만 확인합니다. PostgreSQL이 시작은 됐지만 아직 연결을 받을 준비가 안 된 상태에서 앱이 연결을 시도하면 에러가 납니다. `healthcheck`와 `condition: service_healthy`를 조합하면 DB가 실제로 쿼리를 받을 준비가 된 후에 앱을 시작합니다.

이 설정 하나로 "DB 연결이 안 돼요" 문제를 완전히 해결했습니다.

**볼륨의 `/app/node_modules` 트릭**: 호스트의 소스 코드를 컨테이너에 마운트할 때 `node_modules`도 함께 마운트되면 문제가 생깁니다. 호스트(Mac)의 바이너리와 컨테이너(Linux)의 바이너리가 다르기 때문입니다. `/app/node_modules`를 익명 볼륨으로 설정하면 컨테이너 안에서 설치한 node_modules가 유지됩니다.

### Compose 필수 명령어

```bash
# 모든 서비스 시작 (로그 실시간 출력)
docker compose up

# 백그라운드 실행
docker compose up -d

# 이미지 재빌드 후 시작 (Dockerfile 변경 시)
docker compose up --build

# 서비스 중지 + 네트워크 삭제
docker compose down

# 서비스 중지 + 볼륨까지 삭제 (DB 데이터 초기화!)
docker compose down -v

# 특정 서비스만 재시작
docker compose restart app

# 특정 서비스의 로그만 보기
docker compose logs -f app

# 서비스 상태 확인
docker compose ps
```

## 실전 트러블슈팅: 제가 겪은 문제들

### 문제 1: Apple Silicon에서 이미지가 안 돌아가요

M1/M2 Mac에서 `docker run`을 했는데 `exec format error`가 발생했습니다. x86_64(AMD64) 아키텍처용 이미지를 ARM64에서 실행하려 해서 생긴 문제였습니다.

```bash
# 해결: 플랫폼을 명시적으로 지정
docker run --platform linux/amd64 my-image

# 또는 Dockerfile에서 멀티 아키텍처 빌드
docker buildx build --platform linux/amd64,linux/arm64 -t my-app .
```

하지만 에뮬레이션으로 실행하면 성능이 2-3배 느려집니다. 가능하면 ARM64 네이티브 이미지를 사용하는 것이 좋습니다. `node:20-alpine`, `postgres:16-alpine` 등 주요 이미지들은 이미 멀티 아키텍처를 지원합니다.

### 문제 2: 컨테이너에서 Hot Reload가 안 돼요

Mac에서 Docker 볼륨 마운트를 사용할 때 파일 변경 감지(file watching)가 작동하지 않는 경우가 있습니다. Docker Desktop의 파일 시스템 구현 방식 때문입니다.

```javascript
// next.config.js에서 polling 활성화
module.exports = {
  webpack: (config) => {
    config.watchOptions = {
      poll: 1000,        // 1초마다 파일 변경 확인
      aggregateTimeout: 300,
    };
    return config;
  },
};
```

실제로 사용해보니 polling을 활성화하면 CPU 사용량이 올라가는 단점이 있습니다. Docker Desktop 4.x 이상에서는 VirtioFS를 사용하면 이 문제가 크게 개선됩니다. Settings > General > "Use VirtioFS for shared folders"를 체크하세요.

### 문제 3: 디스크 공간이 부족해요

Docker를 오래 사용하면 사용하지 않는 이미지, 볼륨, 캐시가 쌓입니다.

```bash
# 현재 Docker 디스크 사용량 확인
docker system df

# 출력 예시:
# TYPE            TOTAL     ACTIVE    SIZE      RECLAIMABLE
# Images          45        3         12.5GB    11.2GB (89%)
# Containers      12        2         1.2GB     980MB (81%)
# Build Cache     89        0         3.4GB     3.4GB

# 사용하지 않는 것들 정리
docker system prune -a --volumes
```

저는 매주 월요일 아침에 `docker system prune -a`를 실행하는 습관을 들였습니다. 한 번에 10-20GB가 정리되는 경우도 있었습니다.

### 문제 4: 컨테이너 간 통신이 안 돼요

Docker Compose에서 서비스 간 통신은 서비스 이름으로 합니다. `localhost`가 아닙니다.

```yaml
# 앱에서 DB에 접속할 때
# 잘못된 예: DATABASE_URL=postgresql://user:pass@localhost:5432/mydb
# 올바른 예: DATABASE_URL=postgresql://user:pass@db:5432/mydb
#                                                  ^^
#                                           서비스 이름 사용!
```

이 실수를 팀에서 3명이나 반복했습니다. 결국 README에 "Docker 환경에서는 localhost 대신 서비스명을 사용하세요"라는 경고를 빨간 글씨로 넣었습니다.

## 보안 베스트 프랙티스

### 루트가 아닌 사용자로 실행

기본적으로 컨테이너는 root 사용자로 실행됩니다. 컨테이너 탈출(container escape) 취약점이 있을 때 호스트의 root 권한을 얻을 수 있으므로 위험합니다.

```dockerfile
# 사용자 생성 및 전환
RUN addgroup --system --gid 1001 appgroup
RUN adduser --system --uid 1001 appuser
USER appuser
```

### 시크릿을 이미지에 포함하지 않기

```dockerfile
# 절대 이렇게 하면 안 됩니다!
ENV API_KEY=sk-1234567890
COPY .env /app/.env

# 런타임에 환경 변수로 전달
# docker run -e API_KEY=sk-1234567890 my-app
# 또는 docker-compose.yml의 env_file 사용
```

이미지 레이어에 시크릿이 포함되면 `docker history`로 누구나 볼 수 있습니다. 이미 커밋된 레이어에서 삭제해도 레이어 히스토리에 남습니다.

### 최소한의 베이스 이미지 사용

```dockerfile
# 이미지 크기 비교
# node:20         → 약 1.1GB (Debian 기반, 불필요한 패키지 다수)
# node:20-slim    → 약 200MB (최소한의 Debian)
# node:20-alpine  → 약 130MB (Alpine Linux 기반)

# 대부분의 Node.js 프로젝트에는 alpine이 충분합니다
FROM node:20-alpine
```

다만 alpine은 glibc 대신 musl을 사용하므로, C 확장을 사용하는 일부 npm 패키지(예: sharp, bcrypt)에서 문제가 발생할 수 있습니다. 이 경우 slim 이미지를 사용하세요.

## Docker를 활용한 개발 워크플로우

### 새 팀원 온보딩: 30분 → 5분

Docker 도입 전에는 새 팀원이 개발 환경을 구축하는 데 평균 30분(운이 나쁘면 반나절)이 걸렸습니다.

- Node.js 설치 (버전 맞추기)
- PostgreSQL 설치 및 DB 생성
- Redis 설치
- 환경 변수 파일 작성
- 각종 의존성 설치

Docker 도입 후에는 이렇게 변했습니다.

```bash
git clone https://github.com/company/project.git
cd project
cp .env.example .env
docker compose up
```

끝. 5분이면 동일한 개발 환경이 구축됩니다.

### 프로젝트별 독립된 환경

Docker의 가장 큰 장점 중 하나는 프로젝트별로 완전히 독립된 환경을 유지할 수 있다는 것입니다.

```
프로젝트 A: Node.js 18 + PostgreSQL 14 + Redis 6
프로젝트 B: Node.js 20 + PostgreSQL 16 + Redis 7 + MongoDB
프로젝트 C: Python 3.11 + MySQL 8
```

로컬에 이 모든 것을 설치하면 버전 충돌과 포트 충돌의 지옥에 빠지지만, Docker Compose로 각각 관리하면 완전히 분리됩니다. 프로젝트를 더 이상 사용하지 않을 때 `docker compose down -v` 한 줄이면 깨끗하게 정리됩니다.

### 프로덕션과 동일한 환경에서 테스트

CI/CD 파이프라인에서 Docker를 사용하면 "내 컴퓨터에서는 테스트가 통과했는데 CI에서는 실패"하는 상황을 방지할 수 있습니다.

```yaml
# GitHub Actions에서 Docker Compose로 통합 테스트
jobs:
  integration-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run integration tests
        run: |
          docker compose -f docker-compose.test.yml up --build --exit-code-from test
          docker compose -f docker-compose.test.yml down -v
```

## 마무리: Docker는 투자 대비 가치가 높은 기술입니다

Docker를 처음 배울 때는 명령어가 복잡해 보이고, Dockerfile 작성이 어렵게 느껴집니다. 하지만 기본 개념을 이해하고 자주 쓰는 명령어만 익히면 금방 편해집니다.

제가 3개 프로젝트에 Docker를 도입한 후 체감한 변화를 정리하면 이렇습니다.

| 항목 | Docker 도입 전 | Docker 도입 후 |
|------|---------------|---------------|
| 신규 팀원 온보딩 | 30분 - 반나절 | 5분 |
| "내 PC에서는 되는데" 이슈 | 월 2-3회 | 0회 |
| 프로덕션 환경 불일치 장애 | 분기 1-2회 | 0회 |
| 개발 환경 정리 | 수동, 복잡 | docker compose down -v |
| DB 버전 변경 테스트 | 반나절 | docker-compose.yml 수정 후 5분 |

Docker Compose를 사용하면 프로젝트의 전체 개발 환경을 코드로 정의할 수 있습니다. 새로운 팀원이 합류해도 `docker compose up` 한 줄이면 동일한 개발 환경이 구축됩니다. 이것이 Docker의 진정한 가치입니다.
