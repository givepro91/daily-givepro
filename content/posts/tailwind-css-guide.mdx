Tailwind CSS는 유틸리티 퍼스트 CSS 프레임워크입니다. 미리 정의된 클래스를 조합해서 스타일을 적용하는 방식으로, 별도의 CSS 파일을 작성할 필요 없이 HTML 안에서 직접 디자인할 수 있습니다. 처음에는 클래스가 길어 보여 거부감이 있을 수 있지만, 한번 익숙해지면 기존 CSS보다 훨씬 빠르게 UI를 만들 수 있습니다.

솔직히 고백하면, 저도 처음 Tailwind를 접했을 때 "이게 뭔 인라인 스타일의 변종이야"라고 생각했습니다. HTML에 `className="flex items-center justify-between gap-4 rounded-lg border p-6 shadow-md"`같은 긴 클래스를 쓰는 게 깔끔하다고? 말도 안 된다고 생각했죠.

그런데 팀 리더가 새 프로젝트에서 Tailwind를 도입하자고 했고, 2주간 억지로 사용하면서 "아, 이게 되네"라는 순간이 왔습니다. 지금은 기존 SCSS 방식으로 돌아가라고 하면 절대 못 돌아갑니다. 그 이유를 이 글에서 상세히 설명하겠습니다.

## 왜 Tailwind CSS인가? 기존 CSS의 현실적인 문제점

### 기존 CSS/SCSS 개발의 고질적 문제

10년 가까이 CSS를 작성하면서 겪은 문제들입니다. BEM, OOCSS, SMACSS 등 다양한 방법론을 시도해봤지만 근본적인 해결은 아니었습니다.

**문제 1: 클래스 이름 짓기의 고통**

CSS를 작성할 때 가장 시간이 많이 드는 건 클래스 이름을 짓는 것입니다. `.card-wrapper`, `.card-container`, `.card-inner-container`... 이런 이름 짓기에 실제 스타일링보다 더 많은 시간을 쓴 적도 있습니다.

BEM 방법론을 사용하면 `.card__title--highlighted` 같은 이름이 되는데, 이것도 컴포넌트가 복잡해지면 이름이 끝없이 길어집니다. 팀원마다 네이밍 스타일이 달라서 코드 리뷰에서 "이 클래스 이름을 바꿔주세요"라는 코멘트가 반복됩니다.

**문제 2: CSS 파일의 비대화와 죽은 코드**

프로젝트가 커질수록 CSS 파일이 무한히 증가합니다. 그리고 가장 무서운 건 **사용하지 않는 CSS를 삭제하기가 두렵다는 것**입니다. "이 클래스 지워도 괜찮을까? 어디선가 쓰고 있으면 어쩌지?"라는 두려움 때문에 죽은 CSS가 계속 쌓입니다.

저희 팀의 이전 프로젝트에서 CSS 파일 총 용량이 800KB를 넘었는데, 실제 사용되는 스타일은 40% 정도였습니다. 나머지 60%는 죽은 코드였지만 아무도 삭제할 엄두를 내지 못했습니다.

**문제 3: 컨텍스트 전환의 비용**

HTML을 수정하려면 CSS 파일을 함께 열어야 합니다. "이 버튼의 패딩을 바꾸려면 CSS 파일 어디에 있지?"라며 파일 사이를 오가는 것은 생산성을 상당히 떨어뜨립니다. 특히 컴포넌트 기반 개발(React, Vue)에서는 컴포넌트 파일과 스타일 파일을 번갈아 열어야 하는 것이 더 번거롭습니다.

**문제 4: 전역 스코프의 공포**

CSS는 기본적으로 전역이므로 한 곳에서 수정하면 다른 곳에 영향을 줄 수 있습니다. `.button`이라는 클래스를 수정했더니 전혀 다른 페이지의 버튼이 깨지는 일이 실제로 자주 발생합니다. CSS Modules나 styled-components로 해결할 수 있지만 추가적인 복잡성이 생깁니다.

### Tailwind는 이 문제들을 어떻게 해결하는가

```html
<!-- 전통 CSS -->
<div class="card">
  <h2 class="card-title">제목</h2>
  <p class="card-description">내용</p>
</div>
<!-- card.css 파일을 별도로 열어서 스타일 확인해야 함 -->

<!-- Tailwind CSS -->
<div class="rounded-lg border p-6 shadow-md">
  <h2 class="mb-2 text-xl font-bold">제목</h2>
  <p class="text-gray-600">내용</p>
</div>
<!-- 스타일이 HTML에 바로 보임. 별도 CSS 파일 불필요 -->
```

Tailwind를 사용하면:
- **클래스 이름을 지을 필요 없음**: 미리 정의된 유틸리티 클래스를 조합
- **CSS 파일이 비대해지지 않음**: 사용한 클래스만 빌드에 포함됨 (PurgeCSS 내장)
- **컨텍스트 전환 없음**: HTML만 보면 스타일이 바로 파악됨
- **전역 오염 없음**: 유틸리티 클래스는 범위가 명확함

## 핵심 유틸리티 클래스: 이것만 알면 80%는 커버

Tailwind의 유틸리티 클래스는 수천 개이지만, 실무에서 자주 사용하는 것은 약 50-60개 정도입니다. 처음부터 모든 클래스를 외울 필요는 없습니다. VS Code의 Tailwind CSS IntelliSense 확장 프로그램이 자동완성을 해주니까요.

### 레이아웃: Flexbox와 Grid

레이아웃은 Tailwind에서 가장 자주 사용하는 영역입니다.

```html
<!-- Flexbox: 가로 정렬 (가장 자주 사용) -->
<div class="flex items-center justify-between gap-4">
  <div>왼쪽</div>
  <div>오른쪽</div>
</div>

<!-- Flexbox: 세로 정렬 (카드 내부 등) -->
<div class="flex flex-col gap-2">
  <h3>제목</h3>
  <p>내용</p>
  <button>버튼</button>
</div>

<!-- Grid: 반응형 그리드 (목록 페이지) -->
<div class="grid grid-cols-1 gap-6 sm:grid-cols-2 lg:grid-cols-3">
  <div>Item 1</div>
  <div>Item 2</div>
  <div>Item 3</div>
</div>

<!-- 센터링: 가장 흔한 패턴 -->
<div class="flex items-center justify-center min-h-screen">
  <div>정중앙 배치</div>
</div>
```

제 경험상 `flex items-center justify-between`이 가장 자주 사용되는 조합입니다. 네비게이션 바, 카드 헤더, 리스트 아이템 등 거의 모든 곳에서 쓰입니다.

### 스페이싱: 일관된 여백 시스템

Tailwind의 스페이싱 스케일은 4px 단위입니다. `p-1`은 4px, `p-2`는 8px, `p-4`는 16px, `p-8`은 32px입니다. 이 규칙을 알면 직관적으로 사용할 수 있습니다.

| 클래스 | 크기 | 용도 |
|--------|------|------|
| `p-1`, `m-1` | 4px | 아이콘과 텍스트 사이 등 미세한 간격 |
| `p-2`, `m-2` | 8px | 버튼 내부 패딩, 리스트 아이템 간격 |
| `p-4`, `m-4` | 16px | 카드 내부 패딩, 섹션 간 기본 간격 |
| `p-6`, `m-6` | 24px | 넓은 카드 패딩, 섹션 간 여유 간격 |
| `p-8`, `m-8` | 32px | 페이지 상하 패딩, 큰 섹션 간격 |
| `mx-auto` | - | 가로 중앙 정렬 (블록 요소) |
| `gap-{n}` | 가변 | flex/grid 자식 요소 간 간격 |

**실전 팁**: 저희 팀에서는 "4의 배수 규칙"을 따릅니다. 여백은 항상 4, 8, 12, 16, 24, 32 중에서 선택합니다. 이렇게 하면 디자이너 없이도 일관된 간격의 UI를 만들 수 있습니다.

### 타이포그래피: 텍스트 스타일링

```html
<h1 class="text-3xl font-bold tracking-tight text-gray-900">큰 제목</h1>
<h2 class="text-2xl font-semibold text-gray-800">중간 제목</h2>
<p class="text-base leading-relaxed text-gray-600">
  본문 텍스트입니다. leading-relaxed로 줄 간격을 넓혀 가독성을 높입니다.
</p>
<span class="text-sm text-gray-500">보조 텍스트</span>
<span class="text-xs text-gray-400">매우 작은 텍스트</span>
```

`leading-relaxed`(줄 간격 1.625)와 `tracking-tight`(자간 -0.025em)는 한국어 텍스트에서 가독성을 크게 높여주는 클래스입니다. 특히 본문 텍스트에는 `leading-relaxed` 또는 `leading-7`을 반드시 적용하세요.

### 색상 시스템: 체계적인 팔레트

Tailwind는 체계적인 색상 팔레트를 제공합니다. 각 색상은 50(가장 밝음)부터 950(가장 어두움)까지의 단계를 가집니다.

```html
<!-- 배경색 -->
<div class="bg-blue-500 text-white">파란 배경</div>
<div class="bg-gray-50">매우 밝은 회색 배경 (페이지 배경용)</div>
<div class="bg-gray-100">밝은 회색 배경 (카드 배경용)</div>

<!-- 텍스트 색상 -->
<p class="text-gray-900">주요 텍스트 (거의 검정)</p>
<p class="text-gray-700">본문 텍스트</p>
<p class="text-gray-500">보조 텍스트</p>
<p class="text-gray-400">비활성 텍스트</p>

<!-- 테두리 -->
<div class="border border-gray-200">밝은 테두리</div>
<div class="border border-gray-300">좀 더 진한 테두리</div>
```

**색상 사용 패턴**: 저희 팀에서는 브랜드 색상 외에 gray 계열만 주로 사용합니다. 텍스트는 `gray-900`(제목), `gray-700`(본문), `gray-500`(보조), 배경은 `white`(카드), `gray-50`(페이지), 테두리는 `gray-200`이 기본입니다. 이 규칙만 지켜도 깔끔한 UI가 만들어집니다.

## 반응형 디자인: 모바일 퍼스트의 실전

Tailwind의 반응형 디자인은 모바일 퍼스트 접근법을 사용합니다. 접두어 없는 클래스가 모바일에 적용되고, 브레이크포인트 접두어로 큰 화면의 스타일을 추가합니다.

```html
<div class="grid grid-cols-1 gap-4 sm:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4">
  <!-- 모바일: 1열, sm(640px): 2열, lg(1024px): 3열, xl(1280px): 4열 -->
</div>

<!-- 모바일에서는 세로 정렬, 데스크톱에서는 가로 정렬 -->
<div class="flex flex-col gap-4 md:flex-row md:items-center md:justify-between">
  <h1 class="text-2xl font-bold">페이지 제목</h1>
  <button class="w-full md:w-auto">버튼</button>
</div>

<!-- 모바일에서만 보이는 요소 -->
<div class="block md:hidden">모바일 네비게이션</div>
<!-- 데스크톱에서만 보이는 요소 -->
<div class="hidden md:block">데스크톱 사이드바</div>
```

### 브레이크포인트 참고

| 접두어 | 최소 너비 | 대상 기기 |
|--------|---------|---------|
| (없음) | 0px | 모바일 |
| `sm:` | 640px | 큰 모바일 / 작은 태블릿 |
| `md:` | 768px | 태블릿 |
| `lg:` | 1024px | 작은 데스크톱 |
| `xl:` | 1280px | 데스크톱 |
| `2xl:` | 1536px | 큰 모니터 |

**제 경험에서 온 팁**: 실무에서는 `sm:`, `md:`, `lg:` 세 개만 주로 사용합니다. 대부분의 반응형 레이아웃은 이 세 브레이크포인트로 충분합니다. "모바일(기본) -> 태블릿(md:) -> 데스크톱(lg:)"의 3단계로 생각하면 간단합니다.

### 반응형 디자인 실전 패턴

```html
<!-- 반응형 네비게이션 바 -->
<nav class="flex items-center justify-between px-4 py-3 md:px-8">
  <a href="/" class="text-xl font-bold">Logo</a>

  <!-- 모바일: 햄버거 메뉴 -->
  <button class="md:hidden">
    <MenuIcon />
  </button>

  <!-- 데스크톱: 가로 메뉴 -->
  <ul class="hidden gap-6 md:flex">
    <li><a href="/blog">블로그</a></li>
    <li><a href="/about">소개</a></li>
  </ul>
</nav>

<!-- 반응형 히어로 섹션 -->
<section class="px-4 py-12 md:px-8 md:py-20 lg:py-32">
  <div class="mx-auto max-w-4xl text-center">
    <h1 class="text-3xl font-bold md:text-4xl lg:text-5xl">
      메인 타이틀
    </h1>
    <p class="mt-4 text-lg text-gray-600 md:text-xl">
      서브 타이틀 설명 텍스트
    </p>
  </div>
</section>
```

## 다크 모드: 의외로 간단한 구현

Tailwind는 `dark:` 접두어로 다크 모드 스타일을 쉽게 추가할 수 있습니다.

```html
<div class="bg-white dark:bg-gray-900">
  <h1 class="text-gray-900 dark:text-white">제목</h1>
  <p class="text-gray-600 dark:text-gray-400">내용</p>
  <div class="border border-gray-200 dark:border-gray-700">
    카드 컴포넌트
  </div>
</div>
```

### 다크 모드 색상 매핑 가이드

다크 모드에서 어떤 색상을 사용해야 할지 처음에는 헷갈립니다. 저희 팀에서 사용하는 매핑 규칙입니다.

| 용도 | 라이트 모드 | 다크 모드 |
|------|-----------|---------|
| 페이지 배경 | `bg-white` | `dark:bg-gray-900` |
| 카드 배경 | `bg-white` | `dark:bg-gray-800` |
| 미세한 배경 | `bg-gray-50` | `dark:bg-gray-800/50` |
| 제목 텍스트 | `text-gray-900` | `dark:text-white` |
| 본문 텍스트 | `text-gray-700` | `dark:text-gray-300` |
| 보조 텍스트 | `text-gray-500` | `dark:text-gray-400` |
| 테두리 | `border-gray-200` | `dark:border-gray-700` |
| 호버 배경 | `hover:bg-gray-100` | `dark:hover:bg-gray-700` |

이 매핑 규칙을 팀 내 디자인 가이드에 문서화해두면 모든 팀원이 일관된 다크 모드를 구현할 수 있습니다.

### 다크 모드 토글 구현

Tailwind v4에서는 CSS 파일에서 직접 다크 모드 전략을 설정할 수 있습니다. class 기반 전략을 사용하면 JavaScript로 다크 모드를 토글할 수 있습니다.

```typescript
// 다크 모드 토글 훅
function useDarkMode() {
  const [isDark, setIsDark] = useState(() => {
    if (typeof window === "undefined") return false;
    return document.documentElement.classList.contains("dark");
  });

  const toggle = useCallback(() => {
    const newValue = !isDark;
    setIsDark(newValue);
    document.documentElement.classList.toggle("dark", newValue);
    localStorage.setItem("theme", newValue ? "dark" : "light");
  }, [isDark]);

  return { isDark, toggle };
}
```

## 실전 컴포넌트 예시: 자주 사용하는 패턴 모음

### 카드 컴포넌트 - 블로그 글 카드

```html
<article class="group overflow-hidden rounded-xl border border-gray-200
                bg-white shadow-sm transition-all duration-300
                hover:shadow-lg hover:-translate-y-1
                dark:border-gray-800 dark:bg-gray-900">
  <div class="relative overflow-hidden">
    <img
      class="h-48 w-full object-cover transition-transform duration-300
             group-hover:scale-105"
      src="..."
      alt="..."
    />
    <span class="absolute left-3 top-3 rounded-full bg-blue-600 px-3 py-1
                 text-xs font-medium text-white">
      카테고리
    </span>
  </div>
  <div class="p-5">
    <div class="flex items-center gap-2 text-sm text-gray-500">
      <time>2026.03.10</time>
      <span>·</span>
      <span>5분 읽기</span>
    </div>
    <h3 class="mt-2 text-lg font-semibold text-gray-900 line-clamp-2
               dark:text-white">
      카드 제목이 여기에 들어갑니다
    </h3>
    <p class="mt-2 text-sm text-gray-600 line-clamp-3 dark:text-gray-400">
      카드 설명 텍스트가 여기에 들어갑니다. 최대 3줄까지 표시됩니다.
    </p>
  </div>
</article>
```

`group`과 `group-hover:`의 조합에 주목하세요. 부모 요소에 `group`을 붙이면, 부모를 호버했을 때 자식 요소의 스타일을 변경할 수 있습니다. 카드를 호버하면 이미지가 확대되는 효과를 JS 없이 구현할 수 있습니다.

`line-clamp-2`는 텍스트를 2줄로 제한하고 말줄임표(...)를 표시합니다. 카드 목록에서 일관된 높이를 유지할 때 매우 유용합니다.

</div>

**실전 팁**: `transition-all`보다는 `transition-colors`, `transition-transform` 등 구체적인 속성을 지정하는 것이 성능에 좋습니다. `transition-all`은 모든 CSS 속성의 변화를 감시하므로 불필요한 리소스를 소비합니다.

## Tailwind CSS v4의 변화: 직접 마이그레이션해보니

2026년 기준 Tailwind CSS v4가 출시되면서 중요한 변화가 있었습니다. 저희 팀에서 v3에서 v4로 마이그레이션한 경험을 공유합니다.

### 주요 변화점

| 항목 | v3 | v4 | 체감 차이 |
|------|----|----|---------|
| 빌드 엔진 | JavaScript | Rust (Oxide) | 빌드 속도 5배 향상 |
| 설정 파일 | `tailwind.config.js` | CSS `@theme` | 설정이 더 직관적 |
| 콘텐츠 감지 | `content` 배열 필수 | 자동 감지 | 설정 간소화 |
| CSS 중첩 | PostCSS 필요 | 네이티브 지원 | 의존성 감소 |
| 다크 모드 | `darkMode: 'class'` | CSS에서 직접 설정 | 설정 통합 |

### CSS-first 설정

v4에서 가장 큰 변화는 CSS-first 설정입니다. `tailwind.config.js` 대신 CSS 파일에서 직접 설정합니다.

```css
/* v3: tailwind.config.js */
/* module.exports = { theme: { extend: { colors: { brand: '#3b82f6' } } } } */

/* v4: CSS 파일에서 직접 */
@import "tailwindcss";

@theme {
  --color-brand: #3b82f6;
  --color-brand-light: #60a5fa;
  --color-brand-dark: #2563eb;
  --font-display: "Cal Sans", "Pretendard", sans-serif;
  --breakpoint-3xl: 1920px;
}
```

처음에는 "왜 JS 설정을 CSS로 옮기는 거지?"라고 의아했는데, 실제로 사용해보니 설정과 스타일이 같은 CSS 파일에 있으니 컨텍스트 전환이 줄어들었습니다. IDE의 CSS 변수 자동완성도 잘 동작합니다.

### 마이그레이션 시 주의사항

마이그레이션 자체는 공식 codemod(`npx @tailwindcss/upgrade`)로 대부분 자동 처리되지만, 몇 가지 수동 수정이 필요했습니다.

1. **`tailwind.config.js`의 커스텀 플러그인**: CSS `@plugin`으로 수동 변환 필요
2. **`@apply` 구문의 변경**: v4에서도 사용 가능하지만 일부 동작이 변경됨
3. **서드파티 라이브러리 호환성**: 일부 Tailwind 기반 컴포넌트 라이브러리가 v4를 지원하지 않을 수 있음

저희 팀의 경우 마이그레이션에 약 이틀이 걸렸습니다. 대부분은 codemod로 처리되었고, 수동 수정은 커스텀 플러그인 2개와 `@apply` 구문 일부였습니다.

## 생산성 도구와 팁: Tailwind를 200% 활용하기

### VS Code 확장 프로그램 (필수)

**Tailwind CSS IntelliSense**는 반드시 설치해야 합니다. 이 확장 없이 Tailwind를 사용하는 것은 IDE 없이 코딩하는 것과 같습니다.

제공하는 기능:
- 클래스명 자동완성 (타이핑하면 후보 표시)
- 클래스에 마우스를 올리면 실제 CSS 미리보기
- 잘못된 클래스명 경고
- 클래스 순서 정렬 (Prettier 플러그인과 연동)

### Prettier 플러그인

`prettier-plugin-tailwindcss`를 설치하면 클래스 순서를 자동으로 정렬해줍니다. 이것은 팀 내에서 일관된 클래스 순서를 유지하는 데 매우 중요합니다.

```bash
npm install -D prettier prettier-plugin-tailwindcss
```

정렬 규칙은 Tailwind 공식 권장 순서(레이아웃 -> 크기 -> 스페이싱 -> 타이포그래피 -> 색상 -> 기타)를 따릅니다.

### cn() 유틸리티 함수 - 조건부 클래스의 필수 도구

React에서 Tailwind를 사용할 때 조건부 클래스를 깔끔하게 관리할 수 있는 유틸리티입니다. 거의 모든 Tailwind + React 프로젝트에서 사용됩니다.

```typescript
import { clsx, type ClassValue } from "clsx";
import { twMerge } from "tailwind-merge";

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs));
}
```

`clsx`는 조건부 클래스를 합쳐주고, `twMerge`는 Tailwind 클래스의 충돌을 해결합니다. 예를 들어 `cn("px-4", "px-8")`은 `"px-8"`만 남깁니다.

```typescript
// 사용 예시
function Button({ variant, size, disabled, className, children }) {
  return (
    <button
      className={cn(
        // 기본 스타일
        "rounded-lg font-medium transition-colors",
        // variant별 스타일
        variant === "primary" && "bg-blue-600 text-white hover:bg-blue-700",
        variant === "secondary" && "bg-gray-100 text-gray-700 hover:bg-gray-200",
        variant === "outline" && "border border-gray-300 hover:bg-gray-50",
        // size별 스타일
        size === "sm" && "px-3 py-1.5 text-sm",
        size === "md" && "px-4 py-2.5 text-sm",
        size === "lg" && "px-6 py-3 text-base",
        // 상태
        disabled && "cursor-not-allowed opacity-50",
        // 외부에서 주입하는 className (오버라이드 가능)
        className
      )}
      disabled={disabled}
    >
      {children}
    </button>
  );
}
```

### @apply는 언제 사용해야 하는가

`@apply`를 사용하면 유틸리티 클래스를 묶어서 하나의 CSS 클래스로 만들 수 있습니다. 하지만 남용하면 Tailwind의 장점을 잃어버립니다.

**@apply를 사용해도 좋은 경우:**
- 서드파티 라이브러리의 스타일을 오버라이드할 때
- 매우 반복적인 기본 스타일 (예: 프로즈 타이포그래피)
- 마크다운 렌더링 등 클래스를 직접 추가할 수 없는 경우

**@apply를 사용하지 말아야 하는 경우:**
- 컴포넌트 스타일을 추출할 때 -> React 컴포넌트로 추출
- 조건부 스타일 -> `cn()` 유틸리티 사용
- 일반적인 스타일링 -> 유틸리티 클래스 직접 사용

## 팀에 Tailwind를 도입할 때의 팁: 저항을 줄이는 전략

팀에 Tailwind를 도입하려고 할 때 가장 큰 장벽은 기술적인 것이 아니라 **팀원의 거부감**입니다. 저도 처음에 거부감이 있었기 때문에 잘 알고 있습니다.

### 점진적 도입 전략

1. **작은 프로젝트에서 먼저 시도**: 새로운 사이드 프로젝트나 내부 도구에서 먼저 사용
2. **챔피언 만들기**: 팀에서 1-2명이 먼저 익숙해진 후 전파
3. **비교 데모 준비**: 같은 UI를 기존 방식과 Tailwind로 각각 만들어 비교
4. **강제하지 않기**: 기존 프로젝트는 그대로 두고, 새 프로젝트부터 적용

### 팀 내 규칙 정립

Tailwind를 팀에서 사용할 때는 다음 규칙들을 정하는 것이 좋습니다.

- **Prettier 플러그인 필수**: 클래스 순서 통일
- **cn() 유틸리티 사용**: 조건부 클래스의 일관된 처리
- **컴포넌트 추출 기준**: 클래스가 너무 길어지면 React 컴포넌트로 분리 (3줄 이상 시)
- **커스텀 색상 정의**: 프로젝트별 색상은 @theme에 정의하여 사용
- **@apply 사용 최소화**: 가능한 유틸리티 클래스 직접 사용

## 마무리: 거부감을 넘어선 뒤의 세계

Tailwind CSS는 "유틸리티 클래스가 너무 길다"는 초기 거부감만 넘기면 가장 생산적인 CSS 도구입니다. 저는 Tailwind를 사용한 이후 CSS 관련 작업 시간이 약 60% 줄었고, 디자인 시스템의 일관성도 크게 향상되었습니다.

디자인 시스템의 일관성을 유지하면서도 빠르게 UI를 구현할 수 있고, 사용하지 않는 CSS가 자동으로 제거되므로 번들 크기도 최적화됩니다. React, Vue, Svelte 등 어떤 프레임워크와도 잘 어울리며, Next.js와 조합하면 서버 컴포넌트에서도 문제없이 동작합니다.

처음에 거부감이 있으셨다면 "딱 2주만" 억지로 사용해보세요. 2주가 지나면 두 가지 중 하나입니다 - "이거 별로네"라고 느끼거나 "이걸 어떻게 안 쓰고 살았지"라고 느끼거나. 제 주변에서는 후자가 압도적으로 많았습니다.

다음 글에서는 Tailwind CSS와 Headless UI를 결합하여 접근성까지 갖춘 디자인 시스템을 구축하는 방법에 대해 다루겠습니다.
