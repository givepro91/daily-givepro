웹 프로젝트를 배포할 때 가장 먼저 하는 고민이 "어디에 호스팅하지?"이다. 2026년 현재, 프론트엔드 개발자에게 가장 인기 있는 세 플랫폼은 Vercel, Netlify, Cloudflare Pages다. 나는 지난 2년간 이 세 플랫폼 모두에서 실제 프로젝트를 운영해봤고, 각각의 장단점을 피부로 느꼈다.

이 글은 단순한 스펙 비교가 아니라, 실제로 프로젝트를 운영하면서 겪은 경험을 바탕으로 작성했다. 어떤 프로젝트에 어떤 플랫폼이 적합한지, 예상치 못한 비용 폭탄은 없는지, DX(Developer Experience)는 어떤지를 솔직하게 공유한다.

## 플랫폼 개요

### Vercel

Next.js를 만든 회사답게, Next.js와의 궁합이 가장 좋다. Zero-config 배포를 지향하며, Git에 푸시하면 자동으로 빌드와 배포가 이루어진다. Serverless Functions, Edge Functions, ISR 등 Next.js의 모든 기능을 네이티브로 지원한다.

나는 현재 이 블로그를 포함해 3개의 프로젝트를 Vercel에서 운영하고 있다.

### Netlify

Jamstack의 선구자 격인 플랫폼이다. Vercel보다 먼저 "Git 푸시 = 자동 배포" 워크플로우를 대중화했다. Netlify Functions, Edge Functions, Forms, Identity 등 다양한 기능을 제공한다.

이전 회사에서 마케팅 사이트와 문서 사이트를 Netlify에서 운영했다.

### Cloudflare Pages

Cloudflare의 CDN 인프라 위에 구축된 정적 사이트 호스팅 서비스다. Cloudflare Workers와의 통합이 강점이며, 무료 플랜의 제한이 가장 너그럽다. 비교적 후발 주자이지만 빠르게 성장하고 있다.

사이드 프로젝트 2개를 Cloudflare Pages에서 운영 중이다.

## 가격 비교: 무료 플랜의 진실

가격은 플랫폼 선택에서 가장 중요한 요소 중 하나다. 특히 사이드 프로젝트나 소규모 프로젝트에서는 무료 플랜의 제한이 결정적이다.

### 무료 플랜 비교

| 항목 | Vercel Free | Netlify Free | Cloudflare Pages Free |
|------|-------------|--------------|----------------------|
| 대역폭 | 100GB/월 | 100GB/월 | 무제한 |
| 빌드 시간 | 6,000분/월 | 300분/월 | 500빌드/월 |
| Serverless 실행 | 100GB-Hrs/월 | 125K 요청/월 | 100K 요청/일 |
| 동시 빌드 | 1 | 1 | 1 |
| 프로젝트 수 | 무제한 | 무제한(500사이트) | 무제한 |
| 팀원 수 | 1 | 1 | 무제한 |
| 커스텀 도메인 | 무제한 | 무제한 | 무제한 |
| HTTPS | 자동 | 자동 | 자동 |
| 프리뷰 배포 | 있음 | 있음 | 있음 |

여기서 가장 눈에 띄는 것은 Cloudflare Pages의 무제한 대역폭이다. Vercel과 Netlify는 100GB를 넘기면 추가 비용이 발생하는데, Cloudflare Pages는 대역폭 제한이 없다. 내가 직접 경험한 바로는, 사이드 프로젝트가 예상치 못하게 트래픽이 몰렸을 때(SNS에서 글이 바이럴될 때 등) 대역폭 걱정을 안 해도 되는 것은 큰 안심이다.

반면 빌드 시간은 Vercel이 압도적으로 넉넉하다. 6,000분은 일반적인 프로젝트에서 거의 무제한에 가깝다. Netlify의 300분은 빌드가 잦은 프로젝트에서는 금방 소진될 수 있다. 실제로 이전 프로젝트에서 Netlify 무료 플랜의 빌드 시간을 월 중순에 다 소진한 적이 있다.

### 유료 플랜 비교

| 항목 | Vercel Pro | Netlify Pro | Cloudflare Pages (Workers Paid) |
|------|-----------|-------------|-------------------------------|
| 월 가격 | $20/멤버 | $19/멤버 | $5 (고정) |
| 대역폭 | 1TB | 1TB | 무제한 |
| 빌드 시간 | 24,000분 | 25,000분 | 5,000빌드 |
| Serverless | 1,000GB-Hrs | 포함 | 10M 요청 |
| 동시 빌드 | 2 | 3 | 5 |
| 분석 | 포함 | 추가 비용 | 기본 포함 |

Cloudflare의 가격 경쟁력이 압도적이다. 멤버당 과금이 아닌 월 $5 고정이므로, 팀 규모가 클수록 Cloudflare가 유리하다. 5명 팀이라면 Vercel은 $100/월, Netlify는 $95/월인데 Cloudflare는 여전히 $5/월이다.

### 예상치 못한 비용 폭탄

내가 직접 경험한 비용 관련 사례를 공유한다. Vercel에서 Image Optimization 기능을 사용하고 있었는데, 트래픽이 증가하면서 이미지 최적화 요청 수가 무료 할당량을 초과했다. 월 $40 정도의 추가 비용이 발생했는데, 사전 알림 없이 청구서에 나타나서 당황했다. 이후로는 Vercel의 Usage 대시보드를 주기적으로 확인하고, 알림을 설정해두었다.

Netlify에서도 비슷한 경험이 있다. Netlify Functions의 호출 횟수가 무료 할당량을 넘겨서 추가 비용이 발생했는데, 원인은 봇 트래픽이었다. Rate Limiting을 설정하지 않은 API 엔드포인트에 봇이 대량으로 접근한 것이다. 이 경험 후로는 Serverless Function에 반드시 Rate Limiting을 적용하고 있다.

## 성능 비교: 실제 측정 결과

이론적인 벤치마크보다 실제 프로젝트에서의 성능이 중요하다. 동일한 Next.js 프로젝트를 세 플랫폼에 배포하고 측정한 결과를 공유한다.

### 정적 페이지 응답 시간

동일한 정적 페이지(약 50KB HTML + 200KB CSS/JS)에 대해 한국(서울)에서 측정한 TTFB(Time to First Byte)다.

| 플랫폼 | TTFB (서울) | TTFB (미국 동부) | TTFB (유럽) |
|--------|------------|----------------|------------|
| Vercel | 35ms | 25ms | 30ms |
| Netlify | 80ms | 30ms | 40ms |
| Cloudflare Pages | 20ms | 20ms | 25ms |

Cloudflare Pages가 가장 빠르다. Cloudflare의 글로벌 CDN 네트워크가 가장 촘촘하기 때문이다. 특히 아시아 지역에서의 성능이 두드러진다. 한국에 Cloudflare 엣지 서버가 있어서 TTFB가 20ms 수준으로 매우 빠르다.

Netlify는 서울에서의 TTFB가 상대적으로 높았다. 아시아 지역의 엣지 서버가 Vercel이나 Cloudflare에 비해 적은 것으로 보인다.

### Serverless Function 콜드 스타트

| 플랫폼 | 콜드 스타트 | 웜 스타트 |
|--------|-----------|----------|
| Vercel (Serverless) | 300~500ms | 50~100ms |
| Vercel (Edge) | 50ms 이하 | 10~30ms |
| Netlify Functions | 500~800ms | 80~150ms |
| Cloudflare Workers | 50ms 이하 | 5~20ms |

Cloudflare Workers의 콜드 스타트가 압도적으로 빠르다. V8 Isolate 기반이라 전통적인 컨테이너 기반 Serverless보다 시작 시간이 훨씬 짧다. Vercel의 Edge Functions도 같은 V8 Isolate 기반이라 비슷하게 빠르다.

실제로 사용해보니, 콜드 스타트는 사용자 경험에 직접적인 영향을 미치는 요소다. API 응답이 간헐적으로 느려지면 사용자가 불편을 느끼는데, 콜드 스타트가 주요 원인인 경우가 많다.

## 개발자 경험 (DX) 비교

### 배포 속도

| 플랫폼 | 평균 빌드+배포 시간 | 특이사항 |
|--------|-------------------|---------|
| Vercel | 45초~1분 | 캐시 활용 우수 |
| Netlify | 1분~2분 | 플러그인 활용 시 증가 |
| Cloudflare Pages | 30초~1분 | 정적 사이트 최적화 |

Vercel의 빌드 캐시가 인상적이다. 의존성이 변경되지 않으면 node_modules를 캐시해서 설치 시간을 건너뛴다. 첫 배포는 2분이 걸려도, 이후 배포는 45초 만에 완료되는 경우가 많다.

### 프리뷰 배포

세 플랫폼 모두 PR마다 자동으로 프리뷰 환경을 생성한다. 하지만 세부적인 차이가 있다.

**Vercel:** PR에 자동으로 코멘트를 달아서 프리뷰 URL을 알려준다. 프리뷰 환경에서도 Serverless Functions가 완전히 동작한다. GitHub, GitLab, Bitbucket 모두 지원한다.

**Netlify:** Deploy Preview라는 이름으로 비슷한 기능을 제공한다. Netlify Functions도 프리뷰에서 동작한다. 추가로 Deploy Preview에서 직접 코멘트를 남길 수 있는 Collaborative Deploy Previews 기능이 있는데, 디자이너나 PM에게 피드백을 받을 때 유용하다.

**Cloudflare Pages:** 프리뷰 배포를 지원하지만, Workers 통합이 프리뷰에서 완전히 동작하지 않는 경우가 있다. 이 부분은 개선되고 있지만, 현재로서는 Vercel이나 Netlify보다 약간 부족하다.

### Next.js 지원 수준

Next.js를 사용한다면 플랫폼 선택이 상당히 제한된다. 세 플랫폼의 Next.js 지원 수준을 비교하면 다음과 같다.

| 기능 | Vercel | Netlify | Cloudflare Pages |
|------|--------|---------|-----------------|
| App Router | 완벽 | 좋음 | 제한적 |
| Server Components | 완벽 | 좋음 | 제한적 |
| ISR | 완벽 | 어댑터 필요 | 제한적 |
| Image Optimization | 네이티브 | 플러그인 | 수동 설정 |
| Middleware | 완벽 | 좋음 | 제한적 |
| Server Actions | 완벽 | 좋음 | 제한적 |

Vercel이 Next.js를 만든 회사이기 때문에, Next.js 최신 기능은 항상 Vercel에서 먼저 완벽하게 지원된다. 이것은 Next.js를 사용하는 프로젝트에서 Vercel을 선택하는 가장 큰 이유다.

Netlify도 Next.js 지원이 많이 개선되었다. `@netlify/plugin-nextjs`를 통해 대부분의 Next.js 기능을 지원하지만, 최신 기능은 Vercel보다 지원이 늦는 경향이 있다.

Cloudflare Pages의 Next.js 지원은 `@cloudflare/next-on-pages`를 통해 이루어지는데, Node.js 런타임이 아닌 Workers 런타임 위에서 실행되므로 호환성 문제가 있을 수 있다. Node.js 전용 API를 사용하는 코드는 수정이 필요할 수 있다.

## 각 플랫폼의 고유 강점

### Vercel만의 강점

**Speed Insights:** 실제 사용자의 Core Web Vitals를 자동으로 수집하고 대시보드로 보여준다. 어떤 페이지가 느린지, 어떤 지역에서 문제가 있는지를 한눈에 파악할 수 있다.

**Cron Jobs:** Serverless Function을 정기적으로 실행할 수 있다. 별도의 스케줄러 서비스 없이 Vercel 내에서 처리 가능하다.

**Vercel AI SDK:** AI 애플리케이션을 만들 때 스트리밍 응답 처리, 다양한 LLM 프로바이더 통합 등을 쉽게 할 수 있다.

### Netlify만의 강점

**Netlify Forms:** 서버 없이 폼 제출을 처리할 수 있다. 마케팅 사이트의 문의 폼이나 뉴스레터 구독 폼을 별도 백엔드 없이 구현 가능하다. 이전 회사에서 이 기능 덕분에 백엔드 개발 없이 랜딩 페이지의 리드 수집을 구현했다.

**Split Testing:** A/B 테스트를 CDN 레벨에서 할 수 있다. 별도의 A/B 테스트 서비스 없이, 다른 브랜치의 코드를 특정 비율의 사용자에게 보여줄 수 있다.

**Netlify Identity:** 간단한 인증 기능을 내장으로 제공한다. 소규모 프로젝트에서 Auth0 같은 별도 서비스 없이 인증을 구현할 수 있다.

### Cloudflare Pages만의 강점

**Cloudflare 생태계 통합:** R2(오브젝트 스토리지), D1(SQLite 데이터베이스), KV(키-값 저장소), Queues(메시지 큐) 등 Cloudflare의 다양한 서비스와 네이티브로 통합된다. 특히 R2는 AWS S3 호환이면서 Egress 비용이 무료라서, 파일 저장 비용이 크게 절약된다.

**Web Analytics:** 개인정보 친화적인 웹 분석 도구를 무료로 제공한다. Google Analytics 대안으로 사용 가능하며, JavaScript 추적 없이 서버 사이드에서 데이터를 수집한다.

**WAF와 DDoS 보호:** Cloudflare의 보안 인프라가 기본으로 적용된다. 별도 설정 없이 DDoS 공격으로부터 보호되며, WAF 룰도 무료 플랜에서 일부 사용 가능하다.

## 프로젝트 유형별 추천

내가 2년간 세 플랫폼을 사용하면서 내린 결론이다.

| 프로젝트 유형 | 추천 플랫폼 | 이유 |
|-------------|-----------|------|
| Next.js 풀스택 앱 | Vercel | 최고의 Next.js 지원 |
| 정적 블로그/문서 | Cloudflare Pages | 무제한 대역폭, 최저 비용 |
| 마케팅 사이트 | Netlify | Forms, Split Testing |
| 사이드 프로젝트 | Cloudflare Pages | 가장 너그러운 무료 플랜 |
| 상용 SaaS | Vercel | 안정성, 분석 도구 |
| Astro/Hugo 정적 사이트 | Cloudflare Pages | 속도, 비용 |
| 전체 인프라 Cloudflare | Cloudflare Pages | 생태계 시너지 |

### 마이그레이션 난이도

플랫폼 간 마이그레이션은 어렵지 않다. 세 플랫폼 모두 Git 기반 배포를 사용하므로, 기본적으로 배포 설정만 변경하면 된다. 다만 플랫폼 고유 기능(Vercel의 ISR 캐시 태그, Netlify의 Forms, Cloudflare Workers)을 많이 사용할수록 마이그레이션 비용이 증가한다.

내가 직접 경험한 Netlify에서 Vercel로의 마이그레이션은 약 반나절이 걸렸다. 대부분의 시간은 환경 변수 설정과 DNS 변경에 소요되었고, 코드 변경은 Netlify 플러그인 설정을 제거하는 정도였다.

## 배포 설정 실전 비교

각 플랫폼에서 실제로 프로젝트를 배포할 때의 과정을 비교해보자.

### Vercel 배포 과정

Vercel에서의 배포는 정말 간단하다. GitHub 저장소를 연결하고, 프레임워크를 자동 감지하면 끝이다. 내가 실제로 경험한 과정은 이렇다.

1. Vercel 대시보드에서 "New Project" 클릭
2. GitHub 저장소 선택
3. 프레임워크 자동 감지 (Next.js, Astro 등)
4. 환경 변수 설정
5. "Deploy" 클릭

이후 main 브랜치에 푸시할 때마다 자동으로 빌드와 배포가 이루어진다. `vercel.json` 파일로 세부 설정을 할 수 있지만, 대부분의 경우 기본 설정으로 충분하다. 특히 Next.js 프로젝트는 설정 파일 없이도 모든 기능이 동작한다는 점이 인상적이다.

커스텀 도메인 연결도 간편하다. Vercel 대시보드에서 도메인을 추가하면, 네임서버 설정이나 CNAME 레코드 추가 방법을 안내해준다. Let's Encrypt 기반의 SSL 인증서가 자동으로 발급되고 갱신된다.

### Netlify 배포 과정

Netlify도 비슷하게 간단하지만, 빌드 설정에서 약간의 차이가 있다. `netlify.toml` 파일로 빌드 명령어, 출력 디렉토리, 리다이렉트 규칙 등을 설정한다.

```toml
[build]
  command = "npm run build"
  publish = ".next"

[[plugins]]
  package = "@netlify/plugin-nextjs"

[[redirects]]
  from = "/api/*"
  to = "/.netlify/functions/:splat"
  status = 200
```

Netlify의 장점은 빌드 플러그인 생태계다. 이미지 최적화, 캐시 관리, 접근성 검사 등을 빌드 파이프라인에 쉽게 추가할 수 있다. 다만 플러그인이 많아지면 빌드 시간이 증가하므로 주의가 필요하다.

### Cloudflare Pages 배포 과정

Cloudflare Pages도 Git 연결 기반의 자동 배포를 지원한다. 다만 Serverless Function을 사용하려면 `functions` 디렉토리에 Workers 코드를 작성해야 하고, Next.js를 사용하려면 어댑터 설정이 필요하다.

```bash
# wrangler.toml (Cloudflare 설정)
name = "my-project"
compatibility_date = "2026-03-01"

[site]
bucket = "./out"
```

팀에서 도입했을 때, Cloudflare Pages의 초기 설정이 Vercel이나 Netlify보다 약간 복잡했다. 특히 Next.js 프로젝트에서 `@cloudflare/next-on-pages`를 설정하는 과정에서 시행착오가 있었다. 하지만 한번 설정하면 이후 배포 워크플로우는 다른 플랫폼과 동일하게 간편하다.

## 장애 대응과 안정성

### 가동 시간 (Uptime)

2년간 세 플랫폼을 사용하면서 경험한 장애를 정리한다.

**Vercel:** 2년간 체감 장애가 2번 있었다. 한 번은 빌드 서비스가 약 30분간 지연된 것이고, 한 번은 Edge Function의 응답이 느려진 것이었다. 서비스 자체가 완전히 다운된 적은 없었다. status.vercel.com에서 실시간 상태를 확인할 수 있고, 장애 발생 시 이메일 알림이 빠르게 온다.

**Netlify:** 빌드 서비스 지연이 가끔 있었다. 특히 무료 플랜에서 빌드 큐가 밀리는 경우가 있었는데, 유료 플랜으로 전환하니 해결됐다. CDN 장애는 경험하지 못했다.

**Cloudflare Pages:** Cloudflare의 글로벌 네트워크가 워낙 견고해서, Pages 서비스의 가동 시간은 매우 안정적이었다. 다만 빌드 서비스에서 간헐적인 타임아웃이 있었는데, 재시도하면 해결되는 수준이었다.

### 지원 품질

무료 플랜에서의 기술 지원은 세 플랫폼 모두 커뮤니티 포럼 수준이다. 유료 플랜에서의 차이가 있는데, Vercel Pro는 이메일 지원과 우선 대기열을 제공하고, Netlify Pro는 이메일 지원을 제공한다. Cloudflare는 Workers Paid 플랜에서도 커뮤니티 지원이 기본이지만, Enterprise 플랜에서 전담 지원을 받을 수 있다.

실제로 사용해보니, 세 플랫폼 모두 공식 문서의 품질이 높아서 대부분의 문제는 문서에서 해결할 수 있었다. 커뮤니티 포럼(Discord, GitHub Issues)도 활발해서, 특이한 문제도 대부분 검색하면 답을 찾을 수 있었다.

## 환경 변수와 시크릿 관리

세 플랫폼 모두 환경 변수 관리를 지원하지만, 세부적인 차이가 있다.

**Vercel:** 환경 변수를 Production, Preview, Development 환경별로 분리해서 관리할 수 있다. 이것이 매우 유용한데, 예를 들어 개발 환경에서는 테스트 DB를 사용하고, 프로덕션에서는 실제 DB를 사용하는 설정이 간편하다. `vercel env pull` 명령어로 로컬에 환경 변수를 내려받을 수도 있다.

**Netlify:** 비슷하게 환경별 분리를 지원하고, Build 시점과 Runtime 시점의 환경 변수를 구분할 수 있다.

**Cloudflare Pages:** 환경 변수 외에 Workers KV, Secrets 등 다양한 저장 방식을 제공한다. `wrangler secret put` 명령어로 시크릿을 안전하게 관리할 수 있다.

## 결론

세 플랫폼 모두 훌륭하다. 3년 전만 해도 "Vercel이 압도적"이라고 말했겠지만, 2026년 현재는 각 플랫폼이 자신만의 강점을 확실히 가지고 있다.

**Next.js 프로젝트라면 Vercel.** 이건 거의 의심의 여지가 없다. Next.js의 모든 기능을 제대로 활용하려면 Vercel이 가장 안전한 선택이다.

**비용이 중요하다면 Cloudflare Pages.** 무제한 대역폭, 저렴한 유료 플랜, Cloudflare 생태계의 무료/저렴한 서비스들이 비용 효율을 극대화한다.

**비개발자 협업이 중요하다면 Netlify.** Collaborative Deploy Previews, Forms, Split Testing 등 개발자가 아닌 팀원과의 협업에 유용한 기능이 많다.

어떤 플랫폼을 선택하든, 중요한 것은 프로젝트의 요구사항에 맞게 선택하는 것이다.

한 가지 조언을 덧붙이자면, 처음에 너무 고민하지 않아도 된다. 세 플랫폼 모두 마이그레이션이 어렵지 않으므로, 일단 시작하고 필요하면 나중에 바꾸면 된다. 나도 Netlify에서 시작해서 Vercel로, 사이드 프로젝트는 Cloudflare Pages로 옮겼다. 중요한 것은 플랫폼 선택에 시간을 쓰는 것이 아니라, 프로젝트를 빨리 배포하고 사용자 피드백을 받는 것이다. 완벽한 플랫폼을 찾느라 배포를 미루는 것이 가장 나쁜 선택이다.

그리고 무료 플랜으로 시작하되, 트래픽이 늘어나면 주저하지 말고 유료 플랜으로 전환하자. 서비스가 느려지거나 장애가 발생하는 비용이 월 $5~$20보다 훨씬 크다. 특히 비즈니스가 걸린 프로젝트라면 안정성에 투자하는 것이 현명한 결정이다.

이 글이 그 선택에 도움이 되었으면 좋겠다. 궁금한 점이 있다면 댓글로 남겨주시기 바란다.
