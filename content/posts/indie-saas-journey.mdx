회사를 다니면서 항상 생각했습니다. "내 제품을 만들어보고 싶다." 많은 개발자가 이런 생각을 하지만, 실제로 제품을 출시하고 유료 고객을 확보하는 데까지 이르는 사람은 극소수입니다. 저는 2025년 4월부터 약 10개월간 1인 SaaS 제품을 만들었고, 2026년 1월에 첫 유료 고객을 확보했습니다.

이 글은 그 과정을 월별로 정리한 기록입니다. 성공담이 아니라 솔직한 도전기입니다. 실패한 것, 후회하는 것, 예상과 달랐던 것들을 모두 포함합니다.

## 만든 제품: FeedPulse

개발자 블로그, 뉴스레터, RSS 피드를 한곳에서 모아 보고, AI가 요약해 주는 콘텐츠 큐레이션 서비스입니다. 핵심 기능은 세 가지였습니다.

1. RSS/Atom 피드 구독 및 통합 뷰
2. AI 기반 기사 요약 및 주간 다이제스트 이메일
3. 팀용 공유 보드 (읽은 글 공유, 메모)

## 월별 타임라인

### 2025년 4월: 아이디어 탐색

처음에는 아이디어가 3개 있었습니다.

- **개발자용 피드 리더** — 매일 아침 여러 사이트를 돌아다니는 게 비효율적이라고 느낌
- **코드 리뷰 자동화 도구** — 이미 경쟁자가 너무 많음 (CodeRabbit, Sourcery 등)
- **개발 일지 앱** — 시장이 너무 작고 유료화가 어려움

아이디어 검증을 위해 한 것들:

1. **X(트위터)에서 설문조사**: "매일 아침 개발 관련 뉴스를 어떻게 확인하시나요?" → 127명 응답, 68%가 "여러 사이트를 직접 방문"
2. **Reddit r/webdev, r/programming에서 검색**: "RSS reader for developers" 관련 불만 글이 다수 존재
3. **경쟁 제품 분석**: Feedly(월 $8), Inoreader(월 $6)가 있지만, AI 요약 기능이 없거나 부족

결론: 기존 피드 리더에 AI 요약 기능을 더하면 차별화가 가능하다고 판단했습니다.

### 2025년 5월: 기술 스택 결정

가장 오래 고민한 부분입니다. 1인 개발이기 때문에 **생산성**이 최우선이었습니다.

| 영역 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | Next.js 15 (App Router) | 풀스택 가능, Vercel 배포 편리 |
| 스타일링 | Tailwind CSS v4 | 빠른 UI 구현 |
| DB | Supabase (PostgreSQL) | 무료 티어 넉넉, 인증 내장 |
| AI | OpenAI GPT-4o-mini API | 요약용으로 비용 대비 성능 최적 |
| 결제 | Stripe | 글로벌 결제, 구독 관리 |
| 이메일 | Resend | 개발자 친화적 API, 무료 3000건/월 |
| 호스팅 | Vercel (Pro) | Next.js 최적화, Edge Functions |
| 모니터링 | Sentry + Vercel Analytics | 에러 추적 + 성능 모니터링 |

**후회하는 결정**: Supabase를 선택한 것은 좋았지만, Supabase의 Row Level Security(RLS)를 제대로 이해하지 않고 시작해서 나중에 인증 관련 버그를 3번이나 겪었습니다. RLS 정책을 처음부터 꼼꼼하게 설계했어야 했습니다.

### 2025년 6월~7월: MVP 개발 (8주)

MVP에 포함한 기능을 최소한으로 줄이는 게 가장 어려웠습니다. 처음 기획서에는 20개 기능이 있었는데, 최종 MVP는 5개로 줄였습니다.

**MVP 핵심 기능:**
1. 이메일/구글 소셜 로그인
2. RSS 피드 URL 추가 및 구독
3. 피드 목록 뷰 (최신순 정렬)
4. 개별 글 AI 요약 (버튼 클릭)
5. 주간 다이제스트 이메일

**버린 기능들:**
- 팀 공유 보드 → v2로 미룸
- 모바일 앱 → 반응형 웹으로 대체
- 북마크/하이라이트 → v2로 미룸
- 키워드 알림 → v2로 미룸
- 다크 모드 → 출시 2주 뒤 추가

코드 구조는 다음과 같이 잡았습니다.

```
src/
├── app/
│   ├── (auth)/
│   │   ├── login/page.tsx
│   │   └── signup/page.tsx
│   ├── (dashboard)/
│   │   ├── feeds/page.tsx
│   │   ├── feeds/[id]/page.tsx
│   │   └── settings/page.tsx
│   └── api/
│       ├── feeds/route.ts
│       ├── summarize/route.ts
│       └── webhooks/stripe/route.ts
├── components/
├── lib/
│   ├── supabase/
│   ├── stripe/
│   └── openai/
└── types/
```

**개발 일지에서 발췌한 실제 기록:**

> 6월 15일: RSS 파싱이 생각보다 지옥이다. XML 형식이 사이트마다 제각각이고, 인코딩 문제도 있다. `rss-parser` 라이브러리만으로는 부족해서 `cheerio`로 직접 파싱하는 폴백 로직을 추가했다. 이 작업에만 3일을 소비했다.

> 7월 2일: OpenAI API 비용이 예상보다 높다. 글 1개 요약에 약 500~800 토큰을 사용하는데, 사용자 한 명이 하루에 20개씩 요약하면 월 $2 정도의 API 비용이 발생한다. 무료 사용자에게는 하루 5개 요약 제한을 걸기로 했다.

### 2025년 8월: 비공개 베타 런칭

MVP가 완성되고, 지인 개발자 15명에게 비공개 베타를 공개했습니다.

**피드백 요약:**
- "UI가 깔끔하다" (7명) → 기분 좋았지만 비즈니스 검증은 아님
- "AI 요약 품질이 들쭉날쭉하다" (5명) → 프롬프트 튜닝 필요
- "피드 추가가 번거롭다. OPML 가져오기 지원해달라" (4명) → 급히 추가
- "로딩이 느리다" (3명) → 피드 데이터 캐싱 미비

가장 치명적이었던 피드백은 AI 요약 품질이었습니다. 처음에는 단순히 "다음 글을 3줄로 요약해줘"라는 프롬프트를 사용했는데, 기술 글의 경우 핵심 개념이 누락되는 경우가 많았습니다.

프롬프트를 다음과 같이 개선했습니다.

```
Before: "다음 글을 3줄로 요약해주세요."

After: "당신은 시니어 소프트웨어 엔지니어입니다.
다음 기술 블로그 글을 읽고 아래 형식으로 요약해주세요:

1. **핵심 주제** (1문장): 이 글이 다루는 핵심 문제나 기술
2. **주요 내용** (2-3문장): 저자의 핵심 주장이나 해결 방법
3. **실용적 가치** (1문장): 독자가 이 글에서 바로 적용할 수 있는 것

기술 용어는 그대로 유지하되, 불필요한 서론은 생략하세요."
```

이 변경 후 베타 사용자 만족도가 크게 올라갔습니다.

### 2025년 9월~10월: 공개 런칭 준비

**런칭 전 체크리스트:**
- 랜딩 페이지 제작 (Vercel에 별도 배포)
- 가격 정책 결정
- Stripe 구독 연동 테스트
- 이용약관, 개인정보처리방침 작성
- SEO 기본 설정 (메타 태그, Open Graph, sitemap)
- 구글 서치 콘솔 등록

**가격 정책 (가장 많이 고민한 부분):**

| 플랜 | 월 가격 | 기능 |
|------|---------|------|
| Free | $0 | 피드 10개, AI 요약 5회/일 |
| Pro | $5/월 | 피드 무제한, AI 요약 무제한, 주간 다이제스트 |
| Team | $12/월/인 | Pro + 팀 공유 보드, 관리자 대시보드 |

처음에는 Pro를 $9로 책정했는데, 주변 개발자들에게 물어보니 "RSS 리더에 $9는 좀 비싸다"는 반응이 대부분이었습니다. Feedly의 Pro가 $8인데 인지도 차이를 감안하면 $5가 적당하다고 판단했습니다.

### 2025년 11월: 공개 런칭

**런칭 채널별 결과:**

| 채널 | 방문자 | 가입 | 전환율 |
|------|--------|------|--------|
| Product Hunt | 342 | 28 | 8.2% |
| Hacker News (Show HN) | 186 | 11 | 5.9% |
| X (트위터) 포스트 | 89 | 7 | 7.9% |
| 긱뉴스 (GeekNews) | 127 | 15 | 11.8% |
| 디스콘 (개발자 커뮤니티) | 45 | 4 | 8.9% |

**긱뉴스가 전환율 1위였습니다.** 한국어 서비스이다 보니 한국 개발자 커뮤니티에서의 반응이 가장 좋았습니다. Product Hunt는 방문자는 많았지만, 대부분 해외 사용자라 한국어 위주의 콘텐츠에 관심이 적었습니다.

런칭 첫 주 총계: **방문자 789명, 가입 65명, 유료 전환 0명**

유료 전환이 0이었습니다. 솔직히 충격이었습니다. 65명이나 가입했는데 아무도 돈을 내지 않았습니다.

### 2025년 12월: 가장 힘들었던 시기

가입자는 늘었지만 (총 112명) 유료 전환은 여전히 0이었습니다. 이 시기에 느낀 감정들을 솔직하게 적자면:

- **1주차**: "무료로 충분히 쓸만한가 보다. 무료 기능을 줄여야 하나?" → 무료 AI 요약을 5회에서 3회로 줄임
- **2주차**: "제품 자체에 문제가 있는 건 아닐까?" → 사용자 행동 분석 (Vercel Analytics)
- **3주차**: "차라리 접을까?" → 진지하게 고민. 하지만 매일 접속하는 DAU 사용자가 23명이나 있었음
- **4주차**: 사용자 인터뷰 시작 → 5명의 활성 사용자에게 직접 이메일을 보냄

**사용자 인터뷰에서 알게 된 핵심 인사이트:**

> "AI 요약은 좋은데, 제가 자주 읽는 분야의 글만 추천해주면 돈을 낼 의향이 있어요." — 사용자 A (프론트엔드 개발자)

> "주간 다이제스트가 유료 기능인 줄 몰랐어요. 이메일은 잘 안 보는데, 슬랙 알림이면 좋겠어요." — 사용자 B (DevOps 엔지니어)

이 피드백을 반영해서 12월 마지막 2주 동안 두 가지 기능을 급히 추가했습니다.

1. **개인화 추천**: 사용자가 읽은 글 기반으로 관심 분야 추정, 유사한 글 추천
2. **Slack 알림 연동**: 주간 다이제스트를 이메일 대신 Slack Webhook으로 발송

### 2026년 1월: 첫 유료 고객

1월 8일, 첫 유료 구독이 발생했습니다. Stripe 대시보드에서 "$5.00 — Subscription created" 알림을 보았을 때의 감정은 잊을 수 없습니다. 금액은 작았지만, "누군가 내가 만든 것에 돈을 냈다"는 사실 자체가 엄청난 의미였습니다.

**1월 결산:**
- 총 가입자: 187명
- 유료 전환: 4명
- MRR (Monthly Recurring Revenue): $20
- 전환율: 2.1%

첫 유료 고객이 된 4명의 공통점은 **모두 Slack 연동 기능을 사용하고 있었다**는 것입니다. 이메일 다이제스트보다 Slack 알림이 유료 전환의 핵심 트리거였습니다.

## 마케팅: 효과 있었던 것 vs 없었던 것

### 효과 있었던 것

1. **기술 블로그 글 작성** — "RSS 피드를 AI로 요약하는 시스템 만들기"라는 기술 글을 작성해서 개발자 커뮤니티에 공유. 이 글 하나로 가입자 40명 이상 유입. 제품 홍보가 아니라 기술적 가치를 담은 콘텐츠가 핵심.
2. **긱뉴스/디스콘 활동** — 한국 개발자 커뮤니티에서 꾸준히 활동하면서 자연스럽게 제품을 언급. 노골적 광고가 아니라 관련 주제 논의 중에 언급하는 방식.
3. **사용자 인터뷰 후 기능 반영** — 피드백을 반영했다는 사실을 공유하면, 해당 사용자가 자발적으로 추천해줌.

### 효과 없었던 것

1. **Product Hunt 런칭** — 방문자는 많았지만 실질적 전환은 거의 없음. 영어권 서비스가 아니면 효과 제한적.
2. **인스타그램/페이스북 광고** — $50 지출, 가입 2명. 개발자 타겟팅이 어려움.
3. **이메일 뉴스레터** — 구독자 모으는 데 시간이 오래 걸리고, 초기 단계에서는 비효율적.

## 기술적 후회와 교훈

### 1. 너무 일찍 최적화했다

MVP 단계에서 Redis 캐싱, CDN 설정, 이미지 최적화에 시간을 썼습니다. 사용자가 10명도 안 되는 단계에서 이런 최적화는 무의미했습니다. **사용자가 100명이 될 때까지는 성능보다 기능에 집중해야 합니다.**

### 2. 테스트 코드를 나중에 작성했다

초기 속도를 위해 테스트를 생략하고 기능 개발에만 집중했습니다. 3개월이 지나자 새 기능을 추가할 때마다 기존 기능이 깨지는 일이 빈번해졌습니다. 결국 E2E 테스트(Playwright)와 주요 API의 통합 테스트를 나중에 추가하느라 2주를 소비했습니다. **최소한 핵심 비즈니스 로직에 대한 테스트는 처음부터 작성해야 합니다.**

### 3. 인증 시스템을 직접 만들려 했다

처음에 NextAuth.js로 직접 인증을 구현하다가 2주를 낭비했습니다. 소셜 로그인, 이메일 인증, 세션 관리 등 고려할 것이 너무 많았습니다. Supabase Auth로 전환한 뒤 하루 만에 완성했습니다. **인증은 직접 만들지 마세요. 검증된 서비스를 사용하세요.**

### 4. App Router의 서버 컴포넌트를 제대로 활용하지 못했다

Next.js App Router를 선택해놓고, 습관적으로 `'use client'`를 남발했습니다. 나중에 리팩터링하면서 서버 컴포넌트로 전환할 수 있는 부분이 상당히 많다는 것을 알게 되었습니다. 서버 컴포넌트를 제대로 활용했다면 초기 로딩 속도가 훨씬 빨랐을 것입니다.

## 감정의 롤러코스터

1인 개발에서 가장 힘든 것은 기술적 문제가 아니라 심리적 문제입니다.

- **4월 (설렘)**: "이걸 만들면 대박 나겠다!"
- **6월 (자신감)**: "개발이 순조롭다. 예정보다 빠르게 진행되고 있다."
- **8월 (불안)**: "베타 피드백이 좋기는 한데... 돈을 낼까?"
- **10월 (초조)**: "런칭 준비가 끝이 없다. 완벽하지 않지만 공개해야 할까?"
- **11월 (실망)**: "65명이 가입했는데 유료 0명? 제품이 문제인가?"
- **12월 (좌절)**: "접을까 말까. 시간 낭비인 것 같다."
- **1월 (감동)**: "첫 $5. 누군가 내 제품에 돈을 냈다."

이 과정에서 배운 것은, **감정에 따라 의사결정하면 안 된다**는 것입니다. 12월에 포기했다면 1월의 첫 유료 고객은 없었을 것입니다. 데이터를 보고 판단해야 합니다. DAU가 꾸준히 유지되고 있었기 때문에 계속한 것이고, 그 판단이 맞았습니다.
