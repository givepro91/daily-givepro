Next.js로 블로그를 만들겠다고 결심한 건 2025년 말이었습니다. "React 좀 할 줄 아니까 금방 만들겠지"라는 안일한 생각으로 시작했는데, 결론부터 말하면 완전히 틀렸습니다. App Router의 개념부터 MDX 설정, 다크 모드, SEO까지 거의 모든 단계에서 삽질을 했고, 새벽 3시에 에러 메시지를 구글링하는 날이 한두 번이 아니었습니다.

이 글에서는 제가 실제로 겪은 7가지 삽질과 해결 과정을 구체적으로 공유합니다. 같은 실수를 반복하지 않으시길 바라는 마음에서, 에러 메시지와 잘못된 접근법까지 모두 포함했습니다.

## 삽질 1: App Router에서 RSC와 Client Component 혼동

### 문제 상황

Next.js 14 이후 App Router가 기본이 되면서, 모든 컴포넌트가 기본적으로 Server Component(RSC)로 동작합니다. 처음엔 이게 뭔 소린지 몰랐습니다. 기존에 Pages Router에서 하던 것처럼 `useState`를 쓰려고 했더니 이런 에러가 뜹니다.

```
Error: useState only works in Client Components. Add the "use client" directive at the
top of the file to use it.
```

"아 그러면 전부 `'use client'` 붙이면 되겠네?" 라고 생각했습니다. 이게 첫 번째 실수였습니다.

### 잘못된 접근

```tsx
// ❌ 모든 파일 상단에 'use client'를 때려넣은 잘못된 접근
'use client'

import { useState, useEffect } from 'react'

export default function BlogPost({ params }: { params: { slug: string } }) {
  const [post, setPost] = useState(null)

  useEffect(() => {
    fetch(`/api/posts/${params.slug}`)
      .then(res => res.json())
      .then(data => setPost(data))
  }, [params.slug])

  return <article>{post?.content}</article>
}
```

이렇게 하니 빌드는 됐지만, 모든 페이지가 클라이언트에서 렌더링되면서 SEO가 완전히 망가졌습니다. Google Search Console에서 인덱싱된 페이지가 0개였을 때의 충격은 아직도 기억납니다.

### 올바른 해결법

핵심은 **서버 컴포넌트와 클라이언트 컴포넌트를 분리**하는 것입니다. 데이터를 가져오는 로직은 서버 컴포넌트에 두고, 상호작용이 필요한 부분만 클라이언트 컴포넌트로 분리합니다.

```tsx
// app/posts/[slug]/page.tsx - Server Component (기본값)
import { getPostBySlug } from '@/lib/posts'
import { PostContent } from '@/components/PostContent'
import { LikeButton } from '@/components/LikeButton'

export default async function BlogPost({ params }: { params: { slug: string } }) {
  // 서버에서 직접 데이터를 가져옴 - API 라우트 불필요
  const post = await getPostBySlug(params.slug)

  return (
    <article>
      <h1>{post.title}</h1>
      <PostContent content={post.content} />
      {/* 상호작용이 필요한 부분만 클라이언트 컴포넌트 */}
      <LikeButton postId={post.id} />
    </article>
  )
}
```

```tsx
// components/LikeButton.tsx - 이것만 Client Component
'use client'

import { useState } from 'react'

export function LikeButton({ postId }: { postId: string }) {
  const [liked, setLiked] = useState(false)
  return (
    <button onClick={() => setLiked(!liked)}>
      {liked ? '❤️' : '🤍'}
    </button>
  )
}
```

이렇게 분리하면 블로그 글의 본문은 서버에서 HTML로 렌더링되어 검색 엔진이 완벽하게 읽을 수 있고, 좋아요 버튼 같은 인터랙티브 요소만 클라이언트에서 동작합니다. 번들 사이즈도 극적으로 줄어듭니다.

제가 실제로 측정한 결과, 모든 파일에 `'use client'`를 붙였을 때 첫 로드 JS 번들이 **187KB**였던 것이 분리 후 **43KB**로 줄어들었습니다. Lighthouse 성능 점수도 67점에서 96점으로 올라갔습니다.

## 삽질 2: MDX 설정의 늪

### 문제 상황

블로그에 마크다운으로 글을 쓰되, React 컴포넌트도 사용하고 싶었습니다. MDX가 정답이라는 건 알았는데, 설정 과정이 진짜 지옥이었습니다. 공식 문서를 따라 `@next/mdx`를 설치했더니 이런 에러가 터집니다.

```
Module not found: Can't resolve '@mdx-js/react'
```

설치하고 나면 또 다른 에러:

```
Error: Expected a closing tag for `<InArticleAd />` but got end of file
```

### 잘못된 접근

처음에 `next.config.mjs`에서 `@next/mdx`를 이렇게 설정했습니다.

```js
// ❌ 불완전한 MDX 설정
import createMDX from '@next/mdx'

const withMDX = createMDX()

const nextConfig = {
  pageExtensions: ['js', 'jsx', 'ts', 'tsx', 'md', 'mdx'],
}

export default withMDX(nextConfig)
```

이러면 기본적인 마크다운은 동작하지만, frontmatter 파싱이 안 되고, 커스텀 컴포넌트를 사용할 수 없고, 코드 하이라이팅도 안 됩니다.

### 올바른 해결법

결국 `@next/mdx` 대신 `next-mdx-remote`를 사용하는 방향으로 전환했습니다. `contentlayer`도 시도했었는데, 2025년 기준으로 Next.js 14+와 호환 문제가 있어서 포기했습니다.

```bash
npm install next-mdx-remote gray-matter rehype-highlight rehype-slug remark-gfm
```

핵심은 MDX 파일을 직접 파싱하는 유틸리티 함수를 만드는 것입니다.

```ts
// lib/posts.ts
import fs from 'fs'
import path from 'path'
import matter from 'gray-matter'

const postsDirectory = path.join(process.cwd(), 'content/posts')

export function getPostBySlug(slug: string) {
  const fullPath = path.join(postsDirectory, `${slug}.mdx`)
  const fileContents = fs.readFileSync(fullPath, 'utf8')
  const { data, content } = matter(fileContents)

  return {
    slug,
    content,
    title: data.title,
    description: data.description,
    date: data.date,
    category: data.category,
    tags: data.tags || [],
  }
}

export function getAllPosts() {
  const slugs = fs.readdirSync(postsDirectory)
    .filter(file => file.endsWith('.mdx'))
    .map(file => file.replace(/\.mdx$/, ''))

  return slugs
    .map(slug => getPostBySlug(slug))
    .sort((a, b) => (a.date > b.date ? -1 : 1))
}
```

그리고 서버 컴포넌트에서 `MDXRemote`를 사용해 렌더링합니다.

```tsx
// app/posts/[slug]/page.tsx
import { MDXRemote } from 'next-mdx-remote/rsc'
import { getPostBySlug } from '@/lib/posts'
import rehypeHighlight from 'rehype-highlight'
import remarkGfm from 'remark-gfm'

const components = {
  InArticleAd: () => <div className="ad-container">{/* 광고 컴포넌트 */}</div>,
}

export default async function Post({ params }: { params: { slug: string } }) {
  const post = getPostBySlug(params.slug)

  return (
    <article>
      <MDXRemote
        source={post.content}
        components={components}
        options={{
          mdxOptions: {
            remarkPlugins: [remarkGfm],
            rehypePlugins: [rehypeHighlight],
          },
        }}
      />
    </article>
  )
}
```

이 방식의 장점은 서버 사이드에서 MDX가 렌더링되기 때문에 클라이언트 번들에 MDX 파서가 포함되지 않는다는 것입니다. `next-mdx-remote/rsc` 임포트 경로가 핵심인데, `/rsc`를 빠뜨리면 클라이언트 버전이 임포트되면서 또 다른 에러가 발생합니다.

## 삽질 3: Tailwind CSS v4에서 다크 모드 토글이 안 되는 문제

### 문제 상황

Tailwind CSS v4로 프로젝트를 셋업한 후 다크 모드를 구현하려고 했습니다. v3에서 하던 것처럼 `tailwind.config.ts`에 `darkMode: 'class'`를 설정하고, `<html>` 태그에 `dark` 클래스를 토글하는 방식으로 구현했는데, 다크 모드 스타일이 전혀 적용되지 않았습니다.

```tsx
// 다크 모드가 적용되지 않는다!
<div className="bg-white dark:bg-gray-900">
  <p className="text-black dark:text-white">Hello</p>
</div>
```

개발자 도구를 열어보면 `dark:` 프리픽스가 붙은 클래스는 생성되는데, 실제로 스타일이 적용되지 않습니다.

### 잘못된 접근

```ts
// ❌ Tailwind v4에서는 이 설정이 동작하지 않음
// tailwind.config.ts
export default {
  darkMode: 'class',
  content: ['./app/**/*.{ts,tsx}', './components/**/*.{ts,tsx}'],
  theme: {
    extend: {},
  },
}
```

Tailwind CSS v4는 설정 파일 기반이 아닌 CSS 기반 설정으로 완전히 바뀌었습니다. `tailwind.config.ts` 자체가 기본적으로 무시됩니다.

### 올바른 해결법

Tailwind v4에서는 CSS 파일에서 `@custom-variant`를 사용해 다크 모드를 설정해야 합니다.

```css
/* app/globals.css */
@import "tailwindcss";

/* Tailwind v4: class 기반 다크 모드 설정 */
@custom-variant dark (&:where(.dark, .dark *));
```

이 한 줄이 핵심입니다. v3의 `darkMode: 'class'`와 동일한 효과를 냅니다. 이걸 찾기까지 공식 문서를 세 번은 읽었고, GitHub 이슈를 뒤졌고, 새벽 2시에 결국 Tailwind CSS v4 릴리즈 노트에서 발견했습니다.

다크 모드 토글 컴포넌트는 이렇게 만들었습니다.

```tsx
'use client'

import { useEffect, useState } from 'react'

export function ThemeToggle() {
  const [isDark, setIsDark] = useState(false)

  useEffect(() => {
    // 시스템 설정 또는 저장된 설정 확인
    const saved = localStorage.getItem('theme')
    const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches
    const shouldBeDark = saved === 'dark' || (!saved && prefersDark)

    setIsDark(shouldBeDark)
    document.documentElement.classList.toggle('dark', shouldBeDark)
  }, [])

  const toggle = () => {
    const newValue = !isDark
    setIsDark(newValue)
    document.documentElement.classList.toggle('dark', newValue)
    localStorage.setItem('theme', newValue ? 'dark' : 'light')
  }

  return (
    <button onClick={toggle} aria-label="다크 모드 토글">
      {isDark ? '☀️' : '🌙'}
    </button>
  )
}
```

한 가지 추가 삽질이 있었는데, 페이지 로드 시 흰색 화면이 번쩍이는 FOUC(Flash of Unstyled Content) 문제입니다. 이건 `layout.tsx`의 `<html>` 태그에 인라인 스크립트를 추가해서 해결했습니다.

```tsx
// app/layout.tsx
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ko" suppressHydrationWarning>
      <head>
        <script dangerouslySetInnerHTML={{
          __html: `
            (function() {
              var saved = localStorage.getItem('theme');
              var prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
              if (saved === 'dark' || (!saved && prefersDark)) {
                document.documentElement.classList.add('dark');
              }
            })()
          `,
        }} />
      </head>
      <body>{children}</body>
    </html>
  )
}
```

`suppressHydrationWarning`을 빼먹으면 서버와 클라이언트의 HTML이 달라서 hydration 경고가 뜹니다. 이것도 삽질 포인트입니다.

## 삽질 4: 동적 메타데이터와 SEO 최적화

### 문제 상황

블로그를 만든 목적 중 하나가 검색 엔진 노출이었는데, 빌드 후 소스 보기를 해보니 모든 페이지의 `<title>`이 동일했습니다. 각 포스트별로 고유한 메타데이터가 생성되지 않고 있었던 것입니다.

### 잘못된 접근

```tsx
// ❌ 클라이언트에서 document.title을 변경하는 잘못된 방식
'use client'

import { useEffect } from 'react'

export default function BlogPost({ post }) {
  useEffect(() => {
    document.title = post.title
  }, [post.title])

  return <article>...</article>
}
```

이 방식은 JavaScript가 실행된 후에만 타이틀이 변경되므로, 검색 엔진 크롤러가 올바른 타이틀을 읽지 못할 수 있습니다.

### 올바른 해결법

Next.js App Router에서는 `generateMetadata` 함수를 사용합니다.

```tsx
// app/posts/[slug]/page.tsx
import { Metadata } from 'next'
import { getPostBySlug, getAllPosts } from '@/lib/posts'

export async function generateMetadata(
  { params }: { params: { slug: string } }
): Promise<Metadata> {
  const post = getPostBySlug(params.slug)

  return {
    title: `${post.title} | Daily Givepro`,
    description: post.description,
    openGraph: {
      title: post.title,
      description: post.description,
      type: 'article',
      publishedTime: post.date,
      tags: post.tags,
    },
    twitter: {
      card: 'summary_large_image',
      title: post.title,
      description: post.description,
    },
  }
}

// 정적 경로 미리 생성
export async function generateStaticParams() {
  const posts = getAllPosts()
  return posts.map((post) => ({ slug: post.slug }))
}
```

사이트맵도 자동 생성하도록 했습니다. `app/sitemap.ts` 파일 하나면 됩니다.

```ts
// app/sitemap.ts
import { MetadataRoute } from 'next'
import { getAllPosts } from '@/lib/posts'

export default function sitemap(): MetadataRoute.Sitemap {
  const posts = getAllPosts()
  const baseUrl = 'https://daily-givepro.vercel.app'

  const postUrls = posts.map((post) => ({
    url: `${baseUrl}/posts/${post.slug}`,
    lastModified: new Date(post.date),
    changeFrequency: 'monthly' as const,
    priority: 0.8,
  }))

  return [
    {
      url: baseUrl,
      lastModified: new Date(),
      changeFrequency: 'daily',
      priority: 1,
    },
    ...postUrls,
  ]
}
```

추가로 `robots.ts`도 만들었습니다.

```ts
// app/robots.ts
import { MetadataRoute } from 'next'

export default function robots(): MetadataRoute.Robots {
  return {
    rules: {
      userAgent: '*',
      allow: '/',
    },
    sitemap: 'https://daily-givepro.vercel.app/sitemap.xml',
  }
}
```

이걸 적용하고 Google Search Console에 사이트맵을 제출하니, 2주 후 인덱싱된 페이지가 0개에서 12개로 늘어났습니다. 메타데이터 설정이 SEO에 얼마나 중요한지 체감한 순간이었습니다.

## 삽질 5: 이미지 최적화와 번들 사이즈 전쟁

### 문제 상황

블로그에 스크린샷과 다이어그램을 넣기 시작하니 페이지 로딩 속도가 눈에 띄게 느려졌습니다. Lighthouse를 돌려보니 LCP(Largest Contentful Paint)가 **4.2초**로 나왔습니다. 원인은 최적화되지 않은 PNG 이미지였습니다.

### 잘못된 접근

```tsx
// ❌ 일반 img 태그 사용
<img src="/images/screenshot.png" alt="스크린샷" />
```

이러면 원본 크기 그대로 다운로드됩니다. 2MB짜리 스크린샷이 모바일에서도 그대로 로드됩니다.

### 올바른 해결법

Next.js의 `Image` 컴포넌트를 사용하면 자동으로 WebP 변환, 리사이징, lazy loading이 적용됩니다.

```tsx
import Image from 'next/image'

// 로컬 이미지
import screenshot from '@/public/images/screenshot.png'

<Image
  src={screenshot}
  alt="VS Code에서 MDX 파일을 편집하는 화면"
  placeholder="blur"  // 로컬 이미지는 자동 blur placeholder
  sizes="(max-width: 768px) 100vw, 768px"
  quality={80}
/>
```

하지만 MDX 파일 안에서는 Next.js `Image` 컴포넌트를 바로 사용할 수 없습니다. MDX 컴포넌트 매핑에서 `img` 태그를 오버라이드해야 합니다.

```tsx
const components = {
  img: (props: any) => (
    <Image
      {...props}
      width={768}
      height={432}
      sizes="(max-width: 768px) 100vw, 768px"
      loading="lazy"
      className="rounded-lg"
    />
  ),
}
```

번들 사이즈도 줄이기 위해 `@next/bundle-analyzer`를 설치해서 분석했습니다.

```bash
npm install @next/bundle-analyzer
```

분석 결과 가장 큰 범인은 `rehype-highlight`의 모든 언어 팩이 번들에 포함되는 것이었습니다. 실제로 사용하는 언어만 임포트하도록 변경했습니다.

```ts
import rehypeHighlight from 'rehype-highlight'
import langJavascript from 'highlight.js/lib/languages/javascript'
import langTypescript from 'highlight.js/lib/languages/typescript'
import langBash from 'highlight.js/lib/languages/bash'
import langCss from 'highlight.js/lib/languages/css'
import langJson from 'highlight.js/lib/languages/json'

const options = {
  mdxOptions: {
    rehypePlugins: [
      [rehypeHighlight, {
        languages: {
          javascript: langJavascript,
          typescript: langTypescript,
          bash: langBash,
          css: langCss,
          json: langJson,
        }
      }]
    ],
  },
}
```

이 최적화만으로 번들 사이즈가 **약 120KB** 줄어들었고, LCP는 **4.2초에서 1.8초**로 개선되었습니다.

## 삽질 6: 빌드 시 날짜 포맷팅 에러

### 문제 상황

로컬에서 완벽하게 동작하던 날짜 포맷팅이 Vercel 배포 후에 다르게 표시되는 문제가 있었습니다. 한국어로 "2026년 3월 8일"로 표시되어야 하는데, 배포 환경에서는 "March 8, 2026"으로 나왔습니다.

### 잘못된 접근

```tsx
// ❌ locale에 의존하는 날짜 포맷팅
const formattedDate = new Date(post.date).toLocaleDateString()
```

`toLocaleDateString()`은 실행 환경의 시스템 로케일에 따라 다르게 동작합니다. 내 맥북에서는 한국어가 기본이지만, Vercel의 서버는 영어입니다.

### 올바른 해결법

```tsx
// ✅ 명시적으로 locale을 지정
const formattedDate = new Date(post.date).toLocaleDateString('ko-KR', {
  year: 'numeric',
  month: 'long',
  day: 'numeric',
})
// 결과: "2026년 3월 8일"
```

단순하지만, 이걸 모르면 "로컬에서는 되는데 배포하면 안 돼요"라는 클래식한 문제에 빠집니다. 비슷한 이슈로 `Intl.DateTimeFormat`을 사용할 때도 항상 locale을 명시적으로 전달해야 합니다.

추가로 시간대 문제도 있었습니다. frontmatter에 `date: "2026-03-08"`이라고 쓰면 `new Date("2026-03-08")`은 UTC 자정으로 해석되어, 한국 시간(KST, UTC+9)에서는 "2026년 3월 7일"로 표시될 수 있습니다.

```tsx
// ✅ 시간대 이슈를 피하는 안전한 날짜 파싱
function parseDate(dateString: string) {
  const [year, month, day] = dateString.split('-').map(Number)
  return new Date(year, month - 1, day) // 로컬 시간대로 생성
}
```

## 삽질 7: 정적 빌드에서 동적 콘텐츠 다루기

### 문제 상황

블로그 글 목록 페이지에서 글 개수, 최신 글 등을 보여주는데, 새 글을 추가하고 배포해도 이전 데이터가 그대로 나오는 문제가 있었습니다. 캐싱 문제였습니다.

### 잘못된 접근

Next.js App Router에서 기본적으로 `fetch`의 결과가 캐싱됩니다. 또한 정적으로 생성된 페이지는 `revalidate` 설정이 없으면 영원히 캐싱됩니다.

```tsx
// ❌ 캐싱 제어 없이 데이터를 가져옴
export default async function Home() {
  const posts = getAllPosts() // 빌드 시점의 데이터가 계속 사용됨
  return <PostList posts={posts} />
}
```

### 올바른 해결법

파일 시스템에서 직접 읽는 경우 `revalidate`를 설정해도 효과가 없습니다. 정적 사이트 생성(SSG)에서는 새 글을 추가할 때마다 빌드를 다시 해야 합니다. 하지만 매번 수동으로 빌드하는 건 번거롭습니다.

해결법은 **GitHub Push 시 자동 빌드**입니다. Vercel에 연결하면 기본적으로 `main` 브랜치에 push할 때마다 자동 빌드 및 배포가 됩니다. 추가로 ISR(Incremental Static Regeneration)을 활용할 수도 있습니다.

```tsx
// 외부 CMS를 사용하는 경우 ISR 적용
export const revalidate = 3600 // 1시간마다 재검증

export default async function Home() {
  const posts = await fetchPostsFromCMS()
  return <PostList posts={posts} />
}
```

저처럼 파일 시스템 기반으로 하는 경우에는 GitHub Actions에서 특정 시간에 빌드를 트리거하는 방법도 있습니다.

```yaml
# .github/workflows/scheduled-build.yml
name: Scheduled Build
on:
  schedule:
    - cron: '0 0 * * *' # 매일 자정 UTC
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Trigger Vercel Deploy
        run: curl -X POST ${{ secrets.VERCEL_DEPLOY_HOOK }}
```

## 전체 삽질을 통해 배운 것들

이 7가지 삽질을 거치면서 느낀 점을 정리하면 이렇습니다.

**1. 공식 문서를 먼저 읽자.** 블로그 포스트나 유튜브 튜토리얼은 버전이 맞지 않을 수 있습니다. 특히 Next.js처럼 빠르게 변하는 프레임워크는 6개월 전 글도 구식일 수 있습니다. Tailwind v4 다크 모드 삽질이 대표적입니다.

**2. RSC를 이해하는 것이 핵심이다.** App Router를 사용하면서 Server Component와 Client Component의 경계를 이해하지 못하면 모든 곳에서 문제가 발생합니다. "이 코드는 서버에서 실행되는가, 브라우저에서 실행되는가?"를 항상 자문해야 합니다.

**3. 로컬 환경과 배포 환경의 차이를 인식하자.** 로케일, 시간대, 환경 변수, Node.js 버전 등 로컬과 배포 환경이 다른 부분이 예상보다 많습니다.

**4. 측정하지 않으면 개선할 수 없다.** Lighthouse, Bundle Analyzer, Google Search Console 같은 도구를 적극 활용해야 합니다. "느린 것 같다"가 아니라 "LCP가 4.2초에서 1.8초로 개선됐다"로 말할 수 있어야 합니다.

**5. 완벽하지 않아도 배포하자.** 저도 처음에 모든 것을 완벽하게 만들고 배포하려 했지만, 실제로는 배포한 후에야 발견되는 문제가 대부분입니다. 다크 모드 FOUC, 날짜 포맷 문제, SEO 이슈 모두 배포 후에 발견했습니다.

이 블로그를 만드는 과정 전체가 하나의 큰 프로젝트였고, 가장 많이 성장한 경험이었습니다. 여러분도 직접 블로그를 만들어 보시길 강력히 추천합니다. 힘든 만큼 배우는 것도 많습니다.
