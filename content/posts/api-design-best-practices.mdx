"API를 다시 설계해야 할 것 같습니다." 이 말을 팀 미팅에서 꺼내는 것은 정말 고통스러운 일이다. API는 한번 공개되면 변경하기가 극도로 어렵다. 클라이언트 앱, 서드파티 연동, 문서 등 모든 것이 기존 API에 의존하고 있기 때문이다. 나는 3년간 세 개의 프로젝트에서 REST API를 설계했고, 그 과정에서 잘한 결정과 후회한 결정을 모두 겪었다.

이 글은 교과서적인 REST API 이론이 아니다. 실제로 API를 만들고 운영하면서 "이걸 왜 처음부터 이렇게 안 했을까"라고 후회한 것들, 그리고 "이 결정은 정말 잘했다"고 느낀 것들을 정리한 것이다.

## URL 설계: 첫인상이 가장 중요하다

### 리소스 중심 사고

REST API의 URL은 동사가 아닌 명사여야 한다는 것은 많이 알려진 원칙이다. 하지만 실제 프로젝트에서는 이 원칙을 지키기가 생각보다 어렵다.

```
# ❌ 동사 기반 (잘못된 패턴)
POST /api/createUser
GET /api/getUserById?id=123
POST /api/updateUserProfile
DELETE /api/removeUser?id=123

# ✅ 리소스 기반 (올바른 패턴)
POST /api/users
GET /api/users/123
PATCH /api/users/123
DELETE /api/users/123
```

여기서 내가 직접 경험한 한 가지 교훈을 공유한다. 초기 프로젝트에서 "편의상" 동사 기반 URL을 사용했는데, 프로젝트가 커지면서 URL이 100개 이상이 되었을 때 일관성이 완전히 무너졌다. `/api/createOrder`, `/api/addItemToCart`, `/api/processPayment`... 각 API의 이름을 외워야 했고, 새로운 팀원이 들어올 때마다 "이 API가 뭐하는 거예요?"라는 질문이 반복됐다.

리소스 기반으로 전환하고 나서는 URL만 보고도 어떤 동작인지 예측할 수 있게 됐다. `GET /api/orders`는 주문 목록 조회, `POST /api/orders`는 주문 생성, `PATCH /api/orders/123`은 주문 수정이라는 것을 누구나 알 수 있다.

### 중첩 리소스의 깊이 제한

리소스 간 관계를 URL에 표현할 때, 중첩을 너무 깊게 하면 안 된다.

```
# ❌ 너무 깊은 중첩
GET /api/users/123/orders/456/items/789/reviews/101

# ✅ 적절한 중첩 (2단계까지)
GET /api/users/123/orders
GET /api/orders/456/items
GET /api/items/789/reviews
```

팀에서 도입했을 때 합의한 규칙은 URL 중첩을 최대 2단계로 제한하는 것이었다. 그보다 깊은 관계는 쿼리 파라미터나 별도의 엔드포인트로 처리했다. 이 규칙을 정한 후 API의 복잡도가 크게 줄었다.

### 일관된 네이밍 컨벤션

| 규칙 | 예시 | 설명 |
|------|------|------|
| 복수형 사용 | `/api/users` (O), `/api/user` (X) | 컬렉션은 복수 |
| 소문자 + 하이픈 | `/api/order-items` (O), `/api/orderItems` (X) | URL은 케밥 케이스 |
| 동사 지양 | `/api/users` (O), `/api/getUsers` (X) | HTTP 메서드가 동사 역할 |
| 필터는 쿼리 파라미터 | `/api/users?status=active` (O) | URL은 리소스, 필터는 쿼리 |
| 후행 슬래시 없음 | `/api/users` (O), `/api/users/` (X) | 일관성 |

이 컨벤션을 문서로 정리하고 린터로 강제한 것이 좋은 결정이었다. ESLint 커스텀 룰을 만들어서, 라우트 핸들러의 URL이 컨벤션을 위반하면 경고를 띄우도록 했다.

## HTTP 메서드: 올바르게 사용하기

### 각 메서드의 의미

```
GET     - 리소스 조회 (멱등, 안전)
POST    - 리소스 생성 (비멱등)
PUT     - 리소스 전체 교체 (멱등)
PATCH   - 리소스 부분 수정 (멱등)
DELETE  - 리소스 삭제 (멱등)
```

여기서 `PUT`과 `PATCH`의 차이를 제대로 구분하는 것이 중요하다. 내가 실무에서 겪은 실수인데, 처음에는 모든 수정을 `PUT`으로 처리했다. 하지만 `PUT`은 리소스 전체를 교체하는 것이므로, 이름만 변경하고 싶어도 전체 사용자 데이터를 보내야 했다.

```tsx
// PUT: 전체 교체 - 모든 필드를 보내야 함
PUT /api/users/123
{
  "name": "김개발",
  "email": "dev@example.com",
  "phone": "010-1234-5678",
  "address": "서울시 강남구",
  "role": "developer"
}

// PATCH: 부분 수정 - 변경할 필드만 보내면 됨
PATCH /api/users/123
{
  "name": "김개발"
}
```

실무에서는 대부분의 수정 작업이 부분 수정이므로, `PATCH`를 더 많이 사용하게 된다. `PUT`은 설정 값 전체를 덮어쓰는 것처럼 "전체 교체"가 명확한 경우에만 사용한다.

### POST가 생성만을 위한 것이 아닌 경우

REST 원칙에 깔끔하게 맞지 않는 동작들이 있다. 예를 들어 "이메일 발송", "비밀번호 재설정", "보고서 생성" 같은 행위(action) 기반 작업은 순수 CRUD로 표현하기 어렵다.

```
# 행위 기반 엔드포인트 (현실적 타협)
POST /api/users/123/send-verification-email
POST /api/auth/reset-password
POST /api/reports/generate
```

이런 경우에 대해 팀 내에서 많은 논의가 있었다. 순수 REST를 고집하려면 "이메일 발송"을 "Email 리소스 생성"으로 해석할 수 있지만(`POST /api/emails`), 이는 직관적이지 않다. 우리 팀의 결론은 "실용주의"였다. 대부분의 CRUD는 리소스 기반으로, 순수 행위는 동사 기반으로 허용하되, 반드시 `POST`만 사용하기로 했다.

## 에러 처리: API의 품격을 결정하는 요소

에러 처리는 API 설계에서 가장 중요하면서도 가장 간과되는 부분이다. 잘 설계된 에러 응답은 클라이언트 개발자의 디버깅 시간을 극적으로 줄여준다.

### HTTP 상태 코드 올바르게 사용하기

내가 실제로 사용하는 상태 코드 체계다.

```
# 성공
200 OK              - 일반적인 성공
201 Created         - 리소스 생성 성공
204 No Content      - 성공했지만 반환할 데이터 없음 (DELETE 등)

# 클라이언트 에러
400 Bad Request     - 잘못된 요청 (유효성 검증 실패)
401 Unauthorized    - 인증 필요 (토큰 없음/만료)
403 Forbidden       - 인가 실패 (권한 없음)
404 Not Found       - 리소스 없음
409 Conflict        - 충돌 (중복 이메일 등)
422 Unprocessable   - 문법은 맞지만 의미적으로 처리 불가
429 Too Many Reqs   - Rate Limit 초과

# 서버 에러
500 Internal Error  - 서버 오류
503 Service Unavail - 일시적 서비스 불가
```

여기서 가장 많이 실수하는 것이 `401`과 `403`의 구분이다. `401`은 "누구인지 모르겠다"(인증 실패)이고, `403`은 "누구인지는 알겠는데 권한이 없다"(인가 실패)이다. 초기에 이것을 혼용해서 사용했다가, 클라이언트에서 "로그인 화면으로 보내야 할지, 권한 없음 메시지를 보여줘야 할지" 구분이 안 되는 문제가 생겼다.

### 일관된 에러 응답 포맷

모든 에러 응답이 동일한 구조를 가져야 한다. 우리 팀에서 사용하는 포맷이다.

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "입력값이 올바르지 않습니다.",
    "details": [
      {
        "field": "email",
        "message": "유효한 이메일 주소를 입력해주세요.",
        "value": "invalid-email"
      },
      {
        "field": "password",
        "message": "비밀번호는 8자 이상이어야 합니다.",
        "value": null
      }
    ],
    "timestamp": "2026-03-10T09:30:00Z",
    "requestId": "req_abc123def456"
  }
}
```

핵심 포인트를 설명하면, `code`는 머신 리더블한 에러 코드로 클라이언트가 프로그래밍적으로 에러를 처리할 때 사용한다. `message`는 사람이 읽을 수 있는 메시지다. `details`는 유효성 검증 에러 시 필드별 상세 정보를 제공한다. `requestId`는 디버깅 시 서버 로그와 매칭하기 위한 고유 ID다.

실제로 사용해보니, `requestId`가 가장 유용했다. 클라이언트 개발자가 "이 요청에서 에러가 나요"라고 할 때 `requestId`만 있으면 서버 로그에서 바로 찾을 수 있다. 이것이 없었을 때는 "언제, 어떤 파라미터로 요청했어요?"를 일일이 물어봐야 했다.

## 페이지네이션: 대량 데이터 처리의 핵심

### 세 가지 페이지네이션 방식 비교

실무에서 마주치는 페이지네이션 방식은 크게 세 가지다.

**1. Offset 기반 페이지네이션**

```
GET /api/users?page=3&limit=20
```

응답:
```json
{
  "data": [...],
  "pagination": {
    "page": 3,
    "limit": 20,
    "totalItems": 1234,
    "totalPages": 62
  }
}
```

장점은 구현이 간단하고, 특정 페이지로 바로 이동할 수 있다는 것이다. 단점은 데이터가 추가/삭제되면 페이지가 밀려서 중복이나 누락이 생길 수 있고, 뒤쪽 페이지일수록 쿼리가 느려진다는 것이다(OFFSET이 클수록 DB가 건너뛸 행이 많아짐).

**2. 커서 기반 페이지네이션**

```
GET /api/users?cursor=eyJpZCI6MTIzfQ&limit=20
```

응답:
```json
{
  "data": [...],
  "pagination": {
    "nextCursor": "eyJpZCI6MTQzfQ",
    "hasMore": true
  }
}
```

장점은 데이터 변경에 강하고, 뒤쪽 페이지도 일관된 성능이다. 단점은 "3페이지로 바로 이동"이 불가능하고, 총 개수를 알기 어렵다. 무한 스크롤 UI에 적합하다.

**3. Keyset 기반 페이지네이션**

```
GET /api/users?after_id=123&limit=20
```

커서 기반의 변형으로, 커서 값이 인코딩되지 않은 실제 키값이다. 구현이 더 직관적이지만, 정렬 기준이 변경되면 클라이언트 코드도 수정해야 한다.

### 실무에서의 선택 기준

| 상황 | 추천 방식 | 이유 |
|------|----------|------|
| 관리자 대시보드 (테이블) | Offset | 페이지 번호 필요, 데이터 규모 작음 |
| 소셜 피드 (무한 스크롤) | 커서 | 실시간 데이터 변경, 일관된 성능 |
| 검색 결과 | Offset | 페이지 이동 필요, 총 결과 수 표시 |
| 대용량 로그 조회 | 커서 | 수억 건, 성능 필수 |
| 모바일 앱 목록 | 커서 | 무한 스크롤, 네트워크 효율 |

내가 직접 경험한 교훈을 하나 공유하면, 처음에 모든 목록 API를 Offset 방식으로 구현했는데, 데이터가 100만 건이 넘어가면서 뒤쪽 페이지의 조회 시간이 5초 이상으로 늘어났다. `SELECT * FROM users OFFSET 999980 LIMIT 20`이라는 쿼리가 실제로 100만 행을 스캔하고 있었던 것이다. 결국 해당 API를 커서 기반으로 마이그레이션하는 작업을 해야 했고, 클라이언트 코드도 모두 수정해야 했다. 처음부터 데이터 규모를 예상하고 적절한 방식을 선택했어야 했다.

## 버전 관리: 미래의 나를 위한 투자

API 버전 관리는 API가 공개된 후에 필연적으로 필요해진다. 방법은 크게 세 가지가 있다.

### 1. URL 경로 버전 관리

```
GET /api/v1/users
GET /api/v2/users
```

장점은 명확하고 직관적이며, 라우팅 레벨에서 분리할 수 있다는 것이다. 단점은 URL이 길어지고, 버전 간 코드 중복이 생길 수 있다. 이것이 우리 팀에서 채택한 방식이다.

### 2. 헤더 버전 관리

```
GET /api/users
Accept: application/vnd.myapp.v2+json
```

장점은 URL이 깔끔하다는 것이다. 단점은 테스트와 디버깅이 번거롭고, 브라우저에서 직접 테스트하기 어렵다.

### 3. 쿼리 파라미터 버전 관리

```
GET /api/users?version=2
```

장점은 선택적으로 버전을 지정할 수 있다는 것이다. 단점은 캐싱이 복잡해지고, 컨벤션이 덜 표준적이다.

### 버전 업 전략

실무에서 겪은 교훈을 공유한다. API v2를 출시할 때 v1을 바로 폐기하면 안 된다. 우리 팀의 정책은 이렇다.

1. v2 출시 시 v1은 최소 6개월간 유지
2. v1 사용자에게 v2 마이그레이션 가이드와 일정을 사전 공지
3. v1의 응답 헤더에 Sunset 날짜를 포함 (`Sunset: Sat, 01 Nov 2026 00:00:00 GMT`)
4. v1 사용량을 모니터링하여, 사용량이 충분히 줄면 폐기

이 정책 덕분에 API 소비자(프론트엔드, 모바일 앱, 서드파티)에게 충분한 마이그레이션 시간을 줄 수 있었고, 갑작스러운 장애를 방지할 수 있었다.

## 인증과 보안: 타협할 수 없는 영역

### JWT 기반 인증의 실전 패턴

JWT(JSON Web Token) 기반 인증은 가장 널리 사용되는 방식이다. 하지만 제대로 구현하지 않으면 보안 취약점이 될 수 있다.

실제로 사용해보니, JWT를 사용할 때 주의할 점들이 많았다.

**Access Token의 만료 시간을 짧게 설정하라.** 15분 정도가 적절하다. 토큰이 탈취되어도 피해를 최소화할 수 있다. "15분마다 다시 로그인해야 하면 사용자가 불편하지 않나요?"라는 질문을 받았는데, 이건 Refresh Token으로 해결한다.

**Refresh Token은 반드시 서버에 저장하라.** Access Token은 stateless지만, Refresh Token은 서버 DB에 저장하고 관리해야 한다. 이렇게 해야 특정 사용자의 Refresh Token을 무효화(로그아웃, 계정 잠금 등)할 수 있다.

**Refresh Token 로테이션을 구현하라.** Refresh Token을 사용할 때마다 새로운 Refresh Token을 발급하고, 이전 토큰은 무효화한다. 이렇게 하면 Refresh Token이 탈취되어도 한 번만 사용 가능하므로 피해를 줄일 수 있다.

### Rate Limiting

API를 공개하면 반드시 Rate Limiting을 적용해야 한다. DDoS 공격 방지뿐만 아니라, 클라이언트의 실수(무한 루프에서 API 호출 등)로부터 서버를 보호하기 위해서도 필요하다.

```
# Rate Limit 응답 헤더 예시
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1709013600
Retry-After: 60
```

우리 팀에서 적용한 Rate Limiting 전략이다. 인증되지 않은 요청은 IP당 분당 30회, 인증된 요청은 사용자당 분당 100회, 관리자 요청은 분당 500회로 설정했다. 이 값은 서비스 특성에 따라 조정해야 한다.

### 입력값 검증

클라이언트에서 검증한다고 서버에서 검증을 생략하면 안 된다. 이건 당연한 이야기처럼 들리지만, 실무에서 생각보다 많이 놓치는 부분이다.

서버에서 반드시 검증해야 할 것들을 정리하면, 필수 필드 존재 여부, 데이터 타입 (문자열이 와야 할 곳에 숫자가 오진 않는지), 길이 제한 (SQL Injection 방지에도 도움), 범위 제한 (페이지 번호가 음수이거나, limit이 10000이거나), 형식 검증 (이메일, URL, 전화번호 등), 비즈니스 규칙 (재고보다 많은 수량 주문 등)이 있다.

Zod나 Joi 같은 스키마 검증 라이브러리를 사용하면 선언적으로 검증 로직을 작성할 수 있어서, 코드가 깔끔해지고 유지보수가 쉬워진다.

## 응답 설계: 일관성이 핵심

### 성공 응답 포맷

모든 성공 응답이 동일한 구조를 가져야 한다.

```json
// 단일 리소스
{
  "data": {
    "id": 123,
    "name": "김개발",
    "email": "dev@example.com"
  }
}

// 목록 리소스
{
  "data": [
    { "id": 123, "name": "김개발" },
    { "id": 124, "name": "이프론트" }
  ],
  "pagination": {
    "page": 1,
    "limit": 20,
    "totalItems": 50,
    "totalPages": 3
  }
}
```

### 필드 필터링 (Sparse Fields)

클라이언트가 필요한 필드만 요청할 수 있게 하면, 응답 크기를 줄이고 성능을 개선할 수 있다.

```
GET /api/users?fields=id,name,email
```

모바일 앱에서 특히 유용하다. 사용자 목록을 보여줄 때 프로필 사진 URL만 필요한데, 전체 사용자 데이터(프로필 설명, 설정, 활동 로그 등)를 다 보내면 낭비다.

### 관계 데이터 포함 (Expand/Include)

관련 리소스를 별도 요청 없이 한 번에 가져올 수 있게 하면 N+1 문제를 방지할 수 있다.

```
GET /api/orders/123?include=user,items
```

```json
{
  "data": {
    "id": 123,
    "status": "shipped",
    "user": {
      "id": 456,
      "name": "김개발"
    },
    "items": [
      { "id": 1, "name": "키보드", "price": 150000 }
    ]
  }
}
```

다만 include를 무제한으로 허용하면 보안 문제와 성능 문제가 생길 수 있다. 허용 가능한 include 목록을 명시적으로 정의하고, 깊이 제한을 두는 것이 좋다.

## 문서화: API의 생명줄

아무리 잘 설계된 API도 문서가 없으면 쓸모없다. 내가 실무에서 경험한 문서화 전략을 공유한다.

### OpenAPI (Swagger) 활용

API 문서는 코드에서 자동 생성하는 것이 가장 좋다. 수동으로 관리하면 코드와 문서가 불일치하는 것은 시간문제다.

우리 팀에서는 Zod 스키마를 정의하고, 이를 OpenAPI 스펙으로 자동 변환하는 방식을 사용한다. 코드가 곧 문서가 되므로, 불일치 문제가 원천적으로 방지된다.

### 좋은 API 문서의 조건

3년간 다양한 서드파티 API를 사용하면서 느낀, 좋은 API 문서의 조건은 이렇다.

1. **실행 가능한 예시가 있어야 한다.** "Try it out" 버튼으로 바로 API를 호출해볼 수 있어야 한다.
2. **에러 케이스가 명시되어야 한다.** 어떤 상황에서 어떤 에러가 반환되는지 구체적으로 알 수 있어야 한다.
3. **인증 방법이 명확해야 한다.** 토큰을 어떻게 얻고, 어디에 넣어야 하는지 바로 알 수 있어야 한다.
4. **변경 이력이 있어야 한다.** 어떤 필드가 추가되고, 어떤 엔드포인트가 폐기 예정인지 알 수 있어야 한다.

## 성능 최적화: 느린 API는 잘못 설계된 API다

### 데이터베이스 쿼리 최적화

API 성능 문제의 90%는 데이터베이스 쿼리에서 발생한다. 내가 직접 경험한 성능 이슈와 해결 과정을 공유한다.

프로젝트 초기에 사용자 목록 API의 응답 시간이 3초가 넘었다. 원인을 분석해보니 N+1 쿼리 문제였다. 사용자 100명을 조회한 후, 각 사용자의 최근 주문을 개별 쿼리로 조회하고 있었다. 즉, 1번의 사용자 조회 + 100번의 주문 조회 = 총 101번의 DB 쿼리가 실행되고 있었던 것이다.

이것을 JOIN 또는 ORM의 eager loading으로 변경하니, 쿼리가 2번으로 줄었고 응답 시간이 200ms로 개선됐다. Prisma에서는 `include`를 사용하면 되고, TypeORM에서는 `leftJoinAndSelect`를 사용한다. 이런 기본적인 문제가 실무에서 정말 많이 발생하므로, ORM의 쿼리 로그를 항상 켜두고 확인하는 습관을 들이는 것이 좋다.

### 캐싱 전략

자주 조회되면서 잘 변하지 않는 데이터는 캐싱해야 한다. 우리 팀에서 적용한 캐싱 계층은 이렇다.

**응답 레벨 캐싱:** HTTP 캐시 헤더(`Cache-Control`, `ETag`)를 활용한다. 공개 API의 목록 조회 같은 경우 `Cache-Control: public, max-age=60`으로 설정하면 CDN에서 캐싱되어 서버 부하를 크게 줄일 수 있다.

**애플리케이션 레벨 캐싱:** Redis를 사용하여 DB 쿼리 결과를 캐싱한다. 예를 들어 카테고리 목록처럼 자주 변경되지 않는 데이터는 Redis에 5분간 캐싱하고, 데이터가 변경될 때 캐시를 무효화한다.

**쿼리 레벨 캐싱:** 동일한 요청 내에서 같은 데이터를 여러 번 조회하는 경우, 요청 스코프의 인메모리 캐시(DataLoader 패턴)를 사용한다.

캐싱에서 가장 어려운 것은 캐시 무효화다. "캐시 무효화와 이름 짓기가 컴퓨터 과학에서 가장 어려운 두 가지 문제"라는 유명한 말이 있을 정도다. 우리 팀에서는 "변경이 발생하면 관련 캐시를 즉시 삭제"하는 단순한 전략을 사용하고 있다. 완벽하지는 않지만, 복잡한 캐시 동기화 로직보다 유지보수가 쉽다.

### 응답 압축과 최소화

API 응답의 크기도 성능에 영향을 미친다. 특히 모바일 환경에서는 응답 크기가 체감 성능에 직접적인 영향을 준다.

gzip 또는 Brotli 압축을 적용하면 JSON 응답의 크기를 60~80% 줄일 수 있다. 대부분의 웹 프레임워크에서 미들웨어로 간단하게 적용 가능하다. 그리고 앞서 소개한 필드 필터링을 활용하여 클라이언트가 필요한 데이터만 전송하면 추가적인 최적화가 가능하다.

## API 설계 체크리스트

새로운 API를 설계할 때 매번 확인하는 체크리스트를 공유한다.

| 카테고리 | 항목 | 확인 |
|---------|------|------|
| URL | 리소스 중심 명사형인가? | |
| URL | 복수형, 소문자, 하이픈 사용하는가? | |
| URL | 중첩 2단계 이내인가? | |
| 메서드 | HTTP 메서드를 올바르게 사용하는가? | |
| 요청 | 입력값 검증이 충분한가? | |
| 응답 | 일관된 응답 포맷을 사용하는가? | |
| 에러 | 적절한 상태 코드를 반환하는가? | |
| 에러 | 에러 응답에 충분한 정보가 있는가? | |
| 보안 | 인증/인가가 적용되었는가? | |
| 보안 | Rate Limiting이 설정되었는가? | |
| 성능 | 페이지네이션이 적용되었는가? | |
| 성능 | 필요한 데이터만 반환하는가? | |
| 문서 | API 문서가 업데이트되었는가? | |

## 마치며

좋은 API 설계는 처음부터 완벽하게 나오지 않는다. 하지만 기본 원칙을 지키고, 일관성을 유지하며, 실수에서 배우면 점점 나아진다. 이 글에서 공유한 내용이 모두 처음부터 알고 있었던 것이 아니다. 대부분 실수하고, 수정하고, 다시 설계하는 과정에서 배운 것들이다.

API는 한번 공개되면 변경하기 어렵다는 것을 항상 기억하자. 그래서 설계 단계에서 충분한 시간을 투자하는 것이 결국 시간을 절약하는 길이다.

이 글에서 다루지 못한 GraphQL vs REST, gRPC, WebSocket 등의 주제는 별도의 글에서 다루겠다. 궁금한 점이 있다면 댓글로 남겨주시기 바란다.
